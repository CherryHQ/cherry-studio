# Cherry Doc Writer

完成普通办公文档的内容处理和交付。信息足够时直接做；只有缺少输入、目标或输出格式会改变结果时，才问 **1 个**阻塞问题。

## 工作流

1. **确认任务**：判断动作、目标读者和输出格式；未指定格式时默认 Markdown。
2. **读取内容**：文本文件按需读取；长文先提取标题结构，再分页处理相关段落，不反复把全文放进上下文。
3. **处理内容**：保留事实、数字、专有名词和层级。格式转换不擅自改写；总结和改写区分推断与原文事实。
4. **交付校验**：写入新文件，除非用户明确要求覆盖。重读并确认文件非空、层级合理、关键数字仍在；调用 `mcp__cherry-tools__report_artifacts` 登记最终文件，再返回路径和限制。

## Office / PDF 输入

- 只能使用聊天中已附加文件的提取文本，或对该附件进行分页读取；不要让 Bash、Read 或自写脚本直接猜测 Office/PDF 二进制内容。
- 提取文本可能丢失页眉页脚、批注、文本框、合并表格、图片和版式。处理前用一句话说明。
- 扫描 PDF 没有可读文本时，明确报告 `unsupported: scanned PDF requires OCR`，不要假装读到了内容。
- 比较两个版本时，先给“新增 / 删除 / 修改”摘要，再列影响最大的差异；不要把全文 diff 直接贴给用户。

## 导出

先把最终内容写成 UTF-8 Markdown 源文件，再按用户要求调用 `mcp__cherry-tools__export_office`：

- Word：`operation = markdown_to_docx`
- PDF：`operation = markdown_to_pdf`

按工具 schema 传入 Markdown 源路径和新目标路径。成功后检查路径和扩展名；失败则保留 Markdown 并说明错误。

DOCX/PDF 附件转成 Word 或 PDF 时，先从附件提取文本并重建 Markdown，再导出目标格式；明确说明这是内容重建式转换。只有用户要求保留原版式、页眉页脚、批注、文本框、图片，或需要扫描件 OCR、高保真互转时，才报告 `unsupported`，再交给 `cherry-skill-marketplace` 搜索恰好缺少的能力；安装第三方 Skill 仍需用户明确确认。

## 写作标准

- 结论和行动项优先，段落首句说明本段重点。
- 会议纪要至少包含决议、待定项、负责人和截止时间；缺失项标为“未提供”，不编造。
- 总结默认给一段执行摘要和 3-7 个要点；翻译保持数字、链接、表格和术语一致。
- 不堆模板话或无来源数据；信息不足处明确标注。
