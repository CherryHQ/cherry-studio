# 给顾问的项目简报

## 1. 我是谁 / 项目是什么

我正在 Cherry Studio v2 fork 上做一个 AI 对话产品原型，核心功能是 **Text Anchor Branch / 文本锚点分支**：

> 用户选中 AI 回复中的一段文字 → 右键打开分支 → 在当前聊天界面右侧并列展开一个 branch conversation → AI 针对选中的那段内容继续回答。

产品目标不是普通“复制消息开新对话”，而是更接近：
- 对 AI 回复里的某句话、某个概念、某个段落继续追问；
- 主对话不离开；
- 右侧并列展开局部追问；
- 源文本有高亮，让用户知道 branch 是从哪里来的。

团队目前 2 人：
1. 我：项目负责人 / 架构判断 / 主力决策；
2. 一个低经验但听话的助手：负责低风险 UI 壳子、文案、样式、局部组件等任务。

## 3. 工作纪律 / 技术边界

项目基于 Cherry Studio v2 fork。

开发原则：

- v2 优先走 DataApi / Cache / v2 数据结构；
- UI 优先使用 `@cherrystudio/ui`、Tailwind、Shadcn 风格；
- 不要主动扩展 v1 Redux / Dexie 体系；
- Redux / Dexie 相关路径目前很多是 throwaway / 兼容层，除非为了桥接现有功能，不要把新功能深绑到 v1；
- 每次改动要小步推进，先方案、再实施、再验证；
- 不要一次性塞大改；
- 对高风险架构变更，先让我确认方案；
- Conventional Commits；
- commit 必须 `--signoff`；
- 不提交：
  - `Chat APP-Docs/.obsidian/workspace.json`
  - `Chat APP-Docs/.obsidian/graph.json`

## 6. 相关已知坑 / 不要重复踩

1. **不要回到 topic metadata 路线**
   - v2 `Topic` 没有可写 free-form JSON metadata；
   - P2 存储已经裁决为独立 `branch_anchor` 表。

2. **不要持久化 pending branches**
   - 当前语义是 write-on-keep；
   - 只有 kept branch 才创建 `branch_anchor`。

3. **不要把 `disposition` 当权威**
   - `disposition` 存储是为了前向扩展；
   - 读路径不能靠它判定业务真相。

4. **不要给 `messageId` 补 FK**
   - `messageId` 有意无 FK；
   - `blockId` 在 message data JSON 里，不是表；
   - source message deletion / blockId drift / offset mismatch 后续由 hydration degradation 处理。

5. **CSS Custom Highlight API 不要再建议**
   - 在当前环境中 `CSS.highlights.set()` 成功但不可见；
   - 当前正确方案是 `span.branch-anchor-highlight` wrap。

6. **不要把 branch panel 当 active topic 跳走**
   - 正确方向仍是 side-by-side，主对话不离开。

7. **Open as branch disabled 不一定是数量限制**
   - 如果选区端点落在有效 assistant `MainTextBlock` / `data-block-id` 边界外，`Open as branch` 会 disabled；
   - 这是 selection anchor resolution 行为，不是 active branch count limit；
   - manual smoke 应选择 assistant 回复内部 substring，不要选 line boundary / newline。
