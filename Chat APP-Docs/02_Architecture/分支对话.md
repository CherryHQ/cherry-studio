# 分支对话（branch chat）v2 实现现状

> 调查日期：2026-05-20
> 当前分支 `feature/expand-branch-chat-v2` 命名暗示这条分支在做"扩展对话分支"。本文档梳理它到了哪一步。

## 0. 一句话现状

**Schema + 后端 service + 测试都齐了；renderer UI 还在 v1 形态（`askId` + `foldSelected`），没消费 v2 的 `siblingsGroupId` / `activeNodeId` / `BranchMessagesResponse.siblingsGroup` —— 主分支上没有 v2 `SiblingNavigator` / v2 `MessageGroup`，按 `topics.ts:67-73` 注释，这些 renderer consumer 在另一条分支 `DeJeune/ai-service` 上待合。**

## 1. 数据模型上"分支"是什么

### 1.1 三个字段

| 字段 | 位置 | 语义 |
|---|---|---|
| `Message.parentId` | `packages/shared/data/types/message.ts` | 树父指针；`null` = 根 |
| `Message.siblingsGroupId` | 同上 | `0` = 普通；`>0` = 同一 parentId 下的"多模型回复组"标识 |
| `Topic.activeNodeId` | `packages/shared/data/types/topic.ts:28` | 当前焦点节点；决定下一条消息的默认父、对话历史截断点、tree 展开的活跃路径 |

### 1.2 树形结构

每个 Topic 是一棵**有根树**，single-root 由 service 显式约束（`create()` 中 `null` 分支会校验已有根则报错）。
- 树深度：`parentId` 链
- 树宽度："多模型同回合"通过 `siblingsGroupId` 标记（同 `parentId` + 同非零 `siblingsGroupId`）
- 焦点：`Topic.activeNodeId` 指向当前所在节点

### 1.3 与 v1 概念对照

| v1（renderer 现在用的） | v2（schema 已定） | 备注 |
|---|---|---|
| `Message.askId` | `Message.parentId` | v1 askId 同一组共享，v2 parentId 严格树 |
| `Message.foldSelected: boolean` | `Topic.activeNodeId` | v1 在消息上标"选中态"，v2 在 topic 上记焦点 |
| 同一 askId 下的所有 assistant 消息 | `(parentId, siblingsGroupId>0)` 组 | v1 同问多答靠 askId 隐式分组，v2 显式 |

迁移器（`ChatMigrator`）在 v1→v2 阶段做了语义映射（`askId`+`foldSelected` → `parentId`+`siblingsGroupId`），见 [数据模型.md](./数据模型.md#5)。

## 2. 后端 service 已经实现什么

详细算法在 [服务算法.md](./服务算法.md)。这里只列出**与分支对话直接相关**的能力：

### 2.1 树查询：`MessageService.getTree`
- 端点：`GET /topics/:topicId/tree`
- 入参：`rootId?` / `nodeId?` / `depth?`
- 输出：`{ nodes: TreeNode[], siblingsGroups: SiblingsGroup[], activeNodeId }`
- 行为：
  - depth = `-1`（无限） / `0`（仅活动路径） / `n`（活动路径 + 非活动 n 层）
  - 活动路径**永远完整展开**，不受 depth 限制
  - 同 `(parentId, siblingsGroupId)` 成员数 > 1 时聚合为 `SiblingsGroup`，单成员降级为普通 `TreeNode`

### 2.2 分支线性视图：`MessageService.getBranchMessages`
- 端点：`GET /topics/:topicId/messages?cursor=&limit=&nodeId=&includeSiblings=`
- 入参 cursor 是"before"语义（不含 cursor 本身），按数组位置而非时间戳
- 输出：`{ items: BranchMessage[], nextCursor?, activeNodeId }`，每个 item = `{ message, siblingsGroup? }`
- 行为：
  1. 递归 CTE 从 nodeId 沿 parentId 回溯到根
  2. cursor 在 path 中找到位置，取 `[cursorIndex - limit, cursorIndex)` 切片
  3. `includeSiblings=true` 时批查每条消息的 `siblingsGroup`（同 `(parentId, siblingsGroupId>0)` 的其他成员）

### 2.3 创建：`MessageService.create` 的 parentId 三态
| 入参 | 行为 |
|---|---|
| `undefined`（省略） | 若 topic 无消息 → 作根；若有消息+有 `activeNodeId` → 挂在 active；若有消息+无 active → 抛 INVALID_OPERATION |
| `null`（显式） | 作根；若 topic 已有根 → 抛 INVALID_OPERATION（single-root） |
| `string`（显式 ID） | 挂在指定父；校验父存在 + 同 topic |

副作用：`setAsActive !== false`（默认 true）时同步更新 `topic.activeNodeId = newMessage.id`。

### 2.4 删除：`MessageService.delete` 的两种模式
| cascade | 行为 |
|---|---|
| `true` | 递归 CTE 收集所有 descendant，一次硬删整个子树 |
| `false`（默认） | 直接子节点 `parentId` 改为被删节点的 `parentId`（reparent 到祖父）；删根节点时拒绝（必须 cascade） |

`activeNodeStrategy` 当 `activeNodeId` 落在被删子树内时生效：
- `'parent'`（默认）：回退到被删节点的 parent
- `'clear'`：直接清空（`activeNodeId = null`）

返回 `DeleteMessageResponse = { deletedIds, reparentedIds?, newActiveNodeId? }`。

### 2.5 fork：`TopicService.create({ sourceNodeId })`
**重要：fork 不复制消息树**。
- 新 topic 的 `activeNodeId = sourceNodeId`
- 原树消息**仍属于源 topic**
- 新 topic 的"对话历史"通过 `getBranchMessages(newTopicId, { nodeId: sourceNodeId })` 间接得到 —— 即从根到 sourceNode 的路径
- 也就是说：fork 是「指针重定向」而非「快照复制」

### 2.6 焦点切换：`TopicService.setActiveNode`
- 端点：`PUT /topics/:id/active-node { nodeId }`
- 行为：纯赋值 + 双校验（topic 存在、message 属于该 topic）
- **缺失功能**：`SetActiveNodeSchema` 注释（`topics.ts:67-73`）显式说："a navigator-style `descend` flag (walk down to a leaf before pinning) lives on `DeJeune/ai-service` along with its renderer consumers (`MessageGroup.tsx`, `SiblingNavigator.tsx`). It will be reintroduced when that branch lands; shipping the flag without consumers leaves an unreachable contract surface."
- 即：纯 pin 到精确 nodeId；切到子树后"自动往下走到叶子"的能力未在主分支
