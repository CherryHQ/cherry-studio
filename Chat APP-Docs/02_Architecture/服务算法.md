# Main 端聊天 Service 核心算法

> 调查日期：2026-05-20
> 范围：`src/main/data/services/{MessageService,TopicService,AssistantService,TemporaryChatService}.ts`
> 形式：每个方法给「核心步骤 + 设计取舍 + 行号锚点」，不复制代码。

## 1. MessageService.ts（871 行）

### 1.1 `getTree(topicId, { rootId?, nodeId?, depth? })` — L127–345

**六步流程：**

1. **校验 + 解析根**（L134–152）：topic 存在性；`rootId` 缺省时按 `parentId IS NULL` 查根（single-root 约束的体现）
2. **采集 active path**（L158–172）：递归 CTE 从 `activeNodeId` 回溯到根，结果存入 `Set`
3. **深度受限树查询**（L174–190）：递归 CTE 从 `rootId` 向下扩展，按 depth 归一化（`-1 → 999`、`0 → 0`、`n → n`）
4. **深度溢出补偿**（L211–258）：
   - active path 上有节点超出 depth？补回它们
   - active path 上所有节点的直接子节点？也补回（保证可"往下点一层"）
5. **构建 children map + 聚合 sibling**（L265–335）：
   - id → message、parentId → children[]、id → depth 三个映射
   - 同 `(parentId, siblingsGroupId>0)` 成员 > 1 时 → `SiblingsGroup`；单成员 → 普通 `TreeNode`
6. **递归遍历结果**（L329–338）：
   - `shouldExpand = isOnActivePath || (depth=-1 ? true : currentDepth < depth)`
   - 非 active 分支按 depth 截断；active 分支永远完整

**设计取舍：**
- depth 三态语义清晰，但「补偿步骤 4」让代码复杂；权衡：UI 侧"选中分支总能看见"vs"加载负载小"
- siblingsGroup 节点删除 parentId 字段以避免二度引用（成员里已有 parentId）

### 1.2 `getBranchMessages(topicId, { cursor?, limit?, nodeId?, includeSiblings? })` — L364–500

**关键步骤：**

1. **节点定位**（L368–383）：从 `topic.activeNodeId` 或入参 `nodeId` 出发；空 topic 返回空列表
2. **path 采集**（L385–403）：递归 CTE 从 `nodeId` 沿 `parentId` 回溯，CTE 保证 nodeId→root 顺序；ORM 重排后反向得 root→nodeId
3. **cursor 分页**（L414–426）：
   - 无 cursor：取最后 `limit` 条（最新）
   - 有 cursor：在 path 数组中找 `cursorIndex`，取 `[cursorIndex - limit, cursorIndex)`（**不含 cursor 本身**）
   - `nextCursor = path[startIndex].id`，若 `startIndex === 0` 则无下一页
4. **sibling 聚合**（L436–487）：
   - 扫 paginatedPath 中所有非零 `siblingsGroupId`，按 `(parentId, siblingsGroupId)` 去重
   - 单次批查所有 group → `siblingsMap`
   - 给每条 path 消息附 `siblingsGroup`（排除自身）

**"before cursor" 用数组位置而非 createdAt** 的好处：跨多窗口编辑造成乱序时不会跳行；缺点：cursor 必须配合一个稳定的 nodeId（不同 nodeId 切换会让 cursor 失效，但这正常）。

### 1.3 `create(topicId, dto)` — parentId 三态 — L530–638

| 入参 | 路径 | 校验 | 副作用 |
|---|---|---|---|
| `undefined` | L548–576 | 看 topic 是否有消息；有消息时必须有 `activeNodeId`，否则报 INVALID_OPERATION | parent 推为 active |
| `null` | L577–593 | 检查现存根（`parentId IS NULL`）；已有 → INVALID_OPERATION | 作根 |
| `string` | L594–610 | 父存在 + 同 topic | 挂指定父 |

通用后置（L612–632）：
- `status` 缺省 `'pending'`
- `siblingsGroupId` 缺省 `0`
- `setAsActive !== false` 默认 true → 同步更新 `topic.activeNodeId`

### 1.4 `update(id, dto)` — L646–694

- 可改字段：`data` / `parentId` / `siblingsGroupId` / `status` / `traceId` / `stats`（L681–686）
- **循环引用检查**（L652–657）：改 `parentId` 时先 `getDescendantIds(id)` 看新父是否在自己子树内；事前检查（注释 L650 解释：tx 外执行，trade-off 是 TOCTOU 风险，但 message 数量小且最坏被 DB 拒绝，不破坏数据）

### 1.5 `delete(id, cascade, activeNodeStrategy)` — L717–811

| cascade | 行为 |
|---|---|
| `true` | 递归 CTE 收集所有 descendant → 一次硬删整子树（L753–764） |
| `false` | 直接子节点 `parentId` 改为被删节点的 `parentId`（reparent 到祖父，单次 batch update L775）；删根 → INVALID_OPERATION |

`activeNodeStrategy`：当 `activeNodeId ∈ deletedIds` 时
- `'parent'`：回退到被删节点的 parent
- `'clear'`：清空

返回 `DeleteMessageResponse = { deletedIds, reparentedIds?, newActiveNodeId? }`。

### 1.6 `getPathToNode(nodeId)` — L840–868

递归 CTE 沿 `parentId` 回溯 → ORM 批查 → 按 CTE 顺序重排 → reverse → 返回 root→nodeId 数组。单次 CTE 避免 N+1。

## 2. TopicService.ts（385 行）

### 2.1 `create(dto)` — L121–160

| 模式 | 行为 |
|---|---|
| 普通 | `activeNodeId = null`，插入 topic |
| **fork**（`sourceNodeId`） | 校验 sourceNode 存在 + 未软删 → `activeNodeId = sourceNodeId` → 插入新 topic |

⚠️ **fork 不复制消息树**：新 topic 仍然是空的，仅 `activeNodeId` 是源节点的指针；消息复制语义留给上层。

`orderKey` 用 `insertWithOrderKey` 按 `topicScopePredicate(groupId)`（null → `isNull`；非 null → `eq`）计算 fractional index。

### 2.2 `listByCursor(query)` — L253–332

**两段游标 + 跨界过渡：**

1. cursor 解码（L256）：`pin:orderKey` / `topic:updatedAt:id` / `topic:`（空，表 pin 段刚好用完）三态
2. **pin 段**（L261–297）：`orderBy(pin.orderKey ASC, topic.id ASC)`；取 `limit + 1` 行判断 `hasMoreInPin`
3. **topic 段**（L299–329）：剩余配额 `remaining = limit - items.length`；Tuple 比较 `(updatedAt DESC, id ASC)`；NOT IN pinned 子查询
4. nextCursor 编码（L84–94）：根据当前段 + 边界生成

**设计取舍：**
- pin 与 unpin 独立游标 → 跨界时插入 `topic:` 空标记，下次请求知道从 unpinned 段开始
- Tuple cursor `(updatedAt, id)` → updatedAt 平局时用 id 做次序
- stale cursor 降级（L79–82）：解析失败回退到首页，不让用户卡住

### 2.3 `setActiveNode(topicId, nodeId)` — L213–244

纯赋值 + 双校验（topic 存在、message 同 topic）。无副作用。

### 2.4 `delete(id)` — L198–211

**硬删，非 FK CASCADE**：service 内显式 `DELETE FROM message WHERE topicId=id`，同步清理 `tag` / `pin` 关联（L205–206）。
TODO 在 L196：清理磁盘上的关联文件（图片、附件）—— 尚未实现。

### 2.5 `reorder(id, anchor)` / `reorderBatch(moves)` — L334–382

- 单条：按 `topicScopePredicate(target.groupId)` 作 scope
- 批量：先扫所有 id 的 `groupId`，若存在多个 groupId → **拒绝跨 scope**（L369–370）

## 4. TemporaryChatService.ts（270 行）

### 4.1 存储

```ts
topics:   Map<string, TemporaryTopicRow>
messages: Map<string, TemporaryMessageRow[]>   // 按 topicId 分组的线性数组
```

**纯内存**（非 cacheService）。进程重启即丢失。

### 4.2 `appendMessage` — L104–140

- 拒绝 `parentId` / `siblingsGroupId` / `setAsActive`（强制线性）
- 拒绝 `role = 'system'` 之外的不合规角色
- 拒绝 `status = 'pending'`（持久前必须完成）
- UUID v7 单调递增 ID，append 到数组尾

### 4.3 `persist(id)` — L168–240

1. **快照 + 立即清空 Map**（L171–177）—— 避免并发 append 与 persist 数据竞争
2. INSERT topic（保留原 ID，`orderKey` 按 groupId scope 计算）
3. 线性化插入 messages：`parentId[i] = msgs[i-1].id`（链式）
4. `activeNodeId = msgs[last].id`
5. 异常 → restore 快照（L231–236）—— 允许重试

**特点：** 临时会话不支持分支、不支持多模型 sibling、不支持 tree；persist 后才变成普通 topic 享受完整 v2 能力。

## 5. 一句话总结

**Tree / Branch / Active / Sibling 四个概念在 service 层全部就位且互相正交**：`getTree` 给可视化、`getBranchMessages` 给线性视图、`setActiveNode` 给焦点切换、`siblingsGroupId` 给多模型聚合。Fork 是"指针重定向"而非"快照复制"，是设计取舍而非缺漏。剩下的真正缺口在 renderer 消费侧（见 [分支对话.md](./分支对话.md)）。
