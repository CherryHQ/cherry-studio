# 给顾问的项目简报

## 1. 我是谁 / 项目是什么

我正在 Cherry Studio v2 fork 上做一个 AI 对话产品原型，核心功能是 **Text Anchor Branch / 文本锚点分支**：

> 用户选中 AI 回复中的一段文字 → 右键打开分支 → 在当前聊天界面右侧并列展开一个 branch conversation → AI 针对选中的那段内容继续回答。

产品目标不是普通“复制消息开新对话”，而是更接近：

- 对 AI 回复里的某句话、某个概念、某个段落继续追问；
- 主对话不离开；
- 右侧并列展开局部追问；
- 源文本有高亮，让用户知道 branch 是从哪里来的。

团队目前 2 人：

1. 我：项目负责人 / 架构判断 / 主力决策；
2. 一个低经验但听话的助手：负责低风险 UI 壳子、文案、样式、局部组件等任务。

## 3. 工作纪律 / 技术边界

项目基于 Cherry Studio v2 fork。

开发原则：

- v2 优先走 DataApi / Cache / v2 数据结构；
- UI 优先使用 `@cherrystudio/ui`、Tailwind、Shadcn 风格；
- 不要主动扩展 v1 Redux / Dexie 体系；
- Redux / Dexie 相关路径目前很多是 throwaway / 兼容层，除非为了桥接现有功能，不要把新功能深绑到 v1；
- 每次改动要小步推进，先方案、再实施、再验证；
- 不要一次性塞大改；
- 对高风险架构变更，先让我确认方案；
- Conventional Commits；
- commit 必须 `--signoff`；
- 不提交：
  - `Chat APP-Docs/.obsidian/workspace.json`
  - `Chat APP-Docs/.obsidian/graph.json`
- Claude Code 只在我明确要求时 commit（见 §2 git 政策）；
- 每次给 Claude Code 下任务时，都要明确提醒它同步更新该写的项目文档；
- 文档必须写入 `Chat APP-Docs` 对应位置，不要只在聊天里总结；
- 涉及任务状态、验证结果、已知问题、下一步计划时，至少检查并更新：
  - `Chat APP-Docs/03_Development/当前状态.md`
  - `Chat APP-Docs/03_Development/下一步.md`
  - `Chat APP-Docs/03_Development/开发日志.md`
  - `Chat APP-Docs/03_Development/会话日志.md`
  - 对应 task 目录下的 `README.md` / `验证.md` / `完成总结.md`
  - 如发现 bug，还要更新 `问题与Debug记录.md`

## 4. 当前已完成状态

目前已经完成并提交：

1. **文本选区捕获**
   - 用户可以在 assistant 回复中选中一段文字；
   - 右键菜单可以识别 messageId / blockId / selectedText；
   - 只对 assistant 文本启用 branch 相关菜单。

2. **Side-by-side Branch Panel**
   - branch 不再跳转到一个新 topic 页面；
   - 当前主对话保持在左侧；
   - 右侧并列打开 branch panel；
   - 用户可以在右侧看到 selected passage 和 branch 回复。

3. **隐藏 prompt 模板**
   - 用户不会看到系统拼接的隐藏模板；
   - branch AI 回复会围绕 selectedText 和 follow-up 聚焦回答。

4. **Regenerate / edit / delete 保持 branch focus**
   - branch 对话里的 regenerate 不会丢失 selected passage 语义；
   - 已通过 BranchAssistantContext 等方式保持局部上下文。

5. **可拖拽分隔线**
   - 右侧 Branch Panel 宽度可以拖拽；
   - 拖动时主聊天列会 reflow；
   - 宽度在 session 内保留，刷新后重置。

6. **源选区高亮**
   - 源选区高亮已用 `span.branch-anchor-highlight` wrap 方案完成并通过人工验证；
   - CSS Custom Highlight API 在当前环境中不可见，已经弃用；
   - 关闭 branch 后真实 span 可以清理到 0；
   - A → B 切换不会累积高亮 span。

7. **多分支共存（P1，已完成并入库）**
   - 同一主对话可同时打开多个分支，每个分支独立高亮配色 + 独立折叠 / 关闭；
   - 面板用 accordion 手风琴布局（单滚动区、标题贴正文）；
   - 每个分支可在 conversation 态继续追问（follow-up），路由到正确的 branch topic；
   - 关闭分支会**中止其流式回复**（不再后台空跑）。

8. **关闭去向 disposition + 保留 + 真停后端（P1-S3 + B6，已完成并入库）**
   - 关闭分支按去向路由：默认 **pending → 静默删除 fork topic**；点「保留」**kept → 留在库**；
   - delete-after-settle：删除推迟到流式 finalize 落地后，避免 404；
   - **B6-A**：修复"流式中关分支后端不真停"——abort 取靶改按存活的 abortController（不再依赖会被过早标 success 的 message.status），现已真停 Ollama（Network 请求 canceled）。

## 6. 相关已知坑 / 不要重复踩

1. **测试覆盖有假绿，别信绿灯**
   - `MainTextBlock.test.tsx` 里原来的 4 个高亮用例断言的是 `data-branch-anchored`；
   - 但 `data-branch-anchored` 全仓无人消费，没有 CSS、没有 `querySelector`、没有运行时逻辑依赖；
   - 把真正的 span wrap 高亮实现删掉，这些测试也可能继续绿；
   - `sourceHighlight.test.ts` 之前只测了 capture ↔ rebuild 算法；
   - 决定“用户能不能看见高亮”的 paint / clear 链路此前是 0 覆盖；
   - 这就是为什么之前测试全绿，但真实高亮仍然调了 13 轮；
   - 当前最高优先级是补上这条真实链路测试。

2. **CSS Custom Highlight API 不要再建议**
   - 在当前环境中 `CSS.highlights.set()` 成功，Range 也正确，但页面不可见；
   - 已经切换到 span wrap；
   - 不要再回到 CSS Custom Highlight API。

3. **关闭 branch 的 topic 去向（P1-S3 已处理）**
   - branch fork 是真实 `POST /topics`；
   - **现在**关闭按 disposition 路由：默认 **pending → 静默 `DELETE /topics/:id`**（流式则等 finalize 落地后再删，避 404）；点「保留」**kept → 留库**；
   - 仍 deferred 的只剩"历史遗留 orphan 的批量清理 / kept 列表 UI / 跨刷新持久化"（归 P2）；
   - 早期"关闭只清 UI state、不删 topic"的说法已过时。

4. **不要把 branch panel 当 active topic 跳走**
   - 早期 D-2A 方案是创建 topic 后 `setActiveTopic(newTopic)`；
   - 这不符合产品目标；
   - 现在正确方向是 side-by-side，主对话不离开。

5. **Branch message rendering 的当前结论**
   - branch panel 复用 `MessageGroup`，不是整套 `Messages.tsx`；
   - BranchPane 在 `BranchAnchorContext` Provider 外是有意设计；
   - 这样 branch 内自己的消息不会误读主对话高亮 context；
   - 不要再把这个误判为 context plumbing bug。

6. **高亮清理必须防止 DOM 污染**
   - 之前出现过空 `<span class="branch-anchor-highlight"></span>` 残留；
   - 后来 `clearSourceHighlight()` 改为 `span.replaceWith(...children)` 后通过人工验证；
   - 新测试必须覆盖这个不变量。

7. **jsdom 测试绿 ≠ 真实行为对**
   - 这个项目已经踩过“测试绿但用户看不到高亮”的坑；
   - 关键 UI 行为必须测试真实 DOM 变化，必要时仍要人工验证；
   - 不要只相信 shallow attribute 或无消费属性。
