- ✅ 不阻塞 baseline FK 二连修的手动验证
- ✅ 不阻塞 Expand Branch UI-only 原型（mock 数据驱动）

**临时绕路**：要测真实 AI 回复用别的 provider（OpenAI / Anthropic 等只要 API key 通）。

## R0.2. ⏳ v2 fresh install assistant message 写入 FK 失败（D-002）— 自动化已修复，待手动验证

**状态**（2026-05-20 深夜，T-005B 后）：T-005B 已应用 `isUniqueModelId` boundary 清洗；自动化校验全过；待用户 `rm + pnpm dev + 发消息` 端到端确认后转为「✅ 已修复」。

**原因总结**（保留为参考）：renderer 把 v1 `Model.id`（短字符串 `'qwen'`）当 v2 UniqueModelId（`"providerId::modelId"`）透传给 `POST /topics/:topicId/messages`；v2 `message.modelId` FK 指向 `user_model(id)`，要求 `::` 分隔的 UniqueModelId 且行存在；fresh install 后 `user_model` 表完全为空（无 seeder）。

**当前缓解**（T-005B，已落代码）：`StreamingService.createAssistantMessage` 在构造 DTO 时用 `isUniqueModelId(modelId) ? modelId : undefined` 清洗 —— 边界一次性处理，5 个调用点不用动。

**与 R0.1 同型号**：
- R0.1（D-001 / T-004）：`assistantId: 'default'` 是 v1 sentinel
- R0.2（D-002 / T-005B）：`modelId: 'qwen'` 是 v1 短 id
- 长期解 = 方案 C 路线：renderer 启动时合成真实 UniqueModelId（同样要求用户先配 provider）；完成后 T-004 + T-005B 的 boundary 清洗都可删

**遗留**：
- 5 个 `messageThunk` 调用点仍透传 v1 `Model.id` —— 不破坏 baseline（边界清洗兜底），但 v2 全面切换时要一起改
- `message.modelSnapshot` 仍为 NULL —— 历史追溯丢上下文，需另起任务填充

## R1. 聊天处于「DataApi 是真源、Redux 是显示缓存」过渡期 ⚠️ **2026-05-20 修正**

> 初版（2026-05-19）判断「聊天 UI 100% 仍走 Redux」**不准确**。详细订正见 [DataApi端点.md](../02_Architecture/DataApi端点.md)。

**事实（订正后）：**
- **写路径已部分落地 DataApi**：
  - `POST /topics`（`Inputbar.tsx:328`）
  - `POST /topics/:topicId/messages`（`StreamingService.createUserMessage` / `createAssistantMessage`）
  - `PATCH /messages/:id`（`StreamingService.finalize` —— 流式结束时一次性落库）
- **读路径走桥层** `services/db/DataApiMessageDataSource.ts`：拉 `GET /topics/:id/messages?limit=999&includeSiblings=true`，转 renderer 老类型后喂给 Redux
- **UI 仍订阅 Redux**（`store/messageBlock.ts:2` / `store/thunk/messageThunk.ts:2` 仍标 `@deprecated v2.0.0`）
- **流式 in-flight 状态** 在 `cacheService`（renderer 内存）—— 不走 DataApi、不持久化
- **Agent session 仍走 `dbService`**（未迁 DataApi），与普通 topic 在 `StreamingService.finalize:216` 的 `isAgentSessionTopicId` 分支分流

**风险：**
- 真源已经是 SQLite，但 **UI 还看不到** —— 删除 Redux 聊天 slice 必须在 UI 切到 `useQuery`/`useInfiniteQuery` 之后，顺序反了会黑屏
- `DataApiMessageDataSource` 已自标"临时兼容层"，但**当前所有读路径都依赖它**；删除前 UI 必须直接消费 `BranchMessagesResponse` 形状
- `POST /topics/:topicId/messages` omit `parentId` 时服务端回退到 `topic.activeNodeId` —— **多窗口下有竞态**（已有 TRADEOFF 注释在 `StreamingService.createUserMessage`）

**缓解：**
- 顺序：renderer 切 shared 类型 → UI 切 DataApi hook → 删 Redux slice + `DataApiMessageDataSource`
- 新代码创建消息总是显式传 `parentId`，逐步淘汰 omit 路径

## R2. 没有块级端点；块 ops 走整条消息替换

**事实：**
- DataApi schema 完全没有 `/messages/:id/blocks/*` 端点
- 块的增删改通过 `PATCH /messages/:id` 把新的 `data.blocks[]` 整体替换上去
- `messageThunk.ts:1332` 留有 `// TODO: Migrate block deletion to Data API when block endpoints are available`

**风险：**
- 删一个 block 要往返整条消息的 JSON（消息长时浪费带宽 + IPC payload）
- 流式中途无法原子化"追加 block / 改某块状态" —— 必须自己在 `cacheService` 暂存到 finalize
- 这是当前架构的契约取舍，未必是缺陷；但需要明确写入文档，否则未来开发会反复发明轮子

**缓解（二选一）：**
- A 补 `PATCH /messages/:id/blocks/:blockId` 细粒度端点
- B 把"DataApi 不暴露块级 RPC，块 ops 走 message data 整体替换"作为契约固化到 [DataApi端点.md](../02_Architecture/DataApi端点.md)

## R3. v1 IndexedDB 升级历史复杂（v1→v10）

**事实：** `src/renderer/src/databases/upgrades.ts` 含 v7 关键转换（消息 content → 多个 message_blocks 行）。
**风险：** 跨越多个 schema 版本的旧用户在执行 v2 迁移时，需要先把 Dexie 升到 v10 再走 `ChatMigrator`。如果 Dexie 升级链中有 bug，v2 迁移会读到不完整数据。
**缓解：** 测试 v2 迁移时必须覆盖「从最老 schema 起步」的用户场景，不只是当前 v10 用户。

## R4. UI 库切换工作量被低估

**事实：** Agent 在聊天 UI 区域 grep 到 **53 处 antd import**，加上几乎所有主容器仍用 styled-components。已迁到 `@cherrystudio/ui` 的只有 MessageHeader / MessageAnchorLine / MessageGroupModelList / Topics（仅部分组件）等极少数文件。
**风险：**
- 全量切换需要逐个对照设计稿，DESIGN.md 是强约束
- Tooltip / Dropdown / Modal / Drawer 等高频组件迁移涉及 portal / z-index / 事件冒泡差异，容易踩坑

## R5. LLM 请求由 renderer 直连第三方

**事实：** AI 调用在 renderer 完成（`AiProvider.completions()` → Vercel SDK → fetch），不经 main 进程。
**风险（设计上）：**
- API key 暴露面 = renderer 进程内存，preload 桥与 contextBridge 设置不当会泄露
- CORS / 自定义 Provider URL 若被注入恶意值，可能成为 SSRF 通道
- 网络请求绕过 main 的统一日志 / 重试 / 限流

**缓解：** CLAUDE.md 已有 `strict-url-sanitise` + `ipaddr.js` + `express-validator` 的安全约束 — 但这些主要保护 API server，**renderer 直连 LLM 这条路径**的安全审查在本次任务范围内未深入。建议单独做一次 security review。

## R6. 行号未逐一手工校验

**事实：** 本次源码地图中的行号大多来自 5 个并行 Explore agent，未逐行人工核对。
**风险：** 当未来代码位移时（refactor、PR merge），行号会偏移；引用本文档时不要把行号当合同。
**缓解：** 引用之前 grep 关键字一次（如 `selectMessagesForTopic`、`smartBlockUpdate`、`createStreamProcessor`）。

## R7. classification.json 与 codegen 产物不可手改

**事实：** CLAUDE.md 反复强调：
```
v2-refactor-temp/tools/data-classify/classification.json 是单一事实源
四个 codegen 产物（preferenceSchemas / bootConfigSchemas / PreferencesMappings / BootConfigMappings）**禁止手改**
```
**风险：** 新人不知道 codegen，直接改这四个文件，下一次 `npm run generate` 时被覆盖。
**缓解：** 任何聊天数据相关字段调整，先改 `classification.json` 或 `target-key-definitions.json`，跑 `cd v2-refactor-temp/tools/data-classify && npm run generate`。

## R8. 流式 chunk → block 更新的节流策略不透明

**事实：** `StreamingService.updateBlock()` 中存在 throttled Redux dispatch（具体节流参数未深入查看）。
**风险：** 在低端机器上 / 长输出时，throttled 太狠导致最后一段字消失；太宽松又会 jank。当从 Redux 切到 DataApi 后，节流策略需要重新设计（DataApi 写 SQLite 比 Redux 重得多）。
**待办：** 在做 R1 切换前测量 throttle 间隔 + 长文本完整输出场景。

## R9. `migrate.ts` 已有 206 个版本迁移

**事实：** redux-persist 升级链长达 206 步，单文件可能 3000+ 行。
**风险：** 任何 Redux 聊天 slice 的字段变更都要新增一条迁移函数 + 测试覆盖。一旦遗漏，用户重启会丢配置 / 报错。
**缓解：** v2 切换后 `messages` / `messageBlocks` 不在 persist blacklist 之外，所以这两条 slice 改动**不影响** migrate.ts；但 `assistants` 仍持久化，**修改 assistant 字段必须写迁移**。

## R10. 事件总线（EventEmitter）耦合面广

**事实：** `Messages.tsx` 监听了至少 `SEND_MESSAGE` / `CLEAR_MESSAGES` / `EDIT_CODE_BLOCK` / `NEW_CONTEXT` / `NEW_BRANCH`。代码里很多业务用全局 emitter 通信。
**风险：** 不在 React 数据流里，type-check 不到，新人难以追踪；多个监听器顺序敏感时容易 race。
**缓解：** 后续做事件总线清单（哪个事件谁发、谁听、payload schema），列到 `Docs/02_Architecture/`。
