## ＋ 通用功能（你漏的,需补进 V1 或明确后置）

思维导图里没有,但一个"能让人离不开"的产品通常需要。我分三档:

### 必要(V1 就该有,否则像半成品)⬜

- ⬜ **通知系统**：分支总结生成完、长任务完成、后台流式完成的提示

- ⬜ **快捷键体系**：选中→展开分支 的快捷键(高频操作必须有快捷键)
- ⬜ **历史/最近**：最近对话、最近分支、快速回到上次位置

### 锦上添花(后置)⬜

## UI / 设计的处置（回答你的问题）

你的想法是"按模块走到哪、改哪个模块的界面和图标,不想跟 Cherry 完全一样"。我的判断:

- **不要现在连 Figma,也不要单独起一个 UI skill。** 在 P0/P1 还没稳之前搞设计系统,是又一次"用规划逃避执行"。
- **你的"按模块改 UI"思路是对的**——但要做成"每个模块开发完,附带一个 UI 收尾步骤",而不是单独开一条 UI 战线。
- **务实做法**：先定一份**极简的设计基调**(主色/字体/圆角/间距/图标风格 一页纸),写进 `Chat APP-Docs`,作为约束。之后每做完一个功能模块,让 CC/低经验助手在这个基调下改该模块的界面——**界面跟着功能走,不提前做**。
- Figma 等你有了稳定的设计基调、且要做复杂视觉(比如 Timeline 视图)时再考虑,现在不需要。

## 这份清单怎么用(关键)

1. **每个模块 = 一个或几个独立工作单元 = 一个独立窗口。** 你不再"在一个对话里被分支测试整晕",而是:这个窗口只做模块3的P0,下个窗口只做模块2.2主目标,再下个只做模块4.2分支总结……
2. **顺序建议**(承接路线图):
   `模块3 P0(当前)→ 模块3 P1多分支 → 模块8 Context Controller设计 → 模块4.2分支总结+P2资产化 → 模块2.2主目标 → 模块6 Workspace分区 → 模块5 Timeline → 通用功能穿插补`
3. **这份清单 + 产品路线图 是同一套事实的两个视角**：路线图是"按优先级的时间线",这份是"按模块的全景"。两份都存 `Chat APP-Docs`,开新窗口时按需带。
