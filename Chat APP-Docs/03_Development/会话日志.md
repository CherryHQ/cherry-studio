2. **助手 — 逐文件再核验渲染链（不再信任旧 summary）**：

3. **助手 — 收敛到唯一未排除项：模块对象分裂**。`BranchAnchorContext.tsx` 单一 import path、单一 createContext 字面量、单一物理文件 → **静态层面**不可能分裂。但运行时 trace 与「树连续 + 形式正确 Provider」唯一可调和的解释是 Provider 持有的 context 对象 ≠ consumer 读到的 context 对象
4. **助手 — 找到机理**：`BranchAnchorContext.tsx` 只导出 type/常量/context/hook，**全部是非组件** → `@vitejs/plugin-react` Fast Refresh 不 eligible → 模块更改时 HMR 不能 in-place 替换、向 importers 传播重新 fetch + 重新执行该模块。本会话编辑此模块 ≥3 次（D-011 创建 / D-012 改 / D-013 加 `BRANCH_ANCHOR_DEFAULT`），每次重新执行 `createContext(...)` 产生**新 context 对象**。Chat.tsx 的 Provider fiber 与 MainTextBlock 的 `use()` 在 fiber 树各持不同代次的 context 对象 → consumer 读各自版本的 default → `insideProvider:false`，与「树连续 + 形式正确 Provider」**完全自洽**
5. **对照证据**：`BranchAssistantContext`（D-009）能用 = 本会话**未再编辑** = 一次执行无 HMR-replay → 同一 pattern 文件 + 同一 React 19 form 在「无 HMR 干扰」下工作正常。这同时反证「`<Context value>` provider 形式」与「React 19 hook 顺序」都没问题
6. **助手 — 实施修**：`BranchAnchorContext.tsx` 改为 `globalThis.__BRANCH_ANCHOR_CTX_CACHE__ ??= createCache()` 单例化 context + default（React 社区对 HMR context 分裂的标准修法）。新模块重新执行只取已 cache 的对象，**保证整个 app 生命周期一个 context object**。typecheck:web ✅

### 关键决策

| 选择 | 理由 |
|---|---|
| 单例化 context 而非把它移到组件文件 | 单例化是 1 处局部改动；移到组件文件需引一个不必要的「壳组件」、跨多文件 |
| 同时单例化 default | `insideProvider !== BRANCH_ANCHOR_DEFAULT` 引用判别器在 HMR 后仍要稳定；default 不单例化则 trace 误判 |
| **不**触碰 `BranchAssistantContext` | YAGNI —— 本会话未编辑、未表现出问题；如未来频繁编辑可同 pattern |
| 要求用户**完整重启 dev**（不只 reload）| 残留的 React fiber 仍持有旧 context 对象引用 —— 仅 reload 不能清空 |

### 诚实说明
上一轮我反复强调「全仓单一 provider + 树连续 ⇒ 源 block 必在 Provider 内」—— 这个静态推理在 normal build 下是对的，但**忽略了 dev HMR 重新执行非 FR-eligible 模块这个事实通道**。trace 推翻演绎是正确的——React 没违反 guarantee，但 React 19 的 guarantee 假设的是「同一 context 对象」，而 HMR 把这个假设破了。修是结构性的、永久的（globalThis singleton 也在 prod build 下无副作用）。
