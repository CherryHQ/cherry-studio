## 1. LOCKED 决策（已拍板，按此实现）

### 1.1 锚点重定位（anchor relocation）
- **存什么**：`{ blockId, 块内 RENDERED textContent 的字符 offset start/end }` + 一份 `selectedText` 快照。
- **重开时怎么还原**：先按 offset 重建 `Range`；offset 对不上 → **回退到对 `selectedText` 快照做字符串搜索**；都失败 → **优雅降级**（分支照常打开，只是没有 live 高亮，不报错、不挡路）。
- **强制复用，不另造**：复用现有 `sourceHighlight.ts` 的 **capture↔rebuild offset 表示**，且用**同一参照系 = rendered `textContent`（不是 markdown 源）**。P1 的高亮 capture / repaint 已经是这套（块内字符 offset over flattened text nodes）；持久化只是把这套 offset 落盘 + 读回，**不引入第二套坐标方案**。

### 1.2 存储位置（storage）—— LOCKED（独立持久化表）
- **recon 结论（Phase A，2026-06-25）**：v2 `Topic` **无** free-form JSON metadata 列；`PATCH /topics/:id` 的输入 DTO 是 `TopicSchema.pick(...).partial()`、派生自 `z.strictObject`，对未知键**直接 422 拒绝**（连静默丢弃都不是），且 service 为 column-bound 更新、读回为固定字段序列化——**「既有 topic metadata PATCH 往返」路线已排除**（证据见 §2 Q1）。
- **裁决（Sam，2026-06-25）= 独立持久化表，不给 topic 加 metadata**。新建 **`branch_anchor` 表**，**`parentTopicId` 为主索引**。**目标**：刷新 / 重访 parent topic 时**一次读取**该 topic 名下全部 kept 分支 + 锚点。fork 不带回指（见 §2 Q3），故必须显式存 `branchTopicId` 才能在重访时重新挂回。

**`branch_anchor` 表结构（13 列，LOCKED）**：

| 列 | 类型 / 说明 |
|---|---|
| `id` | 主键 PK |
| `parentTopicId` | **主索引（indexed）** —— 按它一次取全 |
| `branchTopicId` | kept 分支自身 topic id（重挂回用）；**唯一约束**防止同一 kept branch topic 重复 persisted anchor |
| `messageId` | 源消息 id；**有意不设 FK** |
| `blockId` | 源块 id |
| `selectedText` | 选区文本快照 |
| `selectionStart` / `selectionEnd` | 块内字符 offset（渲染 textContent 参照系，§2 Q2） |
| `disposition` | default `'kept'`（**前向扩展、非承重**） |
| `summary` | `TEXT NULL` —— 摘要**并入本表**，不另起表 |
| `summaryUpdatedAt` | `INTEGER NULL` |
| `createdAt` / `updatedAt` | 时间戳 |

- **summary 并入本表**（不另起表），经 **PATCH 更新既有行**。
- **写入时机 = write-on-keep**：只有用户明确 Keep 的分支才创建 `branch_anchor`；**pending branches 不持久化**。
- **不存 topic metadata**：Topic 没有可写 free-form JSON metadata，本方案不再绕回 topic metadata 路线。
- **`branchTopicId` 唯一**：同一个 kept branch topic 只能持久化一条 anchor，真实 DB integration test 已覆盖 duplicate `branchTopicId` 只保留一行。
- **`messageId` 无 FK 是有意设计**：`messageId` 只是父 topic message tree 的 snapshot reference；`blockId` 在 message data JSON 内，不是表。给 `messageId` 单独加 FK 会制造半截完整性错觉，无法覆盖 source message deletion / `blockId` drift / offset mismatch；这些读回异常交给后续 hydration degradation 处理。
- **迁移机制（Step 0 已以 CLAUDE.md 原文确认，2026-06-25）**：建新表 = 改 `src/main/data/db/schemas/**` + 跑 `pnpm db:migrations:generate`，是 **CLAUDE.md 获许路径**（第 157 行：`schemas in src/main/data/db/schemas/, migrations via pnpm db:migrations:generate`；第 212 行：schemas `"may change freely"`，迁移产物每次 schema 变更由 `drizzle-kit generate` 重生成）。**「不写 patch migration」红线精确禁的是**：手写一份 patch migration 去"修"开发期 DB drift（第 212 行 `do not author patch migrations to "fix" it`，it = mid-development drift）——**不禁**"改 schema 后重生成迁移"。**故本方案合规**。
- **⚠ 术语来源澄清（Step 0 副产，2026-06-25）**：本项目文档常说的「`src/main/data` 不改既有内部 / 只许 CALL」与「`sourceHighlight.ts` / `BranchAnchorContext.tsx` / `useBranchFork.ts` / `newMessage.ts` 等保护文件清单」，均为**本项目自管的工程纪律**（源自本 P2 文档 + Sam 裁决），**不是 CLAUDE.md 条文**。Step 0 核实：CLAUDE.md 全文**无**此类保护/冻结清单；唯一「禁手改」清单是第 218–223 行的四个 auto-generated 文件（`preferenceSchemas` / `bootConfigSchemas` / `PreferencesMappings` / `BootConfigMappings`，要改走 `classification.json` + regenerate），**与 P2 无关**。**结论：这些纪律依然有效、依然要守——但别把它们当成 CLAUDE.md 律条引用**。
- **实现纪律**：Step 1 已按 schema + generated migrations + DataApi handler/service 路线落地。保护文件仍只经其**现有导出 / 公开接口**复用（见 §2 Q2）；此处「保护文件」依据的是上一条所述的**项目纪律**，非 CLAUDE.md。

### 1.3 重开（reopen）
- 点击一处**源高亮** → 把**那一个** kept 分支 topic 重新加回**当前 BranchPane**；
- **按分支、点击触发** —— **不**在切 topic 时**自动加载全部 kept 分支**；
- 加回后 **best-effort 重建 live 高亮**（按 1.1 的 offset→快照搜索→降级三级）。

### 1.4 摘要（summaries）
- **先做手动触发**：分支卡片上一个按钮，点了才生成（不自动、不在每条回复后偷偷算）；
- **复用该分支自己的模型**（不另挑模型）；
- **显示在卡片上** → 折叠时也能看到分支结论；
- **与锚点一起持久化**（同一存储决策，见 1.2）。
