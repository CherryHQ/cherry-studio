## 3. 关键决策史 —— S6' 源选区高亮 D-010 → D-013 → span 回退

**这一节最重要**。S6' 走过四轮失败 + 一次架构回退。**目的：阻止下一个会话又去试 CSS Custom Highlight API、撞同一堵墙**。

完整决策史也在 [问题与Debug记录.md](./问题与Debug记录.md) D-010~D-013-CLOSED 与 [tasks/T-006_TextAnchorBranchUI/T-006E_Highlight/README.md](./tasks/T-006_TextAnchorBranchUI/T-006E_Highlight/README.md) 顶部「不要再走的弯路」red list 里。

### D-010：accent token 白底不可见

- **方案**：`block.id === highlightedBlockId` 时给 MainTextBlock wrapper 加 `bg-accent/60`
- **现象**：选中后看不见
- **根因**：`accent` 是 DESIGN.md 中性近透明 token（≈3% 黑叠加），白底上肉眼几乎看不见
- **修**：换 `warning` 色族（暖琥珀）—— 颜色解决
- **教训**：**不要**用 `accent` 做反馈色；用 `warning`/`destructive`/concrete 颜色

### D-011：块级高亮等于整段高亮

- **现象**：颜色对了，但**整条助手回复**被高亮、不是只选中的那一段
- **根因**：本项目里**一整条助手回复 = 单一 MAIN_TEXT block**（`Blocks/index.tsx:193-194` MAIN_TEXT/CODE/UNKNOWN 都路由到 MainTextBlock）。block 级 className 等于整段亮
- **修**：放弃 block 级 className，改实现**精确字符范围高亮**。引入 `BranchAnchor.selectionStart/End` + `captureSelectionOffsets` + `resolveBranchHighlightRange`，画法用 **CSS Custom Highlight API**（`CSS.highlights` + `Highlight` + `::highlight()`）
- **教训**：**不要**用 block 级别 className 做精确反馈；本项目 1 reply = 1 MAIN_TEXT block

### D-012：CSS Custom Highlight API 注册成功但零像素

- **现象**：API `CSS.highlights.has(name) === true`，但页面看不见
- **怀疑（A）**：offsets 漂移（capture 用 `Range.toString().length`、rebuild 用 text-node `.length` 累加 —— 两条遍历可能在跨节点 markdown DOM 上不等）
- **怀疑（B）**：颜色不可见（`::highlight()` 用 `var(--color-warning-bg)`，amber-50 太淡 + `var()` 在 highlight pseudo 内解析不稳）
- **加固**：① 两侧统一改用 `flattenTextNodes`（消除 A）；② 颜色换 concrete `rgb(251 191 36 / 0.45)`（消除 B）；③ 加 rAF 兜底重绘
- **结果**：仍不显示 → 进入 D-013 instrumentation

### D-013：盲修循环 → 全路径 trace → CSS Custom Highlight API 在本 env 不出像

- 三轮 blind-fix 全失败后，**停止猜测、只加 instrumentation**
- 5 阶段 reader trace（`[S6 trace] effect fired / block element / range resolve / paint called / CSS.highlights state`）
- 多轮误诊（双 source / Provider 错位 / id 漂移 / HMR context 分裂）被自身后续 trace 逐个推翻
- 加 `insideProvider` 引用判别器 + 定向比对后实测：**选中块** `insideProvider:true / matched:true / earlyReturn:null / Range 完美 / Highlight registered`，**仍零像素**
- 残留唯一失效点：`paintSourceHighlight` 体内 `paint detail` trace 显示 `rangeResolved:true / rangeCollapsed:false / rangeText 与选中段精确一致 / startContainer endContainer 都是 #text / afterSet.has:true / size:1` —— **API 注册成功，但本 Electron/Chromium + markdown DOM 环境下零渲染**
- **结论**：触发既定 stop-loss，**放弃 CSS Custom Highlight API**

### D-013-FIX-FINAL：切换 `<span>` 包裹

- 保留 capture / resolve / context wiring 全部上游不动（trace 已证全部正确）
- 只换最终 paint：`new Highlight(range)` + `CSS.highlights.set()` → `wrapRangeWithSpans(blockEl, range)` 跨节点 `<span>` 包裹
- 不能用 `range.surroundContents()` —— 选区跨 `<strong>`/`<em>`/`<code>` 等 markdown 元素边界会抛 `INVALID_STATE_ERR`
- 算法：`flattenTextNodes` 切片得到 in-range Text 集合 → 逐节点 `Text.splitText(s)` + `Text.splitText(e)` 剥出 in-range 子串 → `parent.insertBefore(<span>) + span.appendChild(target)` 包裹

### D-013-CLEANUP-LEAK-FIX：旧 cleanup 留 shell + leftover

- 切换后 DOM 里发现两类残留：(a) 完整未清的旧 span（cleanup 没碰到它）；(b) 空 shell `<span class="branch-anchor-highlight"></span>`（子节点搬走但 span 元素留了）
- 根因：旧 `clearSourceHighlight` 走 `while/parent.insertBefore(span.firstChild) + parent.removeChild(span)` **两步**操作 —— React reconciliation 在两步之间动 DOM 时 race 出 shell；某些 cleanup timing 错位下整个 cleanup 都没跑到
- **修**：改用**单原子 DOM 原语** `span.replaceWith(...Array.from(span.childNodes))`。同时移除 span 与插入子节点、无中间状态。`document.querySelectorAll` 全文搜不依赖任何存储引用（React 重渲后旧 ref 失效是 leftover 的真因）。`parent.normalize()` 合并 `splitText` 切出的相邻 Text 节点 → DOM 字节级复原

### D-013-CLOSE-PATH：关闭路径连通

- Conversation 状态下 X 按钮被旧逻辑 `disabled={!isComposing}` 锁死，已 fork 后无法关 → 前端 state 永挂 → 高亮永不清
- **修**：(1) `BranchPane.tsx` 删 disable + tooltip 统一；(2) `Chat.onComposeCancel` 扩展为 `clearSourceHighlight() + setBranchAnchor(null) + setBranchTopic(null) + branchFork.reset()`（先同步擦 DOM，避免 panel collapse 动画前的视觉闪烁）；(3) 测试反转断言

| 方案 | 推翻原因 |
