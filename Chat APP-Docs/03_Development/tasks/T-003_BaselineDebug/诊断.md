# 问题与 Debug 记录

## D-001 v2 fresh install 下无法创建 Topic（FK constraint failed）

- **日期**：2026-05-20
- **任务**：T-003 / Phase 2 Baseline Debug
- **复现方式**：删除 `~/Library/Application Support/CherryStudioDev` → `pnpm dev` → 启动后点击新建 topic
- **状态**：✅ 根因定位完成；❌ 未修复（本轮只读不改代码）

### 现象

```
[useAssistantPresets] Assistant preset with id default not found in state.
[dataApiService] POST /topics body: { name: 'Default Topic', assistantId: 'default' }
[topicService] SQLITE_CONSTRAINT_FOREIGNKEY: FOREIGN KEY constraint failed
[topicService] Topic with id ... not found
```

### 根因（一句话）

**Renderer 端 `AssistantService.ts:77-89` 的 `getDefaultAssistant()` 仍然合成一个 `id: 'default'` 的字面量 sentinel；v2 后端把这串字符串当真 UUID 写到 `topic.assistantId` FK 上，SQLite 没有 `id='default'` 的 assistant 行 → 拒绝。**

`AssistantMigrator.ts:108-167` 注释里写得很明白：「v2 has no system-reserved 'default' row」。意思是 v2 在迁移时把 v1 的 `'default'` 转成 UUID，**但 renderer 自己合成的那一份没有同步改造**。`mapLegacyTopicToDto`（L173）头上有 `// TODO: remove it in v2`，正是这块未做完的工作。

### 完整链路

```
1. AssistantService.ts:77-89   getDefaultAssistant() → { id: 'default', ... }     ← v1 sentinel 源头
                                                ↓
2. store/assistants.ts:39-40   initialState.defaultAssistant = getDefaultAssistant()
                               initialState.assistants = [getDefaultAssistant()]
                                                ↓
3. useAssistant.ts:196-198     const defaultAssistant = useAppSelector(...) 
                                                ↓ (UI 用 defaultAssistant.id = 'default')
4. Inputbar.tsx:324-330        getDefaultTopic(assistant.id)             ← assistant.id = 'default'
                               dataApiService.post('/topics', {
                                 body: mapLegacyTopicToDto(newTopic)     ← { assistantId: 'default' }
                               })
                                                ↓
5. AssistantService.ts:173     mapLegacyTopicToDto pass-through assistantId   ← TODO: remove in v2
                                                ↓
6. main/data/api/handlers/topics.ts:29  POST /topics → topicService.create
                                                ↓
7. SQLite INSERT topic (assistantId='default')
                                                ↓
8. ❌ FK constraint failed (assistant 表没有 id='default' 行)
```

### 旁证

- `src/main/data/db/seeding/index.ts:12-17`：注册的 seeder **不含 AssistantSeeder**（只有 Preference / TranslateLanguage / PresetProvider / MiniApp）。Fresh install 后 `assistant` 表为空。
- `AssistantMigrator.ts:108-167`：v1 → v2 迁移时显式把 `id === 'default'` 重映射为 UUID。但这条只在「v1 迁 v2」时生效，**fresh install 路径不经过迁移器**。
- `ChatMigrator.ts:659-665`：注释复述 "v2 has no system-reserved 'default' row"。
- `useAssistantPresets.ts:45` 的 warning 出现是同因 —— Redux `presets: []` 空数组里查 `'default'` 必然找不到。

### 为什么 `rm` 没用

`rm` 清的是 **SQLite 数据库文件**（`~/Library/Application Support/CherryStudioDev/`）；问题不在数据，在 **renderer 端的初始 Redux state**：每次启动都重新执行 `getDefaultAssistant()`，永远生成 `id: 'default'` 字面量。即使你再清 localStorage（key `cherry-studio`），`initialState` 也是硬编码这串字符串。所以「fresh install」越干净，反而越能稳定复现这个 bug —— 干净到没有迁移路径帮你把 'default' 转成 UUID。

### 修复方案（3 选 1）

#### 方案 A：fresh install 时 seed 一个真实 default assistant
- **改动范围**：新增 `src/main/data/db/seeding/seeders/assistantSeeder.ts`；在 `seeding/index.ts` 注册。
- **是否符合 v2 数据模型**：❌ 不符合。`AssistantMigrator.ts:340-342` 与 `ChatMigrator.ts:350-352` 都明确写了 "v2 has no system-reserved 'default' row"。Seed 一行 id=固定值的 assistant 等于把 v1 sentinel 模式搬进 v2，违背了显式设计决策。
- **风险**：
  - 这条 seed row 永远不会被用户删除（删了又会重新 seed？要看 seeder 设计）
  - 多 device 同步 / 多 profile 时会冲突
  - 让 v2 数据模型"骨架不干净"
- **是否适合 baseline**：❌ 不推荐 —— 不可逆地污染数据模型，未来要拔出来更难。

#### 方案 B：renderer 在转 DTO 时把 `'default'` 当作 null
- **改动范围**：`src/renderer/src/services/AssistantService.ts:173-178` 一行 —— `mapLegacyTopicToDto` 改成 `assistantId: topic.assistantId === 'default' ? null : topic.assistantId`。
- **是否符合 v2 数据模型**：✅ 符合。Topic.assistantId 在 v2 schema 是可空 FK（`SET NULL on delete`），ChatMigrator 也在 `assistantId='default'` 找不到映射时退化为 null（"orphan topic"）。
- **风险**：
  - Topic 创建出来后 `assistantId=null` —— UI 显示"无助手"或回退到运行时合成 defaultAssistant
  - 需要确认 chat completion 路径（`fetchChatCompletion`、`AiProvider.completions`）不直接读 `topic.assistantId` 决定模型；当前看是从 Redux defaultAssistant / 用户 mentions 取的，不依赖 topic 字段
  - **副作用**：topic 没有 assistant 关联，再次打开时 UI 没法回填"这个会话用哪个助手"。但 fresh install 阶段本来就没有助手可选，问题暂时不显著
- **是否适合 baseline**：✅ **推荐**。一行改动，符合 v2，可逆，不污染数据。

#### 方案 C：renderer 启动时把 `'default'` runtime sentinel 映射到真实 UUID
- **改动范围**：在 renderer 启动序列（`entryPoint.tsx` / `init.ts`）里：
  1. 调 `GET /assistants?limit=1`
  2. 若空 → `POST /assistants { name: '默认助手', ... }` 拿到 UUID
  3. `dispatch(updateDefaultAssistant({ assistant: { ...synthesized, id: realUuid } }))`
  4. 把 `getDefaultAssistant()` 标记为 deprecated，新代码用 `store.getState().assistants.defaultAssistant`
- **是否符合 v2 数据模型**：✅ 最符合 —— 严格匹配 `AssistantMigrator` 注释里"v2 在 UI 层运行时合成 default"的设计意图
- **风险**：
  - 启动时序复杂：DataApi 必须先 ready，再读/写 assistant，再 dispatch；要在 lifecycle 里仔细排
  - 第一次启动写一行；以后启动读一行 —— 多窗口启动可能 race（两个窗口同时发现空表都写）
  - 现有用 `getDefaultAssistant()` 同步获取 sentinel 的几十处调用都要审一遍（特别是 `getDefaultTranslateAssistant` 等非聊天路径）
- **是否适合 baseline**：⚠️ 适合作为「正式修复」，但作为 baseline 来 unblock UI 原型开发**过重**。

### 推荐路径

| 阶段 | 方案 | 原因 |
|---|---|---|
| **Baseline（现在）** | **B** | 一行；符合 v2；可逆；让你能继续做 Expand Branch UI 原型 |
| 后续正式修复 | **C** | 干净对齐 v2 设计；和 v2 数据迁移团队的注释意图一致 |
| **不要做** | A | 与 `AssistantMigrator.ts:340-342` / `ChatMigrator.ts:350-352` 明确的设计取向冲突 |

### 验证修复（B 方案）后应该出现的现象

- `POST /topics body: { name: '...', assistantId: null }`
- topic 行被 INSERT 成功，无 FK 错误
- UI 进入 topic 后，`useAssistant(topic.assistantId)` 返回 undefined → 应该 fallback 到 `defaultAssistant`（runtime sentinel）—— 如果 fallback 没写好可能 UI 显示空，需要测一下

### 不阻塞的开发

Expand Branch UI 原型工作**可以继续进行**：
- 分支 UI（`MessageGroup` / `SiblingNavigator`）的渲染逻辑不依赖 topic 创建成功
- 可以用 mock `BranchMessagesResponse` 数据驱动组件开发
- 等 baseline 修好后再接入真实 `useInfiniteQuery('/topics/:topicId/messages')`

但请**在分支 UI 代码里避免**：
- 直接调用 `dataApiService.post('/topics', ...)` 触发 baseline bug
- 假设 `defaultAssistant.id` 是真实 UUID
