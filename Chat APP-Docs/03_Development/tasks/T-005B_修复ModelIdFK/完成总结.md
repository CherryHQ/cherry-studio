## 经验

1. **「集中清洗一处 vs 五处」的取舍**：CLAUDE.md「Surgical Changes」+ Simplicity First 都倾向「最少触动面」。这里 StreamingService 是聊天数据写入 v2 DataApi 的唯一边界点，做边界清洗等于做契约适配 —— 不仅简洁、还有「未来新调用点也自动安全」的额外保障。
2. **`isUniqueModelId` 是宽容判定**（只看是否含 `::`），刚好能拒绝所有 v1 短 id，又不会误伤真 UniqueModelId。这种「在边界做最便宜的格式判定」比「让 caller 改类型」更现实。
3. **跟 T-004 是同型 bug**：v1/v2 边界处的「短 id / sentinel 直接送 FK 校验」类问题，集中爆发在 baseline 阶段；可能还会有 T-005C/D 同源（例如 useAssistantPresets / modelSnapshot 等路径），需要边修边记。
4. **修一层暴露下一层**：T-004 修了 topic 创建，T-005 修了 message 创建；下一层可能是 AI 调用本身（需要真实 provider/API key）—— 这不算 bug，是用户操作流程。
