# T-005A 诊断

## 1. message 表的所有 FK

`src/main/data/db/schemas/message.ts:16-58`：

| 字段 | 引用 | onDelete | 关键性 |
|---|---|---|---|
| `topicId` | `topic(id)` | CASCADE | NOT NULL，必须存在 |
| `parentId` | `message(id)`（self-ref） | SET NULL | 可空 |
| `modelId` | `user_model(id)` | SET NULL | 可空；schema 注释明确 id 是 `"providerId::modelId"` UniqueModelId |

旁证：`message.ts:37-38` 原注释：
```ts
// Model identifier: FK to user_model(id) — UniqueModelId "providerId::modelId"
modelId: text().references(() => userModelTable.id, { onDelete: 'set null' }),
```

## 2. 三个候选字段逐个排查

### 2.1 `parentId`（用户消息 id）—— **正常**
- 链路：用户在 T-004 之后已能成功创建 user message（POST `/topics/:topicId/messages` 带 `role=user`）—— 因为 user message DTO 不带 modelId（见 `StreamingService.createUserMessage` 第 535-554 行），所以**没踩 modelId FK**，成功入库 ✅
- 紧接着发送 assistant message 时 `parentId = userMessage.id` —— 该 id 是上一次 POST 返回的真 UUID，**真实存在于 message 表**
- 此 FK 的 self-ref 是 `SET NULL`，即便对不上也只会 silent null，不会抛 FK 错
- → **不是 parentId 的锅**

### 2.2 `topicId`（当前 topic）—— **正常**
- T-004 修复后 topic 能创建成功（用户原话："Topic 创建成功，用户消息可以显示"）
- 同一次会话里 user message 写入成功 → `topic.id` 确实存在
- `topicId.notNull()`，但 topic 存在，不会失败
- → **不是 topicId 的锅**

### 2.3 `modelId = 'qwen'` —— **🎯 命中**
- 注释要求格式：`"providerId::modelId"`（**带 `::` 分隔符**）
- 实际值 `'qwen'` **不含 `::`**
- → 即便 user_model 表里碰巧有一行 id=`'qwen'`（不会有），格式也不对
- → user_model 表 fresh install 后**完全为空**（见 §3.3 验证）
- → SQLite 拒，**就是这个字段**

## 3. 根因证据链

### 3.1 v1 vs v2 model id 的语义裂痕

`packages/shared/data/types/model.ts:82-86`：
```ts
export const UNIQUE_MODEL_ID_SEPARATOR = '::'
export type UniqueModelId = `${string}${typeof UNIQUE_MODEL_ID_SEPARATOR}${string}`
```

`packages/shared/data/types/model.ts:96-98`：
```ts
export function isUniqueModelId(value: unknown): value is UniqueModelId {
  return typeof value === 'string' && value.includes(UNIQUE_MODEL_ID_SEPARATOR)
}
```

v1 renderer `Model` 类型（`src/renderer/src/types/index.ts:321-324`）：
```ts
export type Model = {
  id: string         // ← 例如 'qwen'、'qwen-turbo'，**短字符串**
  provider: string   // ← 例如 'dashscope'
  name: string
  ...
}
```

v1 设计上 `Model.id` + `Model.provider` 是**两个独立短字段**；v2 把它们合并为单个 `UniqueModelId = "${provider}::${id}"`。renderer 代码大部分仍是 v1 形态，把 `model.id`（短字符串）当 UniqueModelId 透传。

### 3.2 调用面

`grep createAssistantMessage src/renderer/src --include='*.ts'`：

| 文件:行 | 传入 modelId 的值 |
|---|---|
| `src/renderer/src/store/thunk/messageThunk.ts:802` | `mentionedModel.id`（mention 选的 v1 Model） |
| `src/renderer/src/store/thunk/messageThunk.ts:1062` | `assistant.model?.id`（默认 model） |
| `src/renderer/src/store/thunk/messageThunk.ts:1266` | （另一处分支） |
| `src/renderer/src/store/thunk/messageThunk.ts:1318` | （regenerate 路径） |

→ **5 处调用点都直接透传 v1 `Model.id` 短字符串**。

### 3.3 默认 model 来自硬编码 SYSTEM_MODELS

`src/renderer/src/store/llm.ts:67-71`：
```ts
export const initialState: LlmState = {
  defaultModel: SYSTEM_MODELS.defaultModel[0],
  topicNamingModel: SYSTEM_MODELS.defaultModel[1],
  ...
}
```

`SYSTEM_MODELS.defaultModel[0]` 来自 `@renderer/config/models` 的**静态 v1 形态 Model 对象**（`id` 是短字符串）。fresh install 时用户没添加任何 provider/model，但 `Llm.defaultModel` 已经被 v1 静态对象填充。

`assistant.model?.id` 实际上是 `SYSTEM_MODELS.defaultModel[0].id`，比如 `'qwen'`。

### 3.4 user_model 表 fresh install 为空

`src/main/data/db/seeding/index.ts:12-17`：
```ts
export const seeders: ISeeder[] = [
  new PreferenceSeeder(),
  new TranslateLanguageSeeder(),
  new PresetProviderSeeder(),
  new MiniAppSeeder()
]
```

→ **没有 UserModelSeeder**。
`grep -rn 'userModel\|user_model' src/main/data/db/seeding/` 0 命中。
fresh install + 用户未手动添加 model → `user_model` 表是 **完全空**。

→ 任何 `INSERT INTO message (modelId=...)` 都必然 FK 失败（除非 modelId=NULL）。

### 3.5 user message 为什么不挂

`StreamingService.createUserMessage`（约 L535-554）的 DTO：
```ts
const createDto: CreateMessageDto = {
  role: 'user',
  data: { blocks: dataBlocks },
  status: 'success',
  traceId: message.traceId ?? undefined
  // ← 没有 modelId 字段
}
```

user message DTO 不带 modelId → `message.modelId` 默认 NULL → FK 不触发。这解释了用户观察：「用户消息能显示，但 assistant 消息挂了」。

## 4. 总结：bug 是什么

**`messageThunk` 把 v1 短字符串 `model.id`（如 `'qwen'`）当 v2 UniqueModelId（`"providerId::modelId"`）传给 `POST /topics/:topicId/messages`**，message 表 modelId FK 指向 user_model(id)，而 user_model 表 fresh install 后空空如也，必然 FK 失败。

是和 T-004 同一类型的「v1 短 id / sentinel 直接发给 v2 FK 校验」错误。

## 5. 最小修复方案

### 推荐：方案 B-style（一处集中清洗）

改 `src/renderer/src/services/messageStreaming/StreamingService.ts` 的 `createAssistantMessage`（约 L571-591），在构造 DTO 时清洗 modelId：

```ts
// before
const createDto: CreateMessageDto = {
  parentId,
  role: 'assistant',
  data: { blocks: [] },
  status: 'pending',
  siblingsGroupId,
  modelId,
  traceId
}

// after
const createDto: CreateMessageDto = {
  parentId,
  role: 'assistant',
  data: { blocks: [] },
  status: 'pending',
  siblingsGroupId,
  // v1 Model.id is a short string like 'qwen'; v2 user_model.id is a UniqueModelId
  // "providerId::modelId". If the caller hasn't migrated yet, drop the value rather
  // than send a non-existent FK reference. See [[T-005A]] / D-002.
  modelId: isUniqueModelId(modelId) ? modelId : undefined,
  traceId
}
```

`isUniqueModelId` 已经在 `packages/shared/data/types/model.ts:96` 提供，加一行 import 即可。

### 为什么集中在 StreamingService 而不是 5 个调用点

- 一处改 vs 5 处改 —— 简单度差 5 倍
- 未来加新调用点也不用再记得清洗
- `isUniqueModelId` 是宽容判定（只看是否含 `::`），不会误杀真 UniqueModelId
- 副作用：modelId=undefined → DTO schema 允许（optional）→ message.modelId 落库为 NULL → 不阻塞流式继续；UI 端 `convertSharedToRendererMessage` 已经把 `shared.modelId ?? undefined` 接住了（约 L619）

### 副作用评估

| 副作用 | 影响 | 缓解 |
|---|---|---|
| message 行 modelId=NULL | DB 查询时无法 JOIN user_model | renderer 内存里 `message.model` 字段仍有完整 v1 Model 对象；UI 显示不依赖 DB join；只是统计/分析查询会少这一维 |
| `message.modelSnapshot` 也是 NULL | 历史回溯时丢失模型上下文 | v2 设计期望 caller 填 modelSnapshot；当前 renderer 没填，是已知遗留，不属于 T-005A 范围 |
| 是否破坏 `messageStats` 链路 | 不影响（stats 是独立 JSON 字段） | — |

### 不推荐的替代方案

- **方案 A：seed 一些静态 user_model 行** —— 同 T-004 的方案 A，污染数据模型；user_model 应该是用户加 provider 后填，不是系统硬塞
- **方案 C：每次 POST 前 upsert user_model** —— 引入隐式数据创建；现在 fresh install 时 user_provider 表也空，连 FK provider_id 都对不上
- **方案 D：在 5 处调用点逐个清洗** —— 同等效果但 5× 代码量

## 6. 是否阻塞 Expand Branch UI 原型

**不完全阻塞，但建议先修。**

- 分支对话的 UI 渲染（`MessageGroup` / `SiblingNavigator`）逻辑**不依赖 assistant message 写入成功** —— 可以用 mock `BranchMessagesResponse` 数据驱动组件开发
- 但任何**端到端联调**（点击发送 → 看 AI 回复进 sibling group）都会立即撞这个 bug
- 修复 = 一行 + 一个 import；成本极低
- → **强烈建议先花 5 分钟修了再做 UI，避免反复切换 mock / 真实数据**

## 7. 关联同源问题（不在本任务范围）

修了 T-005A 之后仍可能出现以下症状（同源短 id / sentinel 问题，分开 ticket 处理）：

1. **`assistant.id = 'default'` warning** —— useAssistantPresets 找不到，与 T-004 同源；纯 warning 不阻塞
2. **`modelSnapshot` 写入** —— 即使修了 modelId=NULL，未来想保留"那条消息是哪个模型回的"还需要在 DTO 里补 modelSnapshot
3. **`user_provider` / `user_model` 完全空** —— 用户必须手动加 provider 才能真正发消息；这是 v2 设计，不是 bug
4. **AI 调用本身** —— 即使 message 行能落库，`fetchChatCompletion` 还要拿到真实 provider config + API key 才能发请求；fresh install 用户大概率会先卡在 provider 配置上
