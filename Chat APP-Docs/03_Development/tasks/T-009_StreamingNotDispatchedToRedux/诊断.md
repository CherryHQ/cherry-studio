# T-009 诊断：D-004 / D-005 同源 — 流式管线不回灌 Redux

## 1. 根因（一句话）

**整条流式管线（StreamingService + BlockManager + textCallbacks + baseCallbacks）只写 `cacheService` 内存 + DataApi/SQLite，完全没有 dispatch 到 Redux `messageBlocksSelectors` / `messages` slice。但消息渲染层（`MessageBlockRenderer`）仍然只读 Redux。** 所以 assistant 的 message + blocks 在 Redux 里永远是 fresh-create 状态：`blocks: []`、`status: PENDING`。

这是和 T-008C 同款的 v1/v2 数据断层，只是这次发生在消息渲染层而非 picker。

## 2. 现象映射

### D-004：Ask/Open 永远 disabled

- 用户 DevTools 跑 `$$('[data-message-id][data-block-id]').map(el => ({role: el.dataset.messageRole, ...}))`
- 只返回 `role: 'user'` 的 wrapper
- 没有任何 assistant 的 wrapper

**为什么**：

| 链条 | 状态 |
|---|---|
| Redux `state.messages.entities[assistantMsgId].blocks` | `[]`（fresh-create 后从未更新） |
| `MessageBlockRenderer` 的 `renderedBlocks = blocks.map(id => blockEntities[id]).filter(Boolean)` | `[]`（没有 ids 可查） |
| 跑 switch case `MAIN_TEXT` → `<MainTextBlock messageId={...} role={message.role} />` | **永远不进入** |
| MainTextBlock wrapper `<div data-message-id data-block-id data-message-role>` | **不存在于 DOM** |
| `findBlockContext(range)` | 永远返回 null |
| `SelectionContextMenu.hasAnchor = ... && blockContext !== null` | 永远 false |
| Ask / Open 菜单项 | 永远 disabled |

### D-005：3 个点（BeatLoader）永远转

- 用户：assistant 文本可见、后端 PATCH success、但底部 BeatLoader 不消失

**为什么**：

| 链条 | 状态 |
|---|---|
| Redux `state.messages.entities[assistantMsgId].status` | `PENDING`（fresh-create 时设置；流式完成后未更新） |
| `Blocks/index.tsx:123` `const isProcessing = isMessageProcessing(message)` | 永远 true（`status === PENDING` 满足 [`is.ts:165`](../../../src/renderer/src/utils/messageUtils/is.ts) 条件） |
| `Blocks/index.tsx:253-265` 末尾的 PlaceholderBlock | 永远渲染 |
| [`PlaceholderBlock.tsx:10-16`](../../../src/renderer/src/pages/home/Messages/Blocks/PlaceholderBlock.tsx) `<BeatLoader />` | 永远显示 |
| MessageFooter / 操作栏 | 因为 `message.status` 还在 `ing` 状态，UI 仍处于 streaming 视图 |

## 3. 流式管线调用链（实测）

| 阶段 | 文件 / 行 | 写入位置 |

| throttledBlockUpdate | [`messageThunk.ts:495-518`](../../../src/renderer/src/store/thunk/messageThunk.ts) | 注释明确：「Updated to use StreamingService.updateBlock instead of Redux dispatch」`streamingService.updateBlock(id, blockUpdate)` |

| streamingService.updateBlock | [`StreamingService.ts:331-361`](../../../src/renderer/src/services/messageStreaming/StreamingService.ts) | 同样只 `cacheService.set`，无 dispatch |
| baseCallbacks.onComplete | [`baseCallbacks.ts:290-388`](../../../src/renderer/src/services/messageStreaming/callbacks/baseCallbacks.ts) | `streamingService.finalize(assistantMsgId, status)` → DataApi PATCH `/messages/:id` 写 SQLite + `EventEmitter.emit('MESSAGE_COMPLETE', ...)` |
| MESSAGE_COMPLETE 订阅 | grep 全仓只有 emit，**0 个订阅者** | 不会触发 Redux 刷新 |

**结论**：从 sendMessage 到 finalize，Redux 的 assistant message 始终保持 `{ blocks: [], status: PENDING }` 不变。

## 4. 为什么用户能看到 assistant 文本

按上述分析，Redux 永远没有 assistant 文本，渲染层永远渲染不了 MAIN_TEXT block。但用户报告**看到了** "Hello! How can I help you today?"。

**解释**：[`loadTopicMessagesThunk`](../../../src/renderer/src/store/thunk/messageThunk.ts) 在切换 topic / fresh load 时会 `dbService.fetchMessages(topicId)` → `dispatch(upsertManyBlocks(blocks))` + `dispatch(messagesReceived(...))`。**用户必须切走 topic 再切回来**（或 reload 应用）才会从 SQLite 把 assistant message + blocks 拉到 Redux。**流中或刚结束未切 topic 时，UI 仍是空的**。

如果用户切回来后看到了文本，那么：
- MainTextBlock 应该已经被渲染了 → 应该有 `data-message-role="assistant"` wrapper
- `message.status` 应该是 SUCCESS → BeatLoader 应该消失

但用户报告**两者都未发生**。这指向两种可能：
1. 用户没有切换 topic，看到的是某种「正在写」状态的回显（比如某个我没找到的临时渲染路径，或浏览器内 React component 缓存）
2. DataApi PATCH 之后 SQLite 写入了，但 fetch 回来后 message.role / blocks 字段有问题

需要用户在 DevTools 跑下面命令确认 Redux 状态：

```js
// 1. 看 Redux 里 assistant message 的 blocks / status
const state = window.store?.getState?.()
const msgs = state?.messages?.entities ?? {}
Object.values(msgs)
  .filter(m => m?.role === 'assistant')
  .map(m => ({ id: m.id, status: m.status, blocks: m.blocks }))

// 2. 看 Redux 里 messageBlock 切片有哪些 block
const blocks = state?.messageBlocks?.entities ?? {}
Object.values(blocks).map(b => ({ id: b?.id, type: b?.type, status: b?.status, contentPreview: b?.content?.slice(0, 60) }))
```

如果 #1 显示 `blocks: []` / `status: 'pending'` → 证实诊断。
如果 #1 显示 `blocks: [<id>]` 而 #2 找不到该 id → block id 写到了 message 但 block 实体未上 Redux（也是诊断的子情况）。
如果两者都有 → 那本诊断错了，得另查渲染路径。

## 6. 修复路径

### 方案 A：cache → Redux 桥（最小手术）

在 `StreamingService` 的 4 个公共写入点之后，同步 dispatch 到 Redux。

```ts
// StreamingService.addBlock 末尾
store.dispatch(upsertOneBlock(block))
store.dispatch(newMessagesActions.updateMessage({
  topicId: task.topicId,
  messageId,
  updates: { blocks: newMessage.blocks }
}))

// StreamingService.updateBlock 末尾
store.dispatch(updateOneBlock({ id: blockId, changes }))

// StreamingService.updateMessage 末尾
store.dispatch(newMessagesActions.updateMessage({
  topicId: task.topicId,
  messageId,
  updates
}))

// StreamingService.finalize 之后（在 endTask 之前）
const task = getTask(messageId)
const allBlocks = Object.values(task.blocks)
store.dispatch(upsertManyBlocks(allBlocks))
store.dispatch(newMessagesActions.updateMessage({
  topicId: task.topicId,
  messageId,
  updates: { status, blocks: task.message.blocks }
}))
```

- **改动**：~30 行 in StreamingService（复用现有 `store` import + Redux actions）
- **优点**：实时反应；流中也能看到 assistant 内容；D-004 + D-005 一并修；无需碰渲染层
- **缺点**：双写两个 store（cache + Redux），与 v2「砍 Redux」大方向反向；属临时补丁，未来 v2 渲染层迁移完成后这桥要拆掉
- **风险**：throttle / RAF 时机若没处理好，可能引发 Redux 高频重渲染 → 性能问题（StreamingService 内部已 throttle，桥跟着走就行）

### 方案 B：渲染层切 cache（与 v2 方向对齐）

把 `Blocks/index.tsx` 和 `MessageBlockRenderer` 从 Redux `messageBlocksSelectors` 切到 `useSharedCache(getBlockKey(id))` + `useSharedCache(getMessageKey(messageId))`，Redux 当 fallback。

- **改动**：~50 行 in renderer，配合 useSharedCache 订阅
- **优点**：单一数据源；与 v2 cache 架构契合；不留双写
- **缺点**：影响 MessageBlockRenderer 主链路；现有 6+ Block component 都依赖 Redux block entities；测试覆盖面大；与 T-006 Text Anchor 渲染路径相邻，相互影响

### 方案 C：完成时单次 dispatch（最小最小，但只修 D-005 不修 D-004）

在 `onComplete` 调 finalize 之后，同步 dispatch 一次：

```ts
// baseCallbacks.ts onComplete 末尾，finalize 之后
const task = streamingService.getTask(assistantMsgId)
if (task) {
  const allBlocks = Object.values(task.blocks)
  store.dispatch(upsertManyBlocks(allBlocks))
  store.dispatch(newMessagesActions.updateMessage({
    topicId, messageId: assistantMsgId,
    updates: { status, blocks: task.message.blocks }
  }))
}
```

- **改动**：~10 行 in baseCallbacks
- **优点**：极小手术
- **缺点**：**流式中 D-004 / D-005 仍存在**；只有完成后 UI 才正确；用户体验在长回复中明显劣化

### 推荐

**走方案 A**：与 T-008C 的「短期走 v1↔v2 桥」策略一致；D-004 + D-005 一次修；改动定位在 StreamingService 一处，影响面可控；未来 v2 完整迁移时同步删 cache→Redux 桥 + 删 v1 Redux messageBlocks slice，干净。
