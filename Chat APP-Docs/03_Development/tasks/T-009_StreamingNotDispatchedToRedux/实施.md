## 1. 诊断更正

T-009 诊断初版判断：「streaming 管线完全不 dispatch Redux」—— **错误**。实读 StreamingService.ts 发现：

| 方法 | 状态 |
|---|---|
| `addBlock(messageId, block)` (line 279-319) | ✅ dispatch `upsertOneBlock(block)` + `newMessagesActions.upsertBlockReference({ messageId, blockId, status, blockType })` |
| `updateBlock(blockId, changes)` (line 331-381) | ✅ dispatch `updateOneBlock({ id, changes })` + 条件 `upsertBlockReference`（当 status/type 变化时）|
| `updateMessage(messageId, updates)` (line 402-430) | ✅ dispatch `newMessagesActions.updateMessage({ topicId, messageId, updates })` |
| `finalize(messageId, status)` (line 207-233) | ❌ **唯一不 dispatch 的** —— 仅 DataApi PATCH |

注释里都明确写了「TODO: temp fix, it will be removed after message refraction」—— 桥确实是临时贴布，但**已经在线**。

真正的 bug 在 reducer 里：

```ts
// src/renderer/src/store/newMessage.ts:270-276 (改前)
} else if (
  status === MessageBlockStatus.SUCCESS &&
  messageToUpdate.status === AssistantMessageStatus.PROCESSING
) {
  // Tentative success - may need refinement
  // changes.status = AssistantMessageStatus.SUCCESS    ← 注释掉了！
}
```

流式块完成时 `streamingService.updateBlock` 触发 `upsertBlockReference(status=SUCCESS)` → reducer 进 SUCCESS 分支 → **没有 changes** → message.status 永远停在 PROCESSING → `isMessageProcessing` 永远 true → BeatLoader (`Blocks/index.tsx:253-265` 的 PlaceholderBlock) 永远转。

## 2. 修复要点

### 2.1 取消注释 `newMessage.ts:275`

```diff
        } else if (
          status === MessageBlockStatus.SUCCESS &&
          messageToUpdate.status === AssistantMessageStatus.PROCESSING
        ) {
-         // Tentative success - may need refinement
-         // changes.status = AssistantMessageStatus.SUCCESS
+         // T-009 D-005: enabling this transition so PROCESSING → SUCCESS once a
+         // streaming block completes. Without it `isMessageProcessing(message)`
+         // stays true forever, the BeatLoader 3-dots placeholder never goes
+         // away, and the message action bar (copy / regenerate / quote / ...)
+         // never appears.
+         changes.status = AssistantMessageStatus.SUCCESS
        }
```

原始注释「may need refinement」的顾虑是合理的：单一 SUCCESS 块不一定代表整条消息完成（多块、多模型组等）。但 (a) 不转换比转错更糟（D-005 实测影响真实用户）；(b) `streamingService.finalize` 现在也加了防御性 dispatch，会用真实终态覆盖；(c) 多块场景下，最后一个块完成才是最相关的 SUCCESS 信号，正好对应转换发生的时刻。

### 2.2 `StreamingService.finalize` 末尾追加防御性 dispatch

```ts
// ...DataApi PATCH 成功之后
store.dispatch(
  newMessagesActions.updateMessage({
    topicId: task.topicId,
    messageId,
    updates: { status }
  })
)
this.endTask(messageId)
```

覆盖 `upsertBlockReference` 触发不到的边界：
- 整条消息没有任何 streaming block（空回复 / 极快 abort）
- abort / error 路径（baseCallbacks.onError 已经 `streamingService.updateMessage(...{ status: ERROR })`，但走 reducer 通用 update 分支 —— 现在多一道保险）
- 多模型组同 parentId 多消息（每条消息的 task 独立，finalize 各自 dispatch）

### 2.3 PopupContainer / chat-model-popup.tsx 等下游 0 改

整条 messageThunk / Blocks / MainTextBlock / SelectionContextMenu / MessageBlockRenderer 完全没碰；shim/render 链路与今天行为同等。

## 3. 测试覆盖

新文件 `src/renderer/src/store/__tests__/newMessage.upsertBlockReference.test.ts` 6 个用例：

| # | 场景 | 验证 |
|---|---|---|

| 6 | THINKING block 入列 | prepend 到 message.blocks 头部（与既有 reducer 行为一致） |

## 6. 推荐 commit

```
fix(messages): transition message status to SUCCESS when streaming block completes

The `upsertBlockReference` reducer had the PROCESSING → SUCCESS branch
commented out ("may need refinement"), which meant `message.status` stayed
at PROCESSING forever after the streaming block went SUCCESS. Result:
`isMessageProcessing` returned true, the BeatLoader 3-dots placeholder
kept rendering, and the message action bar (copy / regenerate / quote /
...) was never shown.

Also add a defensive dispatch in `StreamingService.finalize` so the
message-level status is mirrored to Redux even when no
`upsertBlockReference` fires (empty-block / abort / multi-model paths).

Closes D-005. Likely closes D-004 (assistant MainTextBlock wrapper
visible once the message exits PROCESSING). Refs: T-009.
```
