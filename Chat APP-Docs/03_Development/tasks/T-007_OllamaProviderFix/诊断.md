# T-007 诊断：D-003 Ollama 自动模型同步失败

## 1. 根因（一句话）

`packages/provider-registry/data/providers.json` 中 Ollama 条目**没有写 `defaultChatEndpoint` 字段**，同时 `endpointConfigs` 里同时存在 `ollama-chat` 和 `anthropic-messages`。主端点回退算法走到 Anthropic，整条链路在三处错位：form 渲染 / v1 shim / sync apiHost。

## 2. 现场对照

`packages/provider-registry/data/providers.json:421–440`：

```json
{
  "id": "ollama",
  "name": "Ollama",
  "description": "Ollama - AI model provider",
  "endpointConfigs": {
    "ollama-chat":         { "baseUrl": "http://localhost:11434" },
    "anthropic-messages":  { "baseUrl": "http://localhost:11434" }
  },
  "metadata": { ... }
}
```

注意：**没有 `defaultChatEndpoint`**。

横向对比（用 `python3 -c "..."` 扫了 63 条 provider）：8 条没写 `defaultChatEndpoint`，分别是 `ovms / ollama / new-api / lmstudio / azure-openai / vertexai / gpustack / aws-bedrock`。其中：

- `azure-openai` / `vertexai` / `aws-bedrock` 由 `presetProviderSeeder.getSeedDefaultChatEndpoint()` 单独打补丁。
- `ovms` / `new-api` / `lmstudio` 的 `endpointConfigs` 里有 `openai-chat-completions` —— **恰好命中主端点优先级表第 1 位**，所以"运气好"，没事。
- **只有 Ollama 的主端点 key 是 `ollama-chat`**，落到优先级第 5 位，前面有个 `anthropic-messages` 横插一脚。

## 3. 调用链

### 3.1 入口：auto sync

`src/renderer/src/pages/settings/ProviderSettings/hooks/providerSetting/useProviderAutoModelSync.ts`

```ts
const topology = getProviderHostTopology(provider)
...
if (!topology.primaryBaseUrl.trim().length && provider.id !== 'vertexai') return skip
...
void syncProviderModels(provider).catch((error) => {
  logger.error('Provider auto model sync failed', { providerId, error })
})
```

判定 `shouldSync = true`（Ollama 不要 apiKey；topology.primaryBaseUrl = 'http://localhost:11434' 非空），调入 `syncProviderModels`。

### 3.2 主端点解析（form + sync 都走这里）

`src/renderer/src/pages/settings/ProviderSettings/utils/providerTopology.ts:4–10,23–35`

```ts
const PRIMARY_CHAT_ENDPOINT_PRIORITY: EndpointType[] = [
  ENDPOINT_TYPE.OPENAI_CHAT_COMPLETIONS,
  ENDPOINT_TYPE.OPENAI_RESPONSES,
  ENDPOINT_TYPE.ANTHROPIC_MESSAGES,        // ← 第 3 位
  ENDPOINT_TYPE.GOOGLE_GENERATE_CONTENT,
  ENDPOINT_TYPE.OLLAMA_CHAT                 // ← 第 5 位，永远轮不到
]

function resolvePrimaryEndpoint(provider) {
  if (provider?.defaultChatEndpoint) return provider.defaultChatEndpoint
  for (const endpoint of PRIMARY_CHAT_ENDPOINT_PRIORITY) {
    if (hasEndpointConfig(provider, endpoint)) return endpoint
  }
  return ENDPOINT_TYPE.OPENAI_CHAT_COMPLETIONS
}
```

Ollama：`provider.defaultChatEndpoint` 是 `null`（seeder 在缺省时存 null，见 `presetProviderSeeder.ts:36` `return presetDefault ?? null`）→ 进 for 循环 → 第 1、2 项 Ollama 没有 → **第 3 项 `anthropic-messages` 命中** → `primaryEndpoint = 'anthropic-messages'`。

`topology.primaryBaseUrl = endpointConfigs['anthropic-messages'].baseUrl = 'http://localhost:11434'`。

### 3.3 form 错位 → 显示 Anthropic 字段

`src/renderer/src/pages/settings/ProviderSettings/ConnectionSettings/ApiHost.tsx:23–82`

```tsx
const { primaryEndpoint, ... } = useProviderEndpoints(provider)   // 内部也用 getProviderHostTopology
const isAnthropicPrimaryEndpoint = primaryEndpoint === ENDPOINT_TYPE.ANTHROPIC_MESSAGES
...
{!isAnthropicPrimaryEndpoint ? <ApiHostField ... /> : <AnthropicApiHostField ... />}
```

`primaryEndpoint === 'anthropic-messages'` → 渲染 `AnthropicApiHostField`（`ApiHostFields.tsx:158–215`）→ 标题用 i18n key `settings.provider.anthropic_api_host`，副本预览 i18n key `settings.provider.anthropic_api_host_preview`。这就是用户看到的 **"Anthropic API Host" + "/v1/messages"**。

### 3.4 sync 错位 → 空 host + 错 type

`src/renderer/src/pages/settings/ProviderSettings/ModelList/modelSync.ts:214`

```ts
const fetched = await fetchModelsStrict(toV1ProviderShim(provider, { apiKey }))
```

`toV1ProviderShim`（`v1ProviderShim.ts:119–149`）：

```ts
function v1ProviderTypeFromV2(v2): ProviderType {
  ...
  const ep = v2.defaultChatEndpoint ?? ENDPOINT_TYPE.OPENAI_CHAT_COMPLETIONS  // ← Ollama 这里 fallback 到 openai
  switch (ep) {
    case OLLAMA_CHAT:
    case OLLAMA_GENERATE: return 'ollama'
    ...
    default: return 'openai'   // ← Ollama 落到这里
  }
}

function defaultChatBaseUrl(v2): string {
  const ep = v2.defaultChatEndpoint ?? ENDPOINT_TYPE.OPENAI_CHAT_COMPLETIONS
  return v2.endpointConfigs?.[ep]?.baseUrl ?? ''   // ← Ollama 没有 openai-chat-completions key → ''
}

return {
  ...
  type: 'openai',          // ← 错：应该是 'ollama'
  apiHost: '',             // ← 错：应该是 'http://localhost:11434'
  anthropicApiHost: 'http://localhost:11434',
  ...
}
```

注意：**注意 sync 走的是 v1 shim 的 fallback，不是 topology**。即便 topology 说主端点是 `anthropic-messages`，v1 shim 这里默认 fallback 到 `openai-chat-completions`，根本没有翻看 topology。两个分支独立错。

### 3.5 落到 fetch

`fetchModelsStrict` → `new AiProvider({ type: 'openai', apiHost: '', ... }).models({ throwOnError: true })`。

`apiHost = ''` 触发 legacy AiProvider 的 OpenAI 客户端用 SDK 内置 default（`https://api.openai.com/v1`）或直接发到空 URL —— 任意一种结果：
- 命中 OpenAI 401（HTML 错误页）→ `response.json()` 失败 → `Invalid JSON response`
- 或 fetch 失败 → 同名错误冒泡

我没继续往下读 AiProvider 的 SDK 适配细节（用户要求"先诊断不大改"），但**这一层是症状不是病因**，根因在 shim 之前 —— 修了数据，shim 就不会算错。

## 4. 最小修复方案

### 方案 A（推荐）：补一行 `defaultChatEndpoint`

`packages/provider-registry/data/providers.json:421`：

```diff
   {
     "id": "ollama",
     "name": "Ollama",
     "description": "Ollama - AI model provider",
+    "defaultChatEndpoint": "ollama-chat",
     "endpointConfigs": {
       "ollama-chat": {
         "baseUrl": "http://localhost:11434"
       },
       "anthropic-messages": {
         "baseUrl": "http://localhost:11434"
       }
     },
     ...
   }
```

修这一行能同时治：

| 现象 | 修复路径 |
|---|---|
| form 显示 Anthropic 字段 | `resolvePrimaryEndpoint` 在 `provider.defaultChatEndpoint` 上短路 → 直接返回 `'ollama-chat'` → `isAnthropicPrimaryEndpoint = false` → 渲染普通 `ApiHostField` |
| sync 用错 type/apiHost | `v1ProviderTypeFromV2` 第 50 行 `ep = v2.defaultChatEndpoint` = `'ollama-chat'` → switch 命中 `OLLAMA_CHAT` → `type: 'ollama'`；`defaultChatBaseUrl` 也走同一 ep → `endpointConfigs['ollama-chat'].baseUrl` = `http://localhost:11434` |
| auto sync 报错 | type 对了 + host 对了 → AiProvider 走 Ollama 适配 → `/v1/models` 拉对 → JSON 解析成功 |

**Schema 安全**：`ProviderConfigSchema` 的 `.refine` 只校验「若 defaultChatEndpoint 非空，则它必须是 endpointConfigs 的一个 key」。`ollama-chat` 在 endpointConfigs 里有，合法。

**数据迁移**：`PresetProviderSeeder.run` 在 fresh install 时 batchUpsert 全部 provider；老用户的 DB 行如果已有 `defaultChatEndpoint = null`，需要看 `providerService.batchUpsertTx` 是否覆盖该列。**未验证**——建议测试时 `rm -rf ~/Library/Application\ Support/CherryStudioDev` 走 fresh install。

### 方案 B（不推荐 / 防御层）：调 PRIMARY_CHAT_ENDPOINT_PRIORITY

把 `OLLAMA_CHAT` 提到 `ANTHROPIC_MESSAGES` 之前。**当前 providers.json 只有 Ollama 自己用 `ollama-chat` 键**，所以提前是安全的；但 PRIMARY_CHAT_ENDPOINT_PRIORITY 是给所有 provider 用的通用回退，影响面大，没必要为单一 provider 改通用规则。**否决，留作 backup**。

### 方案 C（远期）：providers.json schema 要求 `defaultChatEndpoint` 必填

把 `defaultChatEndpoint` 从 nullable 改为 required（azure-openai / vertexai / aws-bedrock / gpustack 也需要补值或改 schema 处理 IAM 路径）。**超出本 issue 范围，留作 cleanup task**。
