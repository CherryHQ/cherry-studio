# T-008 诊断：D-003C Chat 弹窗 v1/v2 数据断层

## 1. 根因（一句话）

`SelectChatModelPopup` 读的是 **v1 Redux `state.llm.providers`**；`useProviderModelSync.syncProviderModels` 写的是 **v2 DataApi `user_model` 表**（通过 `createModels` → `POST /models`）。两条链路在仓库里是两份独立的"真相"，**没有任何桥接代码**把 v2 sync 的结果回写到 v1 Redux —— 所以 Ollama 模型同步成功后，v1 picker 完全无感。

CherryAI Qwen 之所以能在 picker 里看到，是因为 `useProvider.ts:36–41` 的 `selectEnabledProviders` selector 显式做了 `.concat(CHERRYAI_PROVIDER)`，而 `CHERRYAI_PROVIDER` 是 `src/renderer/src/config/providers.ts:7–16` **硬编码**的一行配置（`models: [qwenModel]`，`enabled: true`）—— 与任何 sync 流程无关。**它是一个 selector 级别的特例，不是数据通道的产物。**

## 4. 为什么 CherryAI 显示 / Ollama 不显示

对称破坏只在一个地方：**v1 selector 给 CherryAI 开了硬编码后门**，给 Ollama 没开。

```ts
// hooks/useProvider.ts:36–41
const selectEnabledProviders = createSelector(selectProviders, (providers) =>
  providers
    .map(normalizeProvider)
    .filter((p) => p.enabled)
    .concat(CHERRYAI_PROVIDER)   // ← Ollama 没有这个待遇
)
```

CherryAI 还自带 `models: [qwenModel]`（硬编码 1 个 qwen），所以 picker 渲染时直接看到。Ollama 即使从 v1 角度被 selector 留下（假设你手动设 `enabled: true`），它的 `models: []` 仍然是空，picker 还是看不到模型。换句话说：**Ollama 是双重隐藏 —— provider enabled=false + models=[]**。

CherryAI 是**双重显示 —— 硬编码 enabled=true + 硬编码 models**。

## 5. v2 ModelSelector 已存在（旁证）

`src/renderer/src/components/ModelSelector/` 是一份**已经迁好 v2** 的同款组件：

- `useModelSelectorData.ts:1–7` 用 `useModels` + `useProviders`（v2）
- 已被 `ModelSettings.tsx` / `TranslatePage.tsx` / `OpenClawPage.tsx` / `LibraryEditor/ModelSelectorField.tsx` 等使用
- **唯独 `SelectChatModelPopup`（主聊天弹窗）+ `MessageMenubar` 的模型切换 + `Chat.tsx` 的模型切换还在用 v1 `SelectChatModelPopup`**

这说明 v2 化的工作已经做了一半，剩 Chat 主链路这一段是已知的迁移空洞。Chat 弹窗迁 v2 = "**把 ModelSelector 思路套到 SelectChatModelPopup**"，不是新发明。

## 6. 两条修复路径

### 方案 A：v2 → v1 桥（小手术，保命）

**改什么**：在 `syncProviderModels`（或 `useProviderAutoModelSync` 的成功回调）里，把 created models 反向 dispatch 到 v1 Redux。

伪代码：
```ts
// useProviderModelSync.ts，createdModels 计算完成之后：
import { addProvider, addModel, updateProvider } from '@renderer/store/llm'

// 确保 v1 Redux 有这条 provider 且 enabled
const v1Existing = store.getState().llm.providers.find(p => p.id === providerId)
if (!v1Existing) {
  dispatch(addProvider(v1ShapeFor(provider)))   // 用 toV1ProviderShim 的思路
} else if (!v1Existing.enabled) {
  dispatch(updateProvider({ id: providerId, enabled: true }))
}

// 把 v2 created models 转 v1 Model 形状逐个 add
for (const m of createdModels) {
  dispatch(addModel({ providerId, model: toV1ModelShim(m) }))
}
```

- **改动面**：~30 行 hook + 复用现成的 `toV1ProviderShim` / `toV1ModelShim`（`utils/v1ProviderShim.ts`）
- **优点**：picker 立刻能用；不动 Chat 主链路；不动 SelectChatModelPopup
- **风险**：
  - 双写脏数据 —— 用户在 v2 ProviderSettings 改 model 名 / enabled，v1 不会同步（picker 显示陈旧值）；但是仅当用户跑两条 UI 时才暴露，临时方案可接受
  - 仍然不解决根本：v1 selector 还在跑，CHERRYAI_PROVIDER 硬编码还在
- **可选额外**：在 `useProvider.ts:36–41` selector 临时去掉 `.filter((p) => p.enabled)` 的 Ollama 黑名单语义（不推荐 —— 会让所有 v1 disabled provider 一起冒出来）

### 方案 B：Chat 弹窗迁 v2（大手术，根治）

**改什么**：`chat-model-popup.tsx` 从用 v1 `useProviders` + `provider.models` 改为用 v2 `useProviders` + `useModels()`，**和 `components/ModelSelector/useModelSelectorData.ts` 完全一致的模式**。

- **改动面**：
  - `chat-model-popup.tsx` 重写约 30 行
  - `base-popup.tsx` 把 `Provider | Model` 类型从 `@renderer/types`（v1）改为 `@shared/data/types/*`（v2）—— 此处 ~100 行 type bridging
  - `useDefaultModel` / `setDefaultModel` 仍是 v1 Redux —— 要么让 picker 输出 v2 Model 再适配回 v1，要么同步迁 defaultModel
  - 下游 `Chat.tsx:92` / `MessageMenubar.tsx:553` / `SelectModelButton.tsx:29` 类型也要跟进
- **优点**：彻底消解断层；和 v2 迁移大方向对齐；可以同时砍掉 CHERRYAI_PROVIDER 硬编码（如果 v2 DataApi 已经 seed 了 cherryai provider，那就更自然）
- **风险**：
  - 影响 Chat 主链路 —— 当前在做 T-006 Text Anchor Branch，可能撞 scope
  - `useDefaultModel`/`setDefaultModel`/`assistant.model`/`assistant.defaultModel` 一连串 v1 Model 类型的下游
  - 测试覆盖面：`__tests__/chat-model-popup.test.tsx` 等都要重写

## 7. 推荐

**短期推荐 A**：

- 用户当下需要的是「能在 Chat 里选到 Ollama 模型测 Text Anchor Branch」。这个目标方案 A 一晚上就能做完，让 T-006 不再卡在数据问题上
- v2 迁移的"彻底解决"留作独立 task（T-009? T-010?），不与 T-006 时间窗争抢

**警示**：方案 A 是 v2 迁移**的反方向投资**（增加 v1↔v2 双写），仅当能在短期内（≤ 1 个迭代）跟上方案 B 才合理。如果迁移要拖几个月，方案 A 容易成为长期负担。
