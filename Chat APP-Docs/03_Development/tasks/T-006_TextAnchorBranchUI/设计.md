## 1. Design Question 1 — 选区捕获在哪里做？

### 决策
**在 MainTextBlock 渲染时给 `<div className="markdown">` 加 `data-message-id` 和 `data-block-id` 属性 → 全局菜单触发时从 `selection.anchorNode` 往上 climb DOM 找最近的 data 属性 → 拿到 ids**。

### 候选方案对比

| 方案 | 优 | 劣 |
|---|---|---|
| **A. 在 MainTextBlock 包独立 `<SelectionContextMenu>`** | block 本地知道自己 id | 每个块都是独立 ContextMenu 实例，浪费；嵌套 trigger 与 Messages 全局的 menu 会打架 |
| **B（选）. 加 data 属性 + 全局菜单 climb 找** | 一处菜单管全 vault；MainTextBlock 只多 2 个 props；selection 自然跨 block 工作 | 需要 climb 逻辑（5 行 helper） |
| C. React Context 把 messageId / blockId 注入 MainTextBlock | 不污染 DOM | 选区不一定在 React tree 当前激活分支；用 selection.anchorNode 找上下文更可靠 |

### 实现要点（写到 T-006B）

- `MainTextBlock.tsx` 接收 `messageId` props（从 Blocks/index.tsx 传入）
- 把 `<Markdown />` 包在一层 `<div data-message-id={messageId} data-block-id={block.id}>` 内
- `SelectionContextMenu` 的 `handleOpenChange` 里：
  ```ts
  const range = selection.getRangeAt(0)
  let node = range.commonAncestorContainer
  while (node && !(node as Element).dataset?.blockId) {
    node = node.parentNode!
  }
  const blockEl = node as HTMLElement
  const messageId = blockEl.closest('[data-message-id]')?.getAttribute('data-message-id')
  const blockId = blockEl.dataset.blockId
  ```
- 把 `{ messageId, blockId, selectedText }` 放进 state → 菜单项点击时透传

### 兜底

若 climb 没找到（用户在 message 之外右键、或选区跨 message），分支菜单项 disabled；已有的 Copy / Quote 仍可用。

## 2. Design Question 2 — 右键 vs 浮动菜单？

### 决策
**第一版只做右键（扩展现有 `SelectionContextMenu`）**。浮动气泡推到 v1.1。

### 理由
1. **基础设施在了** —— `SelectionContextMenu` 已实现并已经包裹 Messages 滚动区；加菜单项零成本
2. **浮动菜单实现复杂** —— 需要管理浮层位置、`selectionchange` 事件防抖、桌面/触屏差异、与 popover/dropdown 的 z-index 冲突
3. **CodeViewer 踩过坑** —— `src/renderer/src/components/CodeViewer.tsx:132` FIXME 提到 SelectionToolbar 弹不出走剪贴板回退，说明已有的浮动方案脆弱
4. **桌面端用户熟悉右键** —— Cherry Studio 是桌面客户端，右键是直觉操作
5. **不互斥** —— v1.1 可以在右键之上叠加浮动小气泡（"💬 提问" 按钮），不冲突

### 例外
若用户测试反馈"右键太麻烦"，v1.1 实现一个**hover-after-selection 小气泡**（300ms 防抖后在选区右上角弹一个 ⌘+E 类型快捷按钮）。但 v1.0 不做。

## 3. Design Question 3 — Branch Panel 字段（第一版）

### 决策

```ts
// 状态形态，不一定是 schema（v1 不持久化）
interface BranchAnchor {
  // 上下文
  sourceTopicId: string         // 当前 topic（v2 UUID）
  sourceMessageId: string       // 选区所在 message
  sourceBlockId: string         // 选区所在 main_text block
  // 选区内容
  selectedText: string          // extractSelectedText 提取的文本
  // 时间
  createdAt: string             // ISO timestamp
  // 可选 / 留扩展
  selectionStart?: number       // 在 block.content 字符偏移（v1.0 不存）
  selectionEnd?: number         // 同上
  anchorId?: string             // UUID v7，v1 用 createdAt + sourceBlockId 拼或客户端生成；用来关联高亮 DOM
}
```

### 字段 rationale

| 字段 | 为什么 |
|---|---|
| `sourceTopicId` | Panel 可能跨 topic 显示历史 anchor；fork 时也需要知道源 topic |
| `sourceMessageId` | v2 `Topic.activeNodeId` 的候选；fork 操作 = `POST /topics { sourceNodeId: sourceMessageId }` |
| `sourceBlockId` | 精确定位 + 未来持久化高亮的最小颗粒 |
| `selectedText` | prompt 的种子；fork 后注入新 topic 的 user message |
| `createdAt` | 排序、调试、未来持久化时填 SQLite 时间戳 |
| `selectionStart/End` (opt) | 字符偏移，未来恢复高亮位置；v1 用 selectedText 字符串匹配也能找到（不精确但 80% 够用） |
| `anchorId` (opt) | DOM 上挂 `data-anchor-id` 用于点击重开 Panel |

### 持久化策略（v1.0）

- 不存 SQLite
- 不进 Redux 全局 store（避免 Redux 渐进淘汰路径上的反向投资）
- **用 `useCache` 临时存**：`cacheService.set('branch-anchor.${topicId}.${anchorId}', anchor)` —— 切 topic 后自然过期 / 清空

### 长远（v1.x+）

若用户反馈 anchor 应该跨会话保留，再设计：
- 新表 `message_anchor`（独立表）or
- `message.data.annotations: AnchorRef[]` 字段（嵌入式）

哪条路看高亮量级；annotations 是低频写场景适合嵌入式，跨表查询差但写性能好。

## 4. Design Question 4 — 高亮标注（第一版）

### 决策（综合）

| 维度 | v1.0 决策 |
|---|---|
| 是否临时 | ✅ 是。仅当前 session 有效，切 topic 清空，刷新页面消失 |
| 是否持久化 | ❌ 否（见 §3） |
| 实现路径 | **路径 A：`range.surroundContents(<mark>)` 直接 DOM 注入** —— 简单覆盖 80% 同段落选区 |
| 跨段落选区 | v1.0 不支持，菜单项 disabled 并提示「选区限制在一句话内」 |
| 流式块上禁用 | ✅ 仅 `block.status === 'success'` 时允许高亮，避免 `useSmoothStream` 重置 |

### 候选路径对比

| 路径 | 描述 | v1.0 评分 |
|---|---|---|
| **A（选）** | `range.surroundContents(<mark>)` 直接包 | ⭐⭐⭐ 简单；同段落 OK；跨段落抛错 |
| B | React state + Markdown components hook 在 render 时 wrap | ⭐⭐ 与 useSmoothStream 冲突；改 Markdown 渲染管道有副作用 |
| C | CSS Custom Highlight API (`CSS.highlights`) | ⭐⭐⭐⭐ 不污染 DOM，最干净 —— v1.1 升级目标 |
| D | jQuery 风格 `<span>` 包 text nodes | ⭐ 破坏 React diff |

### 为什么不一上来用路径 C

- 需要 React 重渲染时手动重建 `Highlight` 对象（用 `useEffect`）
- Range 对象在 React 重渲染时可能失效
- 第一版 KISS：用最简单的 surroundContents 验证交互；v1.1 升级到 Custom Highlight API 拿到「DOM 干净 + 重渲染稳定」

### 避免破坏 Markdown 渲染的关键约束

1. **`<mark>` 注入发生在 ReactMarkdown 渲染完成之后**（DOM 操作，不进 `messageContent` 字符串）
2. **不写回 `block.content`** —— 否则会污染 streaming delta 计算
3. **流式中的块 disabled 高亮**（`block.status !== 'success'` 时菜单中"展开分支"项 disabled）
4. **block.id 变化（重新生成）时清掉对应 anchor cache**（在 useEffect 监听 block.id 变更）

### CSS（建议）

```css
mark.branch-anchor-hl {
  background-color: hsl(48 100% 88%);   /* light yellow */
  border-bottom: 1.5px dashed hsl(43 80% 50%);
  padding: 0 1px;
  cursor: pointer;
  transition: background-color 150ms;
}
mark.branch-anchor-hl:hover {
  background-color: hsl(48 100% 78%);
}
@media (prefers-color-scheme: dark) {
  mark.branch-anchor-hl {
    background-color: hsl(48 70% 25%);
    color: inherit;
  }
}
```

放在 `src/renderer/src/pages/home/Messages/Blocks/branch-anchor.css` 或类似（CLAUDE.md：UI 用 `@cherrystudio/ui` + Tailwind；这里若用 Tailwind class 名更好 — 实施时由 T-006E 定）。

## 5. Design Question 5 — fork 行为

### 决策（v1.0）

「展开为分支」=

```ts
// 1. 新建 topic（v2 fork）
const newTopic = await dataApiService.post('/topics', {
  body: {
    name: anchor.selectedText.slice(0, 30),       // 用选区前 30 字做名字
    assistantId: currentAssistant.id ?? null,     // T-004 fix 保证 null 也合法
    sourceNodeId: anchor.sourceMessageId          // v2 fork 关键
  }
})

// 2. 切到新 topic
setActiveTopic(newTopic)

// 3. 把 selectedText 作为 prompt 种子注入输入框（不直接发送）
inputbarController.setText(
  `在这段文字 ："${anchor.selectedText}"\n\n` +
  `针对上面这段，` // 留 caret 在这里
)
```

「针对此处提问」 = 同上，但**不切 topic**，留在当前 topic，把选区注入输入框做引用。

### 为什么 fork 用 v2 `POST /topics { sourceNodeId }` 而不是另起空 topic

- v2 已实现 fork 语义：`Topic.activeNodeId = sourceNodeId`，新 topic 直接接在指定节点之后
- 这是 v2 设计的标准用法，不引入新概念
- `MessageService.getBranchMessages` 会自动把 root → sourceNodeId 的路径作为新 topic 的对话历史

⚠️ **注意**：fork **不复制消息树**（见 [服务算法](../../../02_Architecture/服务算法.md) §2.1）；新 topic 的消息历史是通过 `getBranchMessages(newTopic, { nodeId: sourceNodeId })` 间接得到。这是 v2 设计取舍。

## 6. 整体数据流（v1.0）

```
用户在 markdown 文本上选中文字
   ↓
用户右键
   ↓
SelectionContextMenu.handleOpenChange
   ├ window.getSelection() → range
   ├ extractSelectedText(selection) → string
   └ climbDOM(range) → { messageId, blockId } (T-006B)
   ↓
SelectionContextMenu 渲染菜单 (T-006C 扩展)
   ├ Copy (已有)
   ├ Quote (已有)
   ├ 📌 展开为分支  ← 新增
   └ 💬 针对此处提问 ← 新增
   ↓
用户点 "展开为分支"
   ↓
1. range.surroundContents(<mark class="branch-anchor-hl" data-anchor-id="...">) (T-006E)
2. cacheService.set('branch-anchor.${topicId}.${anchorId}', { sourceTopicId, sourceMessageId, sourceBlockId, selectedText, createdAt, anchorId }) (T-006D)
3. dispatch openBranchPanel(anchorId)  (T-006D)
   ↓
Branch Panel 打开（侧边或右抽屉）
   ├ 显示 selectedText 引用块
   ├ 用户输入提问
   └ 点"发送"
       ↓
4. POST /topics { sourceNodeId: sourceMessageId, name: ..., assistantId } → new topic
5. setActiveTopic(newTopic)
6. 用户消息自动发到新 topic（含 selectedText 引用）
```

## 8. 风险与已知限制

| 风险 | 缓解 |
|---|---|
| 流式块上高亮被冲掉 | 流式中 disable 菜单项；`useEffect` 监听 block.status 变化恢复 anchor |
| 跨段落选区 `surroundContents` 抛错 | v1.0 不支持跨段落；UI 提示 |
| Markdown 重渲染清掉 `<mark>` | 用 `useEffect` + anchor cache 重新注入；v1.1 升级到 CSS Highlight API 彻底干净 |
| `extractSelectedText` 把跨块内容也"扁平"了 | 仅在 climb 成功且 blockId 唯一时启用新菜单项；否则 disabled |
| Branch Panel 与现有 Tabs (Topics 抽屉) 抢右侧空间 | 用 Sheet / overlay 弹层；不动 Chat.tsx 三栏布局；T-006D 决策 |
| **D-001/D-002 baseline 修复未端到端验证** | T-006A 用 mock 数据先跑通组件层；端到端跑通靠 baseline 验证后 |
| **D-003 Ollama 自动同步失败** | 用其他 provider 测试；不阻塞 T-006 |
