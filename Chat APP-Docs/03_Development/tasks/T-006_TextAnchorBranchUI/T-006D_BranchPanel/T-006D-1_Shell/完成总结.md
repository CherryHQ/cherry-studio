## 设计决策（实施时）

| 选择 | 理由 |
|---|---|
| Dialog 而非 Drawer | 全局测试 mock 已覆盖 Dialog 全套，Drawer 未 mock；改全局影响面大；壳子原型用居中 Dialog 已满足验收 |
| `Messages.tsx` 本地 useState 而非 useCache | 「最小」；D-2 切到 useCache 时只换 setBranchAnchor 实现 |
| `onOpenBranchPanel?` 可选 callback prop | SelectionContextMenu 在 `CitationsList` 等地方也用，不能强加 Context 依赖；可选 prop 兼容既有调用 |
| `BranchAnchor` 在 `BranchPanel/types.ts` | 与 BlockContext（DOM 解析中间态）解耦；Panel 领域类型独立 |
| follow-up state 在 BranchPanel 内 + open 变化时清空 | 自治组件；不需要外部协调 |
| 创建按钮 log + 关闭 | 用户原话「disabled 或只打日志」选 log；同时关闭避免 zombie 状态 |

## 经验

1. **「callback 比 Context 通用」** —— SelectionContextMenu 是被多处使用的通用组件，可选 callback 让 host 选择性接入，比强 Context 优雅，且向后兼容既有调用
2. **测试 mock 是先行约束** —— 选 Dialog 不选 Drawer 主要是因为全局 mock 已覆盖；改全局 mock 影响面大于一个壳子原型需要
3. **i18n 命名空间预留扩展** —— `chat.message.anchor.panel.*` 子命名空间为后续 D-2/D-3 留位（如 `panel.fork_success` 等可平铺追加）
4. **每个子任务 strict scope = 干净 diff = 简单 review** —— 净 +116 业务行，5 文件改动，没有 dangling 状态；D-2 接进来时不会被 D-1 的过度设计阻碍

const handleCreateBranch = async () => {
  if (!anchor) return
  try {
    const newTopic = await dataApiService.post('/topics', {
      body: {
        name: anchor.selectedText.slice(0, 30),
        assistantId: currentAssistant.id === 'default' ? null : currentAssistant.id,  // T-004 兼容

    // toast.error or inline error UI
  }
}
