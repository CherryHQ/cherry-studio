## 0. 源码确认结论（先验证再设计）

**`TopicService.create` 的实际行为**（`src/main/data/services/TopicService.ts:121–160`）：

- `sourceNodeId` 只做存在性校验，然后存为新 topic 的 `activeNodeId`。**没有"路径拷贝"，没有"继承 assistant"**。
- `assistantId` 完全照搬 DTO；omit/`undefined` ⇒ 新 topic 不绑定 assistant，**不是**继承源 topic。

**`StreamingService.createUserMessage` 的实际行为**（`src/renderer/src/services/messageStreaming/StreamingService.ts:538, 549`）：

- 明确**不传 `parentId`**，让 DataApi 用 `topic.activeNodeId` 当 parent（line 538 注释原文）。
- 所以：fork 把新 topic.activeNodeId 设成 sourceMessageId → 调既有 `sendMessage` → 新 user message 自动以 sourceMessage 为 parent → lineage 走通。
- **不需要新 thunk，不需要注入 parentId，不需要克隆任何东西。**

这一条决定了 MVP 范围。

## 1. 最小目标

让以下闭环跑通，且**只跑通这个闭环**：

> 选中 assistant 文本 → Panel → 输入 follow-up → 点 Create → `POST /topics { sourceNodeId }` → setActiveTopic → 既有 `sendMessage` 发首条 user message → 模型开始回复 → Panel 关闭

成功判据：fresh install 用户点完整流，自动跳到新 topic 并看到模型流式回复，侧边栏出现新 topic。

## 2. Create branch 的真实状态流转

```text
[idle]                       Panel 开，followUp 已输入
   │
   ▼ 点击 Create
[validating]                 followUp.trim() 长度 ≥ 1？
   │ no  ──▶ inline 红字 "请输入问题"，回 idle
   ▼ yes
[creating]                   Create 按钮 spinner + disabled；Cancel disabled
   │
   ▼ POST /topics { name, assistantId?, sourceNodeId }
   │
   ├── reject ──▶ [error]    红字 + Create 恢复 + followUp 保留
   │
   └── resolve(newTopic)
        │
        ▼ setActiveTopic(newTopic)        ← prop 透传到 Messages 的 host
        │
        ▼ dispatch(sendMessage(userMsg, blocks, assistant, newTopic.id))
        │   ↑ StreamingService 自动以 activeNodeId（=sourceMessageId）为 parent
        │
        ├── reject ──▶ [error]   红字"发送失败"+ followUp 保留
        │              （新 topic 已存在；不回滚）
        │
        └── resolve
             │
             ▼ [done] onOpenChange(false)
                       BranchPanel 自动清 followUp（已有 useEffect）
```

**不变量**：

- `[creating]` 期间 Dialog 不能关
- 失败时 followUp 不清空
- 成功后才关 Panel（关 = anchor 置 null = followUp 清）

## 4. 职责边界

### BranchPanel.tsx —— 纯 UI

输入 props：

```ts
interface Props {
  anchor: BranchAnchor | null
  open: boolean
  onOpenChange: (open: boolean) => void
  // T-006D-2A 新增：
  status: 'idle' | 'creating' | 'error'
  errorMessage?: string
  onCreate: (followUp: string) => void   // 同步签名，hook 负责 async
}
```

职责：

- 显示 `anchor.selectedText` preview
- 管 followUp 本地 useState
- 显示 spinner / 错误红字（根据 status / errorMessage）
- 校验 `followUp.trim().length > 0`
- 点 Create → 调 `onCreate(followUp.trim())`

**不做**：DataApi 调用、Redux dispatch、setActiveTopic、读 useMutation。

### useBranchFork.ts —— 业务编排

```ts
function useBranchFork(args: {
  topic: Topic
  assistant: Assistant
  anchor: BranchAnchor | null
  setActiveTopic: (t: Topic) => void
  onSuccess?: () => void
}) {
  const [status, setStatus] = useState<'idle' | 'creating' | 'error'>('idle')
  const [errorMessage, setErrorMessage] = useState<string>()
  const { trigger: createTopic } = useMutation('POST', '/topics', { refresh: ['/topics'] })

  const fork = async (followUp: string) => {
    if (!anchor) return
    setStatus('creating'); setErrorMessage(undefined)
    try {
      const newTopic = await createTopic({ body: buildCreateBody(anchor, assistant, topic) })
      setActiveTopic(newTopic)
      const { message, blocks } = getUserMessage({ assistant, topic: newTopic, content: followUp })
      await dispatch(sendMessage(message, blocks, assistant, newTopic.id))
      setStatus('idle')
      onSuccess?.()
    } catch (e) {
      setStatus('error')
      setErrorMessage(t('chat.message.anchor.panel.error.create_failed'))
      logger.error('branch-fork failed', e)
    }
  }

  return { fork, status, errorMessage, reset: () => setStatus('idle') }
}
```

职责：

- POST /topics（含 body 构造）
- setActiveTopic
- 构造 user message + dispatch sendMessage（既有路径）
- 状态机三态：idle / creating / error
- 失败时**不回滚新 topic**（接受 known-good-enough，见 §7）

### Messages.tsx —— 装配层

```tsx
const branchFork = useBranchFork({
  topic, assistant, anchor: branchAnchor, setActiveTopic,
  onSuccess: () => setBranchAnchor(null)

<BranchPanel

  errorMessage={branchFork.errorMessage}

```

## 5. POST /topics 的 body 设计（基于 §0 实测）

```ts
function buildCreateBody(anchor: BranchAnchor, assistant: Assistant, currentTopic: Topic): CreateTopicDto {
  return {
    name: anchor.selectedText.trim().slice(0, 30) || 'Branch',
    // 哨兵 'default' → 不传字段（DTO 是 .partial()）
    ...(assistant.id !== 'default' ? { assistantId: assistant.id } : {}),
    // groupId 跟随当前 topic，保持新 topic 在同一组
    ...(currentTopic.groupId ? { groupId: currentTopic.groupId } : {}),
    sourceNodeId: anchor.messageId
  }
}
```

**关键说明（消除上版误解）**：

- 传 `assistantId` 时，传**当前 assistant.id**。这不是"继承源 topic"，而是"用户当前正在用的 assistant"。`TopicService.create` 不会从源 topic 推导 assistantId。
- omit `assistantId`（哨兵情况）= 新 topic 不绑定 assistant。这是**已知后果**，不是"让 server 兜底"。后续 sendMessage 仍用同一个 assistant 实例参数，发送行为不受影响（assistant 由参数传入，不查 topic.assistantId）。
- `groupId` 透传：用户在 "工作"组里 fork 出来还在 "工作"组，否则跑去 root 就丢失分组上下文。

## 6. follow-up message 如何走现有发送路径

**这是 MVP 简化的核心**：什么新代码都不写。

```ts
const { message, blocks } = getUserMessage({ assistant, topic: newTopic, content: followUp })
await dispatch(sendMessage(message, blocks, assistant, newTopic.id))
```

为什么 work（基于 §0 验证）：

1. `sendMessage` 内部调 `streamingService.createUserMessage(topicId, msg, blocks)`（`messageThunk.ts:1020`）。
2. `createUserMessage` 不传 `parentId`，让 DataApi 用 `topic.activeNodeId` 当 parent（`StreamingService.ts:538` 注释 + `:553`）。
3. 新 topic 的 `activeNodeId` 在 §3 fork 那一步就设成了 `sourceMessageId`。
4. 所以新 user message 的 parent 自动是源 assistant message。lineage 走通。

**不嵌引用 selectedText**：MVP 不在 followUp 文本里插 `> <selectedText>` 模板。原因：

- 源 assistant message 已经在 lineage 里，模型读得到完整上下文。
- 引用文本是 UX 增强（让模型聚焦），不是功能必需。
- 等到 D-3 设计"针对此处提问"时再讨论引用拼接（那里两个动作都要拼）。

如果用户使用后觉得模型回复不聚焦，再回头加一行模板，是 XS 改动。先不做。

## 7. 错误处理

| 失败点 | UI | 副作用 | 回滚 |
|---|---|---|---|
| followUp 空 | inline 红字"请输入问题"，按钮不动 | 无 | — |
| POST /topics 失败 | 红字"创建分支失败"，Create 恢复，followUp 保留 | 无 | 无需回滚 |
| sendMessage 失败 | 红字"发送失败"，Create 恢复，followUp 保留 | 新 topic 已创建 + activeTopic 已切走 | **不回滚**：DELETE /topics 增加复杂度，用户也能手动重试输入；新 topic 是合法的空分支 |

**为什么不回滚**：

- 回滚 = `DELETE /topics`，要再处理"删除失败"，结构性递归
- 即使保留空新 topic，用户体感是"分支已经创建好，但首条消息没发出去"——可以在 inputbar 重发，不需要从头再来
- 在文档里写明这是 known-good-enough

`logger.error` 把原始 error 落主进程，UI 只显示 generic 文案。

## 白话总结

**MVP 到哪儿为止**

走完：选 assistant 文本 → 右键 → Panel → 输入问题 → Create → 自动跳到新 topic 且开始流式回复。失败有红字，按钮可重试。**就这么多**，无视觉高亮、无 streaming disable、无 cache、无 anchor list。

**为什么先不做 useCache**

cache 是给"有消费者"的数据建表。当前没有视觉高亮（用户看不到 anchor 在哪）、没有 anchor 列表 UI、没有重开 Panel 的入口——anchor 写进 cache 后没有任何代码会去读。先做 cache = 提前给一个还没出生的功能搭脚手架，违反"无消费者不写"原则。等 T-006E 高亮真的需要点击 mark 重开 Panel 时，cache 才有第一个真正读取者，那时一起建。

**为什么先不做 streaming disable**

它与真 fork 完全正交。"流式中阻止开分支"是用户体验保护，但用户即使在流式中开了分支，只会拿到一个截断的 selectedText——分支照样能创建。失败模式是"模型回答可能不准"，**不是数据错误**。所以可以推后到独立小任务（D-2B），让 D-2A 这个真正改变功能行为的 PR 专注、好 review、好回滚。把 disable 拆出去还能复用：D-3 的"针对此处提问"按钮也需要同样的 disable。
