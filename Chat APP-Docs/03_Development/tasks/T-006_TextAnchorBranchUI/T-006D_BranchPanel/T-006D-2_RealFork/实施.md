## 关键设计决策

### 1. 复用既有 `sendMessage` 而非新 thunk

源码验证：

- `TopicService.create`（`src/main/data/services/TopicService.ts:121–160`）把 `sourceNodeId` 存为新 topic 的 `activeNodeId`，**不复制路径、不继承 assistant**。
- `StreamingService.createUserMessage`（`src/renderer/src/services/messageStreaming/StreamingService.ts:538, 549`）明确不传 `parentId`，由 DataApi 用 `topic.activeNodeId` 当 parent。

→ 新 topic.activeNodeId 自动是 sourceMessageId，调既有 `sendMessage` 时新 user message 自动以源 assistant message 为 parent。lineage 走通。无需注入 parentId、无需克隆消息、无需新 thunk。

### 2. POST /topics body 构造

```ts
{
  name: anchor.selectedText.trim().slice(0, 30) || 'Branch',
  sourceNodeId: anchor.messageId,
  ...(assistant.id !== 'default' ? { assistantId: assistant.id } : {})
}
```

- `assistantId` 哨兵 'default' 时 **omit**（DTO 是 `.partial()`，传 undefined 不会序列化字段）。这是"新 topic 不绑定 assistant"，**不是**"继承源 topic"（server 不做继承）。
- sendMessage 取的 `assistant` 是参数注入，不查 `topic.assistantId`，所以即使 topic.assistantId=null 也能正常发。
- `name` 截 30 字符；空白选区 fallback 字面量 `'Branch'`。
- **groupId 暂不传**（renderer Topic 类型未含 groupId；架构方案 §5 中描述的"groupId 透传"在 renderer 边界拿不到）。已知后果：新 topic 落到 root；用户可手动拖入原分组。后续 v2 完整迁移时一并修。

### 3. 必须调 `addTopic` 防 `useActiveTopic` snap-back

`useTopic.ts:49–53` 的 useEffect：若 activeTopic.id 不在 `assistant.topics[]`，立即回弹到 `topics[0]`。所以创建完 v2 topic 后必须 `dispatch(addTopic({ assistantId, topic: rendererShape }))` 把它注册进 Redux，**然后**才能 `setActiveTopic(newTopic)`。

`addTopic` 是纯 Redux reducer（`store/assistants.ts:125`），不写 Dexie，符合 v2 throwaway 策略。

### 4. 只有 POST /topics 暴露 error 态

`sendMessage` 内部 try/catch 把所有错误吃掉只 `logger.error`（`messageThunk.ts:1080–1082`）— Promise 总是 resolve。所以：

- POST /topics 失败 → BranchPanel 显示红字 + Cancel/Create 恢复
- sendMessage 失败 → 用户在**新 topic** 内看到一条 ERROR-status 消息（与任意其他 topic 发送失败 UX 一致）

实现里 `void dispatch(sendMessageThunk(...))` 不 await — 失败不阻塞 success 回调。这是**已知接受**，不打算改 sendMessage thunk（用户明确要求"不改 messageThunk 现有逻辑"）。

### 5. UI 不变量

- `status='creating'`：Cancel + Create 双 disabled；spinner 在 Create 内；ESC / 遮罩点击通过 `handleOpenChange` 被吞掉
- 失败 (`status='error'`)：错误红字 + followUp 文本**不清空**（用户可改后重试）
- 成功：`onSuccess?.()` 触发 host 把 `branchAnchor` 置 null → Panel 关闭 → useEffect 清空 followUp

## 已知限制（写明，留作 D-2B / D-3 / E 处理）

1. **groupId 不透传** — 新分支落 root，不在源 topic 的 group 里。
2. **流式中可对未完成文本开分支** — D-2B 计划用 `data-block-streaming` 在 SelectionContextMenu 拦截。
3. **anchor 无持久化、无高亮、无重开** — 关 Panel 后 anchor 信息丢失。T-006E 高亮做完才有真消费者。
4. **sendMessage 失败不回滚新 topic** — 用户看到空分支 + ERROR 消息，可在 Inputbar 重发。
5. **selectedText 不嵌引用模板** — 模型靠 lineage 拿到源 assistant message 上下文。
