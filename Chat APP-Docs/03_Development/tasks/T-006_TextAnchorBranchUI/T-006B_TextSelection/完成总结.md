## 解决了什么

铺设了 T-006C–E 的**统一基础设施**：从任意 `Selection` 拿到 `(messageId, blockId)` 三元组，且能正确识别跨 block 选区。

## 设计决策（实施时落地的）

1. **放在 `utils/branchAnchor/`** —— 与既有 `utils/messageUtils/` 同 pattern；纯 DOM helper 适合 utils 而非 components
2. **`messageId` 是 required prop** —— 不给 default，让 TS 强制 caller 传；Blocks/index.tsx 一处传值即可
3. **wrap mention 之外的内容** —— mention header 不进 anchor 范围，跨 mention 选区返回 null 是正确行为
4. **不加 `data-block-streaming`** —— 严格守 Surgical Changes 边界；T-006C 来加更合适（紧贴它的使用点）
5. **测试用 jsdom + createRange** —— 不引入 testing-library Selection mock，纯 DOM API 已足够

## 经验

- **CLAUDE.md「Surgical Changes」是真好用** —— 第一反应想顺手加 `data-block-streaming` 给 T-006C，但严格守在 T-006B scope 里反而让 diff 更小、commit 更清晰、下游任务能独立验证
- **既有测试 fixture 要兼容** —— 加 required prop 时记得改既有 test helper，否则 typecheck 直接挂；这次 catch 在 typecheck:web 阶段，省得跑测试发现
- **「`utils/` 子目录」pattern 沿用** —— 不发明新位置，照 `messageUtils/` 抄结构

## 给后续任务的"接口"

### T-006C 直接可用

```ts
import { findBlockContext } from '@renderer/utils/branchAnchor/findBlockContext'

// 在 SelectionContextMenu.handleOpenChange 里：
const selection = window.getSelection()
if (!selection || selection.rangeCount === 0) return
const range = selection.getRangeAt(0)
const blockContext = findBlockContext(range)
// blockContext === null 时把新菜单项 disabled，已有 Copy/Quote 保留
```
