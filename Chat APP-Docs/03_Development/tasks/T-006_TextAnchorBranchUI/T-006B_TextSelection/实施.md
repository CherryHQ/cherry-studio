## 1. `findBlockContext` 实现要点

`src/renderer/src/utils/branchAnchor/findBlockContext.ts` —— 单纯 DOM helper。

**契约**：
- 入参：`Range | null | undefined`
- 出参：`{ messageId, blockId } | null`
- 失败场景全部返回 null（不抛）：
  - 空 / collapsed range
  - 任一端点未在任何 `[data-block-id]` 祖先内
  - 两端点在不同 block（跨 block 选区）
  - block 没有 `[data-message-id]` 祖先（防御）

**关键设计**：
- 用 `Element.closest()` 一行爬树，文本节点先 step up 到 `parentElement`
- 跨 block 检测 = 比较 `startContainer` 与 `endContainer` 各自解出的 blockId
- **未引入任何 React / Redux 依赖** —— 是纯函数 DOM 工具，方便测试 + 多处复用（T-006C / T-006D / T-006E）

**放置位置**：`utils/branchAnchor/` 子目录，与既有 `utils/messageUtils/` 同 pattern。

### 为什么是 `<div>` 而不是把属性直接挂 Markdown.tsx 的根 div

- 改 `MainTextBlock` 一处比改 `Markdown.tsx` 接口侵入小
- 既要覆盖 markdown 路径，又要覆盖 `role==='user' && !renderInputMessageAsMarkdown` 的 `<p>` 路径 —— wrapper 是唯一统一点
- 多一层 div 对 CSS 影响微乎其微（Markdown.tsx 本来就用 `<div className="markdown">` 做容器）

### 是否加 `data-block-streaming`

按 T-006C 设计原本要加。但本任务严格只做 capture context（per CLAUDE.md「Surgical Changes」），T-006C 实施时再加即可，本次不预投资。
