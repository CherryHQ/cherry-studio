# T-006F 折叠 / 展开交互

**状态**：🤔 **scope-uncommitted**（v1.1 候选，本次 v1.0 不一定做）
**依赖**：T-006D + T-006E 完成且有用户反馈
**预估**：0.5 天（若做）

## 目标候选

让 anchor 不只是「跳到另一个 topic」，而是能在**原消息附近 inline 展示分支对话**。

## 候选形态

| 形态 | 描述 | 复杂度 |
|---|---|---|
| **A** | 高亮处右上角显示 `→ 2 replies` 小 badge；点击在高亮下方 inline 折叠展开 mini chat | 中 |
| **B** | 所有 anchor 在消息流末尾以 footnote 卡片列出，可点击展开 | 低 |
| **C** | 树状 tooltip：hover 高亮显示 mini 时序图 | 高 |

## 为何 v1.0 不做

1. v1.0 目标是「先验证文本 anchor → fork 的核心交互链路」—— 用户能完整走通 选中→菜单→Panel→新 topic，已经是闭环
2. 折叠展开依赖 v1.0 用户反馈：若用户都用 fork 切新 topic，inline 展开可能根本没需求
3. inline 折叠展开会牵扯到流式渲染兼容、消息流滚动定位等复杂度，需要单独评估

## 触发条件

T-006D + T-006E 上线后观察 **2 周** 真实使用：
- 若用户经常说"我想看哪几段被人讨论了" → 做 **B**（footnote 列表）
- 若用户说"我希望不离开当前 topic 也能问" → 改 T-006D，让 "针对此处提问" 默认走 inline mini-chat 而不是注入 inputbar
- 若都没明显需求 → **不做**，给用户的是「fork 切走」单一路径
