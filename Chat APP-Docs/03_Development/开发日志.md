### 现象
用户删 `~/Library/Application Support/CherryStudioDev` + `pnpm dev` 后，新建 topic 仍失败：
- `Assistant preset with id default not found in state.`
- `POST /topics body: { name: 'Default Topic', assistantId: 'default' }`
- `SQLITE_CONSTRAINT_FOREIGNKEY: FOREIGN KEY constraint failed`

### 根因
**renderer `AssistantService.ts:77-89` 的 `getDefaultAssistant()` 合成 `id: 'default'` 字面量 sentinel，v2 后端把它当真 UUID 写到 `topic.assistantId` FK 上 → SQLite 没有 `id='default'` 的 assistant 行 → 拒绝**。

链路：
```
getDefaultAssistant() → { id: 'default' }
  → Redux assistants.initialState.defaultAssistant
    → useAssistant().defaultAssistant
      → Inputbar.tsx:325 getDefaultTopic(assistant.id)
        → mapLegacyTopicToDto({ assistantId: 'default' })
          → POST /topics → topicService.create
            → INSERT INTO topic (assistantId='default')
              → ❌ FK constraint failed
```

### 关键证据
- `AssistantService.ts:79` 字面量 `id: 'default'`
- `store/assistants.ts:39-40` Redux initialState 使用 sentinel
- `Inputbar.tsx:324-330` 触发 POST /topics
- `AssistantService.ts:173` `mapLegacyTopicToDto` 有 `// TODO: remove it in v2` 但未做
- `db/seeding/index.ts:12-17` 已注册的 4 个 seeder **不含 AssistantSeeder**
- `AssistantMigrator.ts:340-342` & `ChatMigrator.ts:350-352` 明确："v2 has no system-reserved 'default' row"
- 上述迁移器有 v1 → v2 的 'default' UUID 重映射，但**只在 v1 数据迁移路径生效**；fresh install 不走迁移器

### 修复方案（3 选 1）
| 方案 | 改动 | 符合 v2 设计 | 适合 baseline |
|---|---|---|---|
| A：seed 一行固定 id assistant | 新增 seeder | ❌ 违背"no system-reserved 'default' row" | ❌ |
| **B：`mapLegacyTopicToDto` 转 'default' → null** | 一行 | ✅（topic.assistantId 可空） | ✅ **推荐** |
| C：renderer 启动时合成真实 UUID | 启动序列改动 + Redux init | ✅ 最对齐设计意图 | ⚠️ 重，留给正式修复 |

- `descend` flag 在主分支已删 + 加了回归测试，等下游 consumer 一起加回来
- **Fork 是「指针重定向」而非「快照复制」**：新 topic 的 `activeNodeId = sourceNodeId`，原树消息仍属于源 topic

**服务算法：**
- `getTree` 用递归 CTE + active path 强制完整展开 + 深度溢出补偿，6 步流程；depth 三态（-1/0/n）语义清晰
- `getBranchMessages` 的 "before cursor" 是**数组位置**而非时间戳，跨多窗口编辑导致的乱序不会跳行
- `MessageService.create` 的 parentId 三态（undefined / null / string）逻辑完整；`undefined + 有消息但无 activeNodeId` 会显式抛错避免歧义
- `MessageService.delete` cascade=false 时把直接子节点 reparent 到祖父，仅一层；孙节点不动
- `TopicService.listByCursor` 的 pin/unpin 两段 cursor 跨界处理（pin: / topic:updatedAt:id / topic: 三态）
- `AssistantService` 关联数组三态统一：`undefined → no-op` / `[] → 清空` / `[...ids] → 完全替换`
- `TemporaryChatService` 纯内存 Map；persist 时快照-清空-写，失败回滚

**事件与 IPC：**
- `EVENT_NAMES` 25 个常量，与聊天直接相关约 16 个
- `SpanManagerService` 重度依赖 EventEmitter 关联 trace span —— 是脆弱点
- 主干通信通道：**`DataApi_Request` 一根 IPC channel 承载所有聊天数据 CRUD**
- Agent session 走独立的 `AgentMessage_*` / `AgentSessionStream_*` IPC 组（未迁 DataApi）
- `DataApi_Subscribe` 端点存在但 renderer 未消费 → 多窗口数据同步当前**不会自动**

### 已知脆弱点（汇入下次风险清单）
- `NEW_BRANCH` 事件是 v1 概念，与 v2 `siblingsGroupId` / `activeNodeId` 不是一回事
- `CHANGE_TOPIC` 事件 + `setCurrentTopicId` Redux action 双轨
- EventEmitter payload 无 TS 类型保护，emit / listen 名字不匹配时静默失败
