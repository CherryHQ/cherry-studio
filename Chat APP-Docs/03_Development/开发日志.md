### 关键发现

**分支对话：**
- Schema + service + 测试 **全部就位** —— tree / branch / cursor 分页 / sibling 聚合 / fork / setActiveNode 都有实现
- Renderer **零消费 v2 概念**：`siblingsGroup` 和 `activeNodeId` 在 `src/renderer/src/pages/home/Messages/` 下零命中
- 现存的 `MessageGroup*.tsx` 都是 v1 形态（`askId` + `foldSelected`）
- v2 的 `SiblingNavigator.tsx` / v2 `MessageGroup.tsx` 在 `DeJeune/ai-service` 分支，等待合入（`topics.ts:67-73` 注释明确说了）
- `descend` flag 在主分支已删 + 加了回归测试，等下游 consumer 一起加回来
- **Fork 是「指针重定向」而非「快照复制」**：新 topic 的 `activeNodeId = sourceNodeId`，原树消息仍属于源 topic

**服务算法：**
- `getTree` 用递归 CTE + active path 强制完整展开 + 深度溢出补偿，6 步流程；depth 三态（-1/0/n）语义清晰
- `getBranchMessages` 的 "before cursor" 是**数组位置**而非时间戳，跨多窗口编辑导致的乱序不会跳行
- `MessageService.create` 的 parentId 三态（undefined / null / string）逻辑完整；`undefined + 有消息但无 activeNodeId` 会显式抛错避免歧义
- `MessageService.delete` cascade=false 时把直接子节点 reparent 到祖父，仅一层；孙节点不动
- `TopicService.listByCursor` 的 pin/unpin 两段 cursor 跨界处理（pin: / topic:updatedAt:id / topic: 三态）
- `AssistantService` 关联数组三态统一：`undefined → no-op` / `[] → 清空` / `[...ids] → 完全替换`
- `TemporaryChatService` 纯内存 Map；persist 时快照-清空-写，失败回滚

**事件与 IPC：**
- `EVENT_NAMES` 25 个常量，与聊天直接相关约 16 个
- `SpanManagerService` 重度依赖 EventEmitter 关联 trace span —— 是脆弱点
- 主干通信通道：**`DataApi_Request` 一根 IPC channel 承载所有聊天数据 CRUD**
- Agent session 走独立的 `AgentMessage_*` / `AgentSessionStream_*` IPC 组（未迁 DataApi）
- `DataApi_Subscribe` 端点存在但 renderer 未消费 → 多窗口数据同步当前**不会自动**

### 已知脆弱点（汇入下次风险清单）
- `NEW_BRANCH` 事件是 v1 概念，与 v2 `siblingsGroupId` / `activeNodeId` 不是一回事
- `CHANGE_TOPIC` 事件 + `setCurrentTopicId` Redux action 双轨
- EventEmitter payload 无 TS 类型保护，emit / listen 名字不匹配时静默失败

### 关键修正（vs 2026-05-19）
- **"聊天 UI 100% 走 Redux" 是错的**。写路径已经走 DataApi；读路径走 DataApi → 桥层 → Redux；UI 还是订阅 Redux。准确说法是"DataApi 是真源、Redux 是显示缓存"。
- R2 改写为「没有块级端点；块 ops 走整条消息替换」—— 这不是"端点缺失"而是"契约取舍"。

### 关键发现
- **13 个聊天 DataApi 端点全部有 schema + handler + service**，编译期强约束（`apiHandlers` 必须穷尽 `ApiSchemas` 否则 ts 报错）
- **`/topics` POST 支持 fork**（`sourceNodeId` 字段），不需要单独的 `POST /topics/:id/fork`
- **`POST /topics/:topicId/messages` 的 `parentId` 三态语义**：omit / null / string —— omit 时多窗口竞态已被注释为 TRADEOFF
- **Pin 不在 Topic 上**，是独立资源（`POST /pins` / `DELETE /pins/:id`）
- **Agent session 有平行端点**（`/agents/:agentId/sessions/:sessionId/messages*`）但 StreamingService 未消费 —— 单独的迁移分支
- **临时会话端点不对称**：无单条读、无树、无 branch 分页 —— 设计上临时会话本就线性

### 关键发现（摘录）

- **会话载体就是 Topic**（v1、v2 都是），无独立 `Conversation` 类型。
- **聊天 UI 数据流是 v1（Redux + Dexie）100% 主导**：5+ 个核心聊天组件实测仅订阅 Redux，DataApi 端点已就位但聊天侧未消费。
- **Redux 聊天 slices 已标 `@deprecated v2.0.0`**：`store/messageBlock.ts` 顶部 + `store/thunk/messageThunk.ts` 顶部。`messageThunk.ts:1332` 留有 `// TODO: Migrate block deletion to Data API when block endpoints are available`。

- **LLM 请求由 renderer 直连**，不走 IPC。Main 端只处理 RAG / 文件 / DataApi 业务数据 / Agent 历史。
- **v1 Block 与 v2 Block 的最大差**：v1 块独立表 + `message.blocks: string[]` 引用；v2 块直接内嵌 `message.data.blocks[]`（JSON）。

- **redux-persist version 206**，blacklist 含 `messages / messageBlocks` — 即聊天数据不走 localStorage，依赖 Dexie 持久 + 启动时 hydrate。
