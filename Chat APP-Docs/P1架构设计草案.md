# P1 架构设计（定稿 v1）：多 branch 并存 + 分型 + 去向决策

> 设计窗口产物 · 决策已拍板 · 据此拆 S1→S2→S3 执行。
> 范围对齐 roadmap P1。**P1 全程不碰持久化**——这是后面所有取舍的总闸。

---

## 0. 目标与边界

**P1 交付**：同一主对话里可同时存在多个 branch；右侧面板是**可同时展开的卡片列表**（你能一眼看到所有 branch 的问与答）；源文本可有多处高亮、每 branch 一色、用户能映射高亮↔卡片；关闭 branch 按"去向"决定丢弃（删 topic）或保留（不删）。

**明确不做（写死防越界）**：
- 不持久化 `branches[]`——刷新即丢（P1↔P2 缝）。
- **不做"页边锚定 + 连线"布局**（你贴的那张批注图）——它是状态地基稳了之后的下一个视觉里程碑，量级近 E4 Branch Map，不进 P1。
- 不做保存 branch 的完整重访 / 锚点持久化 / 高亮重建 / branch 摘要（P2）。
- 不做"右键高亮、模仿其他软件"那整套 UX（菜单组织 / 与 `SelectionContextMenu` 融合 / 独立于 branch 的纯高亮）——地基稳后单开一轮设计。
- 不深绑 v1 Redux/Dexie；branch 状态走内存。

---

## 1. State shape（目标全貌 + 字段到位时机）

source of truth 仍在 `Chat.tsx` 顶级 state（沿用现有架构，不引入 Redux）。

```ts
interface Branch {
  id: string                       // 客户端 branch id（≠ topic.id）；UI / 高亮归属的稳定键
  source: {
    messageId: string
    blockId: string
    selectedText: string
    offsets: { start: number; end: number }
  }
  topic: Topic | null              // fork 出的 topic；POST 在途时为 null（沿用现有时序）
  color: string                    // 调色板 key —— S2 引入
  disposition: 'pending' | 'kept'  // 关闭去向 —— S3 引入
  createdAt: number
}

interface BranchPanelState {       // 内存态，住 Chat.tsx
  branches: Branch[]
  collapsedBranchIds: Set<string>  // 默认不含 = 展开（多展开模型）
  panelWidth: number               // 沿用 BranchPane 现有，可后续上提
}
```

**字段到位时机（避免无人消费的 dangling 字段——上一仗的教训）：**

| 字段 | 何时引入 | 谁消费 |
|---|---|---|
| id / source / topic / createdAt | S1 ✅ | 现有 paint + synthetic assistant |
| collapsedBranchIds | S1 起存在 ✅；S2b 起有 UI 意义 | 卡片折叠 |
| color | S2a | 多高亮上色（data-hl）+ 卡片色块 |
| disposition | S3 | 关闭去向路由 |

**去掉 `activeBranchId`**：多展开下"唯一展开项"无意义；高亮强调改 **hover 双向联动**；新建 branch 自动展开 + 滚动到它。

## 2. 两个派生 context 如何泛化（S1 的核心）

现状（交接）：`Chat.tsx` 两个 useState → 派生两个 context。

- **BranchAnchorContext**：单个 `{ highlightedBlockId, selectionStart, selectionEnd }`（默认 null）→ 泛化成**列表** `{ anchors: { branchId, blockId, selectionStart, selectionEnd }[] }`（默认 `{ anchors: [] }`）。MainTextBlock effect 从"读单个 `highlightedBlockId`"改成"遍历 anchors，对命中本块 blockId 的逐个 paint"。**S1 阶段 anchors 长度 ≤1，行为与现在完全一致。**
- **BranchAssistantContext**：synthetic `{...main, topics:[...main.topics, branchTopic]}` → `{...main, topics:[...main.topics, ...branches 的 topic]}`。messageThunk 按 `branchTopicId` 命中不变。**长度 ≤1 时一致。**
- **`__BRANCH_ANCHOR_CTX_CACHE__` HMR 缓存不动**（交接点名：删了高亮失效 / D-013-FIX 真根因之一）。

## 3. 多高亮（S2a 实现 / S2b UI）

> 拆分原则：**实现先行**。span 打标 + 定向 clear + 调色板属于"动 sourceHighlight.ts"的危险改动，放 S2a 单独验透（行为仍 ≤1 branch）；放开 length>1 的 UI 放 S2b，骑在已验证的定向 clear 上。

- span：`<span class="branch-anchor-highlight" data-branch-id="X" data-hl="c3">…</span>`
  - `data-branch-id`：**承重**——定向 clear 靠它 `[data-branch-id="X"]` 只移除某 branch 的 span。**必须有真测试**（别成下一个假绿）。
  - `data-hl`：调色板 key，CSS 上色。
- 颜色：固定调色板（CSS 变量 `--branch-hl-1..n`）；创建时自动分配下一个，卡片色块可改。
- **不变量（上一轮测试成果的多 branch 版，S2a 必须 mutation/red-test 验）**：
  - paint(B) 不动 A 的 span；clear(A) 只移除 A 的、保留 B 的；
  - 仍 `replaceWith(...children)` + normalize，无空 span、无累积。
- 区分主要靠颜色 + hover 联动；重叠选区允许 span 嵌套、按 `data-branch-id` 定向 clear，"高度重叠视觉可能叠加"列为已知小限制（决策 3）。

## 3b. 面板 UI 规格（accordion 手风琴 · 定稿）

> 演进记录：
> 1. S2b-1/2 "竖直折叠卡片栈"。
> 2. S2b-3 想让所有标题滚动时常驻 → `display:contents` 手搓 sticky-stacking → **B1 卡片重叠**（弃）。
> 3. S2c 改 master/detail（标题全在顶上 master 区、正文全在下面 detail 区，两区分开）。dev-smoke 暴露：master/detail 的"标签与正文分家"使得"Branch N 标题下方夹着别的 branch 内容"，**不符合产品预期**——用户要的是"标题正下方就是它自己的内容"。**弃 master/detail。**
> 4. **定稿 = accordion 手风琴**（用户画图确认）。注：S2c 的 shrink-0 真实盒子修复、B3 统一键盘、BranchTab/BranchDetail 组件都复用，accordion 只是把"两区"重新并成"一个滚动区 + 标题贴内容" + 加 locate。

右侧面板 = **单一滚动区**，所有分支按**创建顺序**竖排，每个分支 = 一个盒子（已用户画图确认）：

- **每个分支盒子** = 标题（始终显示）+ 内容（仅展开时显示，紧贴标题正下方）。
  - 标题：彩色编号块 + 选中文字片段（截断）+ chevron（展/收）+ X（关闭）；颜色 = 该分支高亮色。
  - 内容（展开时）：compose 态 = selected passage + 初始 composer + Create/Cancel；conversation 态 = selected passage + 对话流 + follow-up composer。
- **折叠 = 只剩标题**；**展开 = 标题正下方直接是它自己的内容**（标题↔内容贴合，中间不夹别的分支）。
- **多展开**：toggle `collapsedBranchIds`，多个可同时展开。
- **滚动**：盒子是 `shrink-0` 真实盒子（S2c 修复），单滚动区接管滚动；**标题随内容一起滚**——长内容下滑时上面分支的标题会滚出视野（**不 sticky、不 `display:contents`**；用户已接受此取舍）。严格创建顺序，Branch N+1 永远在 N 下面。
- **locate（自动定位）**：新建分支 → 默认展开 + 其标题 `scrollIntoView(block:start)` 滚到顶；点折叠分支标题 → 展开 + 滚到顶；点已展开标题（chevron）→ 折叠（不滚）。
- **compose composer label**：给初始 composer 自己的 label，不再撞用 "Follow-up question"。
- **颜色分配**：创建时取调色板未占用色（占满再循环）。
- **面板开关**：开第一个分支面板出现，关掉最后一个收起。
- **X 语义**：仅"从 branches[] 移除 + clearSourceHighlight(其 id)"；**流式中关闭要 abort（B5，单独成步）**；disposition（丢弃/保留）是 **S3**。
- **可选后续增强（非现在，且与翻车那套不同）**：只让**当前**分支标题在滚它自己内容时 sticky 顶住（标准分节头，不堆叠全部标题、不用 `display:contents`）。先把朴素手风琴跑通。



- **创建**：选区 → 新 Branch(生成 id) → push → 自动展开 + 滚动 → POST fork topic → topic 回填 → paint。
- **折叠/展开**：点卡头 toggle `collapsedBranchIds`；多张可同时展开。
- **关闭（卡片 X）**：
  - `pending`（默认）→ DELETE fork topic（吸收 orphan 债）+ 移除 branch + clear 其高亮。
  - `kept` → 不删 topic；移出面板但进"本会话保留列表"可见。
  - 翻成 kept 的时机：branch 内"收藏/转笔记"动作（P1 先用显式「保留」按钮）。

**P1↔P2 缝**：保留 = topic 不删（SQLite 活着）+ 本会话列表可见；但 branches[] 内存态、刷新丢，刷新后找回它（点回去 + 高亮重建）是 P2。**S3 别顺手做持久化。**

## 5. 执行步骤

> 每步：真行为测试（非假绿）；CC 做完停下等确认 + 更新 Chat APP-Docs；GitHub Desktop 手动提交。

- **S1 ✅ — 状态地基（单 → 数组，零行为变化）** [已完成并经浏览器 smoke 验证]
  - `Chat.tsx` 两 useState → `branches[]` + `collapsedBranchIds`；Branch 只含 S1 字段（id/source/topic/createdAt）。
  - 两个 context 单 → 列表泛化（见 §2）；MainTextBlock effect 遍历 anchors。
  - **不碰 sourceHighlight.ts**（paint/clear 实现不动，无 data-branch-id/color）；**不碰 HMR 缓存**。
  - branches[] 长度保持 ≤1（不加多建 UI）。
  - 验收：36 测试全绿；sourceHighlight.ts git diff 空；浏览器 open→close 高亮干净、streaming 不闪不漂。

- **S2a ✅ — 高亮实现扩展（行为仍 ≤1 个 branch，单独验透危险模块）** [已完成：data-branch-id + data-hl + 定向 clear；M1/M2 mutation 通过]
  - paint 盖 `data-branch-id` + `data-hl`(颜色)；`clearSourceHighlight` 改按 `data-branch-id` 定向。
  - 加 `color` 字段 + 固定调色板（CSS 变量 `--branch-hl-*`）；S2a 创建时给默认色（cycle 到不同色留 S2b）。
  - 扩 `sourceHighlight.test.ts`：多 branch-id 注入 / 定向 clear 只删自己不碰别人 / 嵌套选区定向 clear / 保持 `replaceWith`+normalize 不变量。
  - **重新上 mutation/red-test**（这步真动 sourceHighlight.ts）。
  - 边界：UI 仍只能建 1 个 branch；不做卡片列表 / 折叠 / hover。
  - 关键认识：**测试可直接构造多 branch-id 的 DOM 夹具来验定向 clear**——实现支持 N，即便 UI 还停在 1（同 S1 用 ≤1 夹具的思路，反向）。

- **S2b-1 — 多分支跑起来 + 折叠卡片栈（集成步，sourceHighlight.ts 永久不改）**
  - SelectionContextMenu/openBranchAnchor 从 replace 改 **append**；颜色 cycling（取未占用色）。
  - 渲 N 张折叠卡片栈（见 §3b）：标签头 + 收/展（消费 `collapsedBranchIds`）+ 每张展开态有自己的对话 + composer + X（仅移除+clear，不做 disposition）。
  - 多高亮在真实多 block 上经 effect **共存**（data-hl 颜色真的变了）；面板开关随首个/末个分支。
  - 测试：append 语义 / 颜色 cycling 给不同色 / N 张卡渲染 + 折叠 toggle / 经 effect 的 [A]+[B] coexist + 关 A 留 B（定向 clear 在 N>1 真路径）/ **data-hl mutation**（验颜色断言非空转，临时改 sourceHighlight.ts 后还原，最终字节不变）。
  - **集成风险（写进 prompt 防雷）**：N 张卡 = N 个 `MessageGroup`，各按自己 `branchTopicId` 解析正确 topic、仍待在 `BranchAnchorContext` Provider **外**、互不串台。
  - 边界：不做 hover 联动 / 自动滚动 / 立体标签样式（S2b-2）；不做 disposition（S3）。

- **S2b-2 — 交互打磨（骑在能跑的多分支上）**
  - hover 卡片↔高亮双向强调；新建自动展开 + 滚动入视；标签"探出/斜边"立体样式 + 动画。

- **S3 — 去向决策**
  - 关闭按 disposition 路由 + 保留列表最小可见面 + 「保留」入口。

## 一句话

P1 = 把"单 branch"长成"一组可同时展开、各自带色的 branch"，关闭时用去向落地分型。
守两条底线：**多高亮定向 clear 的不变量**（前一轮测试的延伸）+ **内存态不持久化的边界**（防 S3 越界进 P2）。
