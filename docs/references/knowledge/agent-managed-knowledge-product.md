# Agent 管理型知识库产品文档

本文沉淀本轮 `$grill-with-docs` 讨论后确认的知识库产品语义。它描述产品行为和边界，不约束具体表结构、IPC 形态或最终 UI 组件实现。

## 1. 产品定位

Cherry Studio 的新知识库应从“模型优先的 RAG 配置”转向“文件夹优先、Agent 可管理的资料空间”。

用户创建知识库时只需要输入名称，系统创建一个独立的知识库根目录。用户可以无脑往这个目录里添加文件、文件夹、URL、笔记快照或未来的云端文档快照；Cherry Studio 负责让这些材料可见、可搜索、可被 Agent 使用。Embedding、rerank、文件处理器等能力是增强项，不是创建知识库的前置条件。

核心愿景：

```text
创建知识库
  -> 得到一个真实文件夹
  -> 添加文件、文件夹、URL、笔记或云端来源
  -> 来源被复制或沉淀为知识库自有资料/快照
  -> UI 展示真实目录
  -> 系统维护全文索引
  -> 配置 embedding 后增强为语义检索
  -> Agent 可搜索、读取、整理、刷新、添加、删除资料
```

用户侧仍称为“知识库”。“File Mode”只可作为工程内部或历史文档中的过渡称呼，不应作为面向用户的主要文案。

## 2. 产品原则

### 2.1 导入即复制，知识库拥有稳定副本

第一原则：用户通过上传或导入来源，就是把来源复制或沉淀为知识库自有资料/快照。

这符合用户对“上传”的直觉：上传完成后，知识库里有一份稳定的、属于知识库自己的材料。外部来源后续怎么变化，不会自动改写知识库中的内容。

这个原则适用于所有来源：

- 文件：复制文件到知识库目录。
- 文件夹：复制文件夹及其层级到知识库目录。
- Cherry Studio 笔记：复制该笔记索引到的本地源文件，文件名使用源文件名。
- URL：抓取内容并保存为 Markdown 快照。
- 云端文档：未来进入知识库时保存为本地快照。
- Agent 添加材料：遵循和用户手动添加一致的复制或快照语义。

因此，知识库删除、刷新、处理、索引和 UI 展示都应围绕“知识库自有副本/快照”来设计，而不是围绕外部实时引用来设计。

### 2.2 真实文件夹是用户可见事实

每个知识库有自己的本地根目录：

```text
knowledge-root/
  <baseId>/
```

创建空白知识库时立即创建该目录。用户上传文件或文件夹时，系统通过 main/file 的文件复制能力把原文件复制到这个目录下，不主动改名、不重排、不拆分目录结构。

如果上传的是文件夹，保留原文件夹层级。

### 2.3 UI 展示真实目录，不维护虚拟目录表

知识库 UI 的目录展示以文件系统 watcher / fs list 为准。用户或 Agent 可以按自己的方式组织目录，例如：

```text
raw/
processed/
reports/
```

也可以完全平铺。产品不强制 `raw/`、`processed/` 等固定目录。

这意味着 UI 不通过额外表保存“应该显示什么文件”，也不让 metadata 表替代真实目录树。如果用户或 Agent 创建 `raw/`、`processed/` 这样的目录，UI 只把它们当普通目录展示，不赋予固定产品语义。如果 Agent 移动或重命名文件，UI 通过 watcher 看到真实目录变化并展示最新路径。

系统内部资产不展示，例如：

```text
.cherry/
```

索引、内部状态、缓存等系统资产不属于用户材料。

### 2.4 索引是派生产物，用户材料是文件夹中的可见内容

知识库里的可见文件是用户或 Agent 可以管理的材料。索引、chunk、向量、缓存是系统派生产物，不出现在 UI 的材料目录中。

文件处理器生成的 Markdown 是例外：如果用户明确选择了 MinerU、Paddle 等文件处理能力，那么 PDF 处理后生成的 Markdown 是用户可见材料，不是隐藏索引资产。隐藏的是 OCR 临时文本、chunk、向量、缓存等系统 representation；显式处理器产出的 Markdown 是 processed material。

## 3. 用户材料类型

### 3.1 本地上传材料

用户上传文件或文件夹时：

- 系统把源文件复制到知识库根目录或当前目标目录。
- 文件名和目录层级默认保持不变。
- 上传成功后，知识库拥有这份本地副本。
- 删除知识库中的文件，不会删除用户原始外部文件。

### 3.2 URL 快照材料

用户添加 URL 时：

- 系统通过 Jina fetch 获取 URL 内容。
- 获取结果沉淀为 Markdown 文件。
- Markdown 文件保存在当前目录，像普通文档一样出现在 UI 中。
- 默认文件名暂定来自页面标题；没有标题时使用 URL fallback。
- URL 内容是快照，不是实时引用。

URL 快照支持刷新：

- 用户点击刷新，或 Agent 调用刷新工具。
- 刷新会重新抓取 URL。
- 刷新会覆盖当前快照文件，包括用户对该快照文件的本地编辑。
- 覆盖前需要提示用户确认。
- 刷新完成后更新相关索引。

如果用户移动或重命名 URL 快照文件，刷新身份仍应尽量保留，刷新时覆盖移动后的那个文件，而不是重新创建到原路径。

### 3.3 云端快照材料

云端文档是后续功能，本轮暂不细化权限和同步策略，但技术设计必须预留。

暂定语义：

- 云端文档进入知识库时保存为本地快照。
- 快照手动刷新，不实时同步。
- 刷新覆盖当前快照文件，刷新前需要确认，刷新后更新对应索引。
- 默认放在服务商目录下，例如 `飞书/`、`腾讯文档/`。
- 服务商目录名需要 i18n 支持。
- 删除知识库中的云端快照只删除本地快照和索引，不删除云端原文档。

云端快照的命名、移动后的刷新路径、同名冲突策略暂缓定稿。

### 3.4 Cherry Studio 笔记快照材料

Cherry Studio 笔记按“导入即复制”的第一原则处理。

当前理解下，Cherry Studio 笔记不是一个需要在知识库中实时引用的内部文档，而是一个外部本地文件索引。用户把笔记加入知识库时：

- 系统复制该笔记索引到的本地源文件。
- 复制后的文件进入知识库当前目录，成为知识库自有快照。
- 文件名使用用户本地源文件名，不使用笔记标题或笔记 id 作为默认文件名。
- 源文件或笔记后续变化，不自动同步到知识库。
- 用户可以手动刷新该笔记快照。
- 刷新会重新读取源文件并覆盖知识库中的当前快照文件，覆盖前需要确认。
- 删除知识库中的笔记快照，只删除知识库中的副本和索引，不删除源文件，也不删除 Cherry Studio 笔记。

如果用户移动或重命名知识库中的笔记快照，刷新身份应尽量保留；如果无法确认来源身份，则不应靠路径猜测。

### 3.5 文件处理生成材料

用户配置文件处理器后，上传 PDF 等富文档会生成可见 Markdown，例如：

```text
资料.pdf
资料.md
```

确认语义：

- PDF 本身进入知识库目录并可被 fs 搜索找到。
- 处理生成的 Markdown 也是独立可见文件。
- 向量和全文索引优先索引 Markdown，而不是同时索引 PDF，避免重复来源。
- 搜索结果返回 Markdown，不返回 PDF。
- fs 搜索能搜到 PDF 和 Markdown 是正常的，因为两者都真实存在。
- Markdown 生成后生命周期独立。
- 删除 PDF 不自动删除 Markdown。
- 移动 PDF 不自动移动 Markdown。
- 重新处理 PDF 会按当前 PDF 位置生成新的 Markdown。
- 如果目标 Markdown 已存在，按普通同名冲突处理。
- 如果用户已经把旧 Markdown 移走，则重新处理不会影响旧 Markdown。

处理生成的 Markdown 可以在 metadata 中记录来源 PDF，但这种 provenance 不绑定生命周期。

### 3.6 Agent 生成材料

Agent 可以在知识库中生成报告、摘要、学习笔记、整理文档等。这些文件是普通可见材料：

- UI 按真实目录展示。
- 可以被索引和搜索。
- 可以被用户或 Agent 移动、重命名、删除。
- 可以记录它引用了哪些材料或 search locator。
- 源材料变化后，可以标记为可能过期，但不自动重写。
- 如果用户后来编辑了 Agent 生成材料，来源关系可以保留，但 UI 应能标记它已被用户编辑，不能继续假装它是未经修改的 Agent 原稿。

产品不设置隐藏的“Agent 成果池”或“加入目录前的正式/非正式文档”状态。Agent 如果只是把内容留在聊天里，它不是知识库材料；Agent 一旦把内容写入知识库目录，它就是普通可见文件。UI 可以在任务完成后就地展示“已生成文件”的反馈和打开入口，但文件归属仍由真实目录决定。

## 4. 冲突、重复和命名

### 4.1 同路径冲突

当新增材料的目标路径已存在时，弹出与文件管理器一致的冲突 dialog：

- 覆盖
- 保留副本
- 跳过

保留副本时自动使用 `_2`、`_3` 这样的中性后缀。

### 4.2 内容重复

产品不按内容哈希阻止重复上传。两个文件内容相同但路径不同，可以同时存在。

重复文档的整理、合并、删除可以交给用户或 Agent 后续处理。

### 4.3 自动改名

同路径冲突选择“保留副本”时自动改名，不再让用户手动输入新名字。

## 5. 删除和移动

### 5.1 删除材料

删除知识库中的材料表示删除知识库保存的本地文件或快照，并清理对应索引。

删除不会影响外部来源：

- 不删除用户上传前的原文件。
- 不删除 URL 对应网页。
- 不删除云端文档。
- 不删除 Cherry Studio 源笔记或源文件。

### 5.2 外部移动和重命名

当前项目中的 watcher 基于 chokidar，外部 Finder 或系统文件操作的 rename 通常表现为 `unlink + add`，不是稳定的 rename 事件。

产品语义上：

- UI 必须反映真实目录变化。
- 索引可以按路径删除旧项、添加新项。
- provenance 和刷新身份只能 best-effort 保留。
- 如果无法确认移动前后的材料身份，不应仅靠路径猜测，应标记 provenance 不可用。
- provenance 不可用不影响该材料作为普通文件继续使用。

应用内发起的明确 rename / move 可以保留更强的身份关系；外部文件系统操作不承诺一定保留。

### 5.3 本地内容变更和外部删除

知识库根目录内的文件内容如果被用户或 Agent 修改，watcher / rescan 检测后应更新对应全文索引；如果该材料启用了语义索引，也应更新对应向量索引。

如果用户在 Finder 或其他文件系统入口删除知识库文件，系统应视为用户已经完成删除动作：UI 移除该文件，并清理对应索引，不再弹出二次确认。应用内发起的删除仍按删除操作规则处理，需要在破坏性场景下确认。

## 6. 来源、metadata 和可追溯性

知识库需要记录来源信息，但不应让 UI 目录依赖这些来源记录。

来源信息用于：

- 搜索结果展示来源类型。
- Agent 判断材料是否是处理产物、快照或 Agent 生成材料。
- URL 和云端快照刷新。
- 处理产物追溯源 PDF。
- Agent 生成材料追溯引用过的 search locator。
- 源材料变化后判断生成材料可能过期。

来源信息可以在嵌入、chunk 或搜索 metadata 中携带，使 search 结果能返回这些上下文。

chunk/search metadata 不是可追溯关系的唯一来源。刷新身份、处理产物来源、Agent 引用过的 locator 等业务层 provenance 需要有可查询的来源记录；chunk/search metadata 只是这些信息在检索结果中的展示副本，方便 UI 和 Agent 使用。

需要区分两类身份：

```text
显示身份：文件当前路径，由真实目录决定
来源身份：材料来自哪里，用于刷新、追溯、过期提示
```

路径不是唯一可靠来源身份。用户重命名或移动后，如果没有其他可确认信号，就不能强行溯源。

## 7. 索引和搜索

### 7.1 默认全文索引

知识库创建后默认可做全文搜索。用户不需要先配置 embedding model。

可直接索引的材料包括：

- Markdown
- txt
- JSON
- CSV
- URL 快照 Markdown
- 云端快照文本
- Cherry Studio 笔记快照
- Agent 生成的文本材料

### 7.2 语义索引

配置 embedding model 后，知识库可以建立语义索引。embedding model、维度等是知识库配置项，不是创建知识库的前置输入。

文件处理器生成 Markdown 时，索引目标是生成后的 Markdown，而不是原 PDF。这样 search 结果能指向可读文本。

### 7.3 Rerank

Rerank 是检索结果排序增强，不决定知识库是否可用。

### 7.4 搜索结果

搜索结果需要返回足够的 locator，至少能让 Agent 后续读取附近上下文：

```text
knowledgeBaseId
source path
chunk id / chunk index
offset / range
score
retrieval mode
metadata
```

metadata 应尽量包含材料类型、来源、处理产物关系、快照刷新状态等信息。

search locator 的产品目的，是让 Agent 在搜索片段不足时通过 read 读取相邻上下文，而不是重新扫描知识库目录或直接读取任意文件。

## 8. Agent 知识库能力

本节中的 `list` 指“列出知识库”的 Agent 工具能力，不是 UI 文件树使用的 fs list，也不是“列出某个知识库目录内容”。为了避免实现时混淆，后续工具命名可以区分为 `listKnowledgeBases`、`listFiles` / `tree` 等更明确的名称。

### 8.1 Agent 如何知道候选知识库

用户可以在 Agent 配置中绑定知识库，也可以在对话中 `@知识库`，也可以从知识库详情页进入 Agent 对话并把当前知识库作为本轮临时上下文。

产品语义：

- Agent 配置的知识库会在每轮对话中携带。
- 用户 `@知识库` 会把这些知识库 id 加入当前提示词上下文。
- 从知识库详情页发起的 Agent 对话，会把当前知识库 id 加入本次会话上下文，但不写入 Agent 或 Assistant 的持久配置。
- 这些 id 是候选知识库，不是权限边界。
- 候选 id 的作用是提示 Agent 默认优先或可以考虑搜索，不强制限制 search 范围。
- 临时候选不用纠结持久化，按当前对话语义处理。

### 8.2 list 工具

`list` 工具应列出当前用户可见的所有知识库。

它不是“只列出当前候选知识库”。Agent 可以通过 prompt 看到候选 id，但 `list` 的产品语义是枚举可见知识库集合，便于 Agent 选择和确认。

未来企业版权限会影响“可见知识库集合”，但当前不在本轮定稿。

当前产品边界下，“所有知识库”指当前用户可见的知识库集合，不是跨用户、跨租户或绕过权限的全局枚举。

### 8.3 search 工具

`search` 必须显式传入知识库 id。

它不限制传入的 id 必须来自候选知识库。候选 id 只是提示 Agent “这些通常相关”，不是 search 的 allowlist。

这样设计是为了支持：

- Agent 根据 `list` 发现其他知识库。
- 用户临时指定 id。
- 后续更复杂的工作流跨多个知识库检索。

权限校验未来在工具封装时再确认。

即使 search 不要求 id 来自候选列表，它仍应受当前用户可见知识库集合约束。候选 id 不是 allowlist，但用户权限仍是边界。

### 8.4 read 工具

Agent 读取上下文不应长期依赖原始 filesystem 工具。

原因是未来会有云端知识库和企业权限：

- 员工可能无法通过本地 fs 读取云端知识库。
- 员工可能没有权限读取某些材料。
- search 命中的材料不一定是本地文件。

因此，知识库系统应封装 read 能力：Agent 使用 search locator 读取附近上下文，由知识库系统处理本地文件、云端快照、权限和范围控制。

read 的优先输入应是 search locator，而不是任意 filesystem path。它是搜索后的上下文补充工具，不是让 Agent 任意浏览或读取知识库路径。未来如果支持按路径读取，也应先经过知识库目录边界和权限校验。

短期验收或调试中如果用 filesystem 补上下文，应视为本地过渡方案，不进入长期产品承诺。

### 8.5 tree / list files 能力

Agent 如果要整理目录、移动文件、删除重复项或确认某个目录下有哪些材料，需要知识库系统提供目录浏览能力。

该能力和 `list` 知识库工具不同：

- `list`：列出当前用户可见的知识库。
- `tree` / `listFiles`：列出某个知识库内的可见文件树或指定目录内容。

短期可以由 filesystem 工具过渡，但长期应由知识库工具封装，以支持云端知识库和企业权限。它不是 UI watcher 的裸接口，而是受知识库边界约束的目录浏览能力。

### 8.6 refresh 工具

Agent 可以刷新 URL 或云端快照。

刷新是覆盖操作，需要用户确认，除非后续明确建立安全的自动确认规则。

### 8.7 add / delete 工具

Agent 需要能添加和删除知识库材料：

- `add`：把本地路径、文件、文件夹、URL、Cherry Studio 笔记、云端文档来源或纯文本笔记加入知识库。
- `delete`：删除知识库中的材料，并清理索引。

破坏性操作、覆盖操作、批量删除操作需要确认。

低风险的整理、移动、生成报告等操作可以执行后报告。

## 9. 旧知识库迁移

旧知识库的文件逻辑是：用户上传文件或文件夹时，通过 file manager 纳入 internal 管理。

迁移到新知识库时：

- 创建新的 folder-based 知识库副本。
- 从 internal 中重建原文件和文件夹。
- 保留旧数据中可以证明的文件夹层级关系。
- URL 在迁移时重新拉取，沉淀为 Markdown。
- 复用旧知识库的配置，包括 embedding model 等。
- 非 URL 材料内容保持不变，且能证明旧 chunk / loader metadata 可以映射到新材料时，原有向量内容应作为复用目标；无法证明映射时跳过向量复用并记录诊断。
- URL 重新拉取后的内容需要更新对应全文索引和向量索引。
- 迁移是一次性复制，不自动同步旧知识库后续变化。
- 迁移后不自动替换 Agent 或 Assistant 已绑定的旧知识库配置。

迁移不根据文件名或路径片段猜测层级、来源或向量映射。如果旧数据无法证明某个层级关系、材料类型、向量维度或旧 chunk 到新材料的映射关系，应跳过不可证明的部分并记录诊断，而不是生成看似正确但无法追溯的迁移结果。

如果用户创建空白新知识库，则只创建新目录，不要求模型配置。

## 10. UI 展示规则

### 10.1 文件管理器式主视图

知识库详情页采用文件管理器式主视图。UI 的主对象是文件和目录，页面主体展示真实目录中的可见文件。来源、处理、索引、刷新、Agent 生成等信息作为文件状态和详情解释，不重新成为页面主结构。

第一阶段的主视图形态是类似 ima 的单区资料浏览器：用户在当前目录中浏览资料，可以在列表视图和网格视图之间切换。目录导航可以通过返回、前进、面包屑或进入文件夹完成；默认不要求左侧常驻目录树。

用户点击列表行或网格卡片时，打开资料详情弹层。这个弹层和当前知识库 RAG 配置使用同一类页面级覆盖层心智：底层资料浏览器保持可见但不可直接操作，弹层展示选中资料的来源、索引、处理、刷新和可用操作，关闭后回到原来的列表或网格位置。

第一阶段不做用户可见的变更日志、操作历史、文件版本或 Agent 操作审计。Agent 移动、生成、删除文件后，UI 通过真实目录展示当前结果；如果需要解释，优先由 Agent 在当前对话里即时报告，不沉淀为知识库长期日志。

例如用户上传 PDF 且启用了文件处理：

```text
knowledge/
  report.pdf
  report.md
```

UI 显示两个文件是符合预期的，因为它们都是真实文件。`report.md` 可以标记为处理产物，但不能隐藏。

为了降低重复感，UI 应显示二者关系：`report.md` 是由 `report.pdf` 生成的 processed material，也是实际被索引和搜索返回的文本来源；`report.pdf` 是原始上传文件，可用于预览、重新处理或留档。

### 10.2 状态和标签

列表行和网格卡片只展示关键状态；资料详情弹层展示完整来源、刷新、索引、处理和 Agent 关系。UI 可以从 metadata 中显示辅助状态，例如：

- URL 快照
- 云端快照
- 处理产物
- 处理来源
- 笔记快照
- Agent 生成
- 索引中
- 索引失败
- 可刷新
- 可能过期

这些状态不改变文件树的显示来源。文件树仍以 fs watcher/list 为准。

### 10.3 刷新确认

对 URL 或云端快照点击刷新时，提示用户该操作会覆盖当前保存的快照文件。

用户确认后执行覆盖并更新索引。

### 10.4 不做固定 raw/processed tab

不把 UI 固定拆成“原始资料”和“处理后资料”两个 tab。

原因：

- 用户可以自由组织目录。
- Agent 可能整理出非原始结构。
- 固定 tab 会让真实目录和用户理解冲突。

如果需要辅助筛选，可以做过滤器或标签，但默认视图仍是统一资料浏览器。

### 10.5 不以数据源列表作为主视图

资料来源、URL、云端文档、笔记快照、处理来源等信息需要保留，但它们是文件详情和筛选条件，不是知识库详情页的主结构。

这样做是为了避免 Agent 整理、用户移动文件、PDF 生成 Markdown 等场景下，来源列表和真实目录产生冲突。旧的 `Data Source` 表格需要在新 UI 中重新定位为文件详情、筛选或诊断信息。

## 11. 参考 MinerU Document Explorer 的数据思路

`/Users/eeee/downloads/MinerU-Document-Explorer` 只作为数据设计参考，不参考 UI。

可参考的思想：

- 用 collection 表达资料集合。
- 用 path 表达用户可见路径。
- 用 hash 或内容身份表达材料内容。
- 用 active 状态处理删除或失效。
- 把 content、页面缓存、目录缓存、wiki source、wiki log 等派生产物分层。
- 把 raw collection 和 wiki / generated collection 的关系拆开。

不可直接照搬的点：

- Cherry Studio 的 UI 要展示真实知识库目录，不引入固定虚拟目录。
- MinerU 的 collection 是数据组织参考，不等于 Cherry UI 的虚拟分组，也不等于固定 raw/wiki 双集合。
- Cherry Studio 的 Agent 管理目标更强调用户和 Agent 可自由移动、生成、整理文件。
- 表结构要等技术设计阶段结合 Cherry Studio 的 DataApi、SQLite、watcher、file service 和权限模型再定。

## 13. 非目标

第一阶段不做：

- 强制用户选择 embedding 后才能创建知识库。
- 强制用户使用固定 `raw/` 或 `processed/` 目录。
- 用额外 UI 表决定目录展示。
- 用数据源列表替代文件管理器式主视图。
- 用户可见的变更日志、操作历史、文件版本或 Agent 操作审计。
- 内容重复自动拦截。
- URL 或云端内容实时同步。
- Cherry Studio 笔记实时引用或自动同步。
- 删除 PDF 时自动删除生成的 Markdown。
- 自动刷新并覆盖 URL 快照。
- 自动改写 Agent 生成的报告。
- 把候选知识库 id 当成权限 allowlist。

## 14. 验收口径

### 14.1 创建和上传

通过标准：

- 用户只输入名称即可创建知识库。
- 创建后有独立根目录。
- 上传文件夹后保留层级。
- 上传文件不改变文件名。
- 知识库详情页使用文件管理器式主视图，主对象是文件和目录。
- 第一阶段主视图支持列表和网格两种资料浏览方式。
- UI 通过真实目录显示材料。
- 来源、处理、索引、刷新和 Agent 生成状态作为文件状态或详情解释，不替代文件树。
- 文件内容变化后更新对应索引。
- 外部文件系统删除后 UI 移除文件并清理索引，不二次确认。
- `.cherry/` 等系统目录不显示。
- 第一阶段不展示变更日志、操作历史、文件版本或 Agent 操作审计。

### 14.2 URL 快照

通过标准：

- 添加 URL 后生成 Markdown 文件。
- 文件在当前目录可见。
- 搜索能搜到 Markdown 内容。
- 刷新前提示覆盖。
- 刷新后覆盖原快照并更新索引。

### 14.3 Cherry Studio 笔记快照

通过标准：

- 添加 Cherry Studio 笔记后，复制其索引到的本地源文件。
- 文件名默认使用源文件名。
- UI 把它显示为真实文件。
- 源笔记或源文件变化后，不自动同步知识库副本。
- 手动刷新前提示覆盖。
- 刷新后覆盖当前快照并更新索引。
- 删除知识库副本不删除源文件或 Cherry Studio 笔记。

### 14.4 PDF 处理

通过标准：

- 启用文件处理器后上传 PDF，会生成 Markdown。
- UI 同时显示 PDF 和 Markdown。
- 搜索返回 Markdown 内容。
- 删除 PDF 不删除 Markdown。
- 重新处理 PDF 时如目标 Markdown 冲突，进入普通冲突流程。

### 14.5 Agent 工具

通过标准：

- Agent 配置的知识库每轮携带候选 id。
- `@知识库` 会把知识库 id 加入当前提示词。
- 从知识库详情页进入 Agent 时，当前知识库作为本次会话候选，不写入持久配置。
- `list` 可以列出当前用户可见的知识库。
- `search` 必须传入知识库 id。
- `search` 不要求 id 来自候选列表。
- `search` 仍受当前用户可见知识库集合约束。
- search 结果包含 locator，Agent 可用 read 工具读取附近上下文。
- Agent 可通过 tree / listFiles 能力浏览知识库内文件树。
- Agent 可 add、delete、refresh，但覆盖和删除需要确认。

### 14.6 迁移

通过标准：

- 旧知识库迁移创建新的知识库副本。
- internal 文件和文件夹层级被重建。
- 无法证明的层级、材料类型或向量映射不猜测，记录诊断。
- URL 在迁移时拉取为 Markdown。
- 旧 embedding 等配置被复用。
- URL 新内容对应索引被更新。
- 不自动修改 Agent / Assistant 的旧绑定。
