# 知识库 UI 呈现方案文档

本文记录 Agent 管理型知识库的 UI 呈现决策，并保留被讨论过的替代方案，方便后续技术设计和产品细化时对照。

当前已确定：

- 用户上传或导入任何来源，都是把来源复制或沉淀为知识库自有资料/快照。
- 产品内部总称统一为“资料”（Knowledge Material），但具体 UI 显示名后续可按设计再定。
- 知识库详情页使用文件管理器式主视图。
- UI 主对象是文件和目录。
- 第一阶段主区是类似 ima 的单区资料浏览器，支持列表视图和网格视图切换。
- 点击资料后打开类似当前 RAG 配置的页面级详情弹层，展示完整资料详情。
- 真实目录是页面主体，来源、处理、索引、刷新、Agent 生成等信息作为文件状态和详情解释。
- 第一阶段不做用户可见的变更日志、操作历史或文件版本。

本文不替代 [Agent 管理型知识库产品文档](./agent-managed-knowledge-product.md) 中已经确认的产品语义。

## 1. 当前已经确定和还没确定的事

已经确定：

- 新知识库是 folder-based，用户材料放在知识库自己的真实目录下。
- 文件、文件夹、Cherry Studio 笔记、URL、云端文档等来源进入知识库时，都是复制或沉淀为知识库自有资料/快照。
- 内部产品语言统一称为“资料”（Knowledge Material）；最终 UI 文案是否显示为“资料”“文件”或其他名称，留到 UI 设计阶段决定。
- UI 不能用虚拟表替代真实目录树。
- metadata 只能解释文件，不能决定目录树显示什么。
- 知识库详情页主对象是文件和目录，采用文件管理器式主视图。
- 第一阶段不默认做左侧常驻目录树，而是用单区资料浏览器展示当前目录，并支持列表 / 网格切换。
- 点击列表行或网格卡片后打开资料详情弹层；它不替代资料浏览器，也不在第一阶段改成页面内固定详情栏。
- 用户启用文件处理器后，PDF 生成的 Markdown 是独立可见文件。
- Agent 写入知识库目录后的文件是普通可见材料。
- URL 和云端快照刷新会覆盖当前保存的快照，覆盖前需要确认。
- 第一阶段不做最近变更、操作日志、版本历史或 Agent 操作审计。

还没确定：

- 具体 UI 上 `Data Source` tab 和相关入口最终显示什么文案。
- 列表视图和网格视图分别展示哪些状态信息。
- 状态标签、详情弹层、搜索结果跳转之间如何配合。
- Agent 整理目录后的即时反馈如何在对话中表达，但不沉淀为知识库长期变更日志。

关键提醒：

```text
现在讨论的不是：

  UI 是否展示真实目录

这个已经确定。

现在真正要讨论的是：

  真实目录作为事实之后，
  来源、状态、搜索、处理产物和 Agent 行为应该怎样解释给用户。
```

## 2. 为什么 UI 是当前最重要的问题

UI 主形态会牵引很多产品决定：

| 被牵引的问题 | 影响 |
|---|---|
| 主对象命名 | UI 叫“资料”“文件”“来源”“条目”，会影响用户理解 |
| 添加资料入口 | 是添加到当前目录，还是添加为一条来源记录 |
| PDF 和 Markdown 关系 | UI 要不要同时显示、怎么降低重复感 |
| URL 快照刷新 | 刷新按钮放在文件行、详情弹层，还是来源列表 |
| Agent 整理结果 | Agent 移动文件后，UI 是展示新目录，还是保留旧来源分组 |
| 搜索结果跳转 | 跳到文件、chunk、来源条目，还是预览面板 |
| 旧知识库迁移 | 迁移后用户看到真实目录，还是旧的数据源列表 |
| 来源追溯 | provenance 是文件详情里的解释，还是列表主结构 |

已经确认 UI 以文件目录为主。因此，来源、索引和处理关系都需要变成文件旁边的解释信息，而不是重新成为页面主结构。

## 3. 已选方案：单区资料浏览器

### 3.1 列表视图草图

```text
+--------------------------------------------------------------------+
| <  >  产品资料                              [搜索] [网格] [+ 添加] |
| 32 个文件 · 28 个已索引 · 2 个可刷新 · 最近更新 3 分钟前            |
+--------------------------------------------------------------------+
| 名称                                      类型      大小   更新时间 |
|--------------------------------------------------------------------|
| raw/                                      文件夹    --     15:13    |
| reports/                                  文件夹    --     15:12    |
| 产品白皮书.pdf                            PDF       5 MB   15:11    |
| 产品白皮书.md                处理产物      MD        82 KB  15:11    |
| 官网介绍.md                  URL快照       MD        12 KB  昨天     |
+--------------------------------------------------------------------+
| 选中：产品白皮书.md                                                |
| 来源：由 raw/产品白皮书.pdf 处理生成                               |
| 搜索：当前搜索优先使用这个 Markdown，不直接索引 PDF                 |
| 操作：[预览] [打开所在目录] [重新处理] [刷新索引] [删除]             |
+--------------------------------------------------------------------+
```

### 3.2 网格视图草图

```text
+--------------------------------------------------------------------+
| <  >  产品资料                              [搜索] [列表] [+ 添加] |
| /                                                                  |
+--------------------------------------------------------------------+
| +----------------+ +----------------+ +----------------+           |
| | 预览缩略图      | | 预览缩略图      | | 文件夹          |           |
| | 产品白皮书.pdf  | | 产品白皮书.md   | | reports/       |           |
| | PDF · 已索引    | | 处理产物        | | 12 个文件       |           |
| +----------------+ +----------------+ +----------------+           |
+--------------------------------------------------------------------+
| 点击卡片后弹出详情弹层：来源、索引、处理、刷新和可用操作            |
+--------------------------------------------------------------------+
```

### 3.3 这个方案如何解释资料

```text
用户看到的主对象：

  文件 / 目录

文件旁边显示：

  类型、来源、处理状态、索引状态、刷新能力、Agent 生成状态

文件详情解释：

  它来自哪里
  它是否被搜索使用
  它是否能刷新
	它和其他文件有什么关系
```

列表和网格只是同一真实目录的两种呈现方式，不改变资料身份、目录结构或来源解释。

目录导航可以通过返回、前进、面包屑和进入文件夹完成；第一阶段不要求左侧常驻目录树。

点击资料打开详情弹层。这个弹层和当前知识库 RAG 配置弹层属于同一类页面级覆盖层：底层资料浏览器保持可见但不可直接操作，关闭后回到原列表或网格状态。它不是行内展开，也不是三点菜单那种小操作 Popover。

### 3.4 典型操作

用户添加文件夹：

```text
用户选择 folder/
        |
        v
复制到知识库当前目录
        |
        v
UI 出现 folder/ 及其原始层级
        |
        v
系统逐个建立索引并在文件行显示状态
```

用户上传 PDF 并生成 Markdown：

```text
用户上传 report.pdf
        |
        v
目录出现 report.pdf
        |
        +-- 未启用处理器
        |       |
        |       v
        |   report.pdf 作为原文件展示
        |
        +-- 启用处理器
                |
                v
            生成 report.md
                |
                v
            UI 同时显示 report.pdf 和 report.md
                |
                v
            搜索优先返回 report.md
```

URL 刷新：

```text
官网介绍.md  [URL快照] [可刷新]
        |
        v
用户点击刷新
        |
        v
提示：刷新会覆盖当前文件内容
        |
        +-- 取消
        |       |
        |       v
        |     不变
        |
        +-- 确认
                |
                v
            重新抓取 URL
                |
                v
            覆盖当前 Markdown
                |
                v
            更新全文 / 语义索引
```

### 3.5 优点

- 最符合 folder-based 知识库方向。
- 贴近 ima 式资料浏览体验，列表适合扫描字段，网格适合看缩略图和快速识别文档。
- 用户和 Agent 对目录的整理结果可以直接被 UI 表达。
- PDF 和 Markdown 同时存在这件事比较容易解释。
- 未来支持云端快照时，也可以作为普通文件显示，并用状态解释来源。
- 与 Finder / 文件管理器的心智接近。

### 3.6 风险

- 需要设计好状态标签，否则文件行会变得很吵。
- 网格视图空间更少，不能展示和列表完全相同的信息，需要定义状态优先级。
- 需要一个清晰的详情弹层，否则来源关系、搜索使用关系不容易讲明白。
- 现有 `Data Source` 表格和 chunk detail 需要重新定位。

### 3.7 结论

确定作为第一阶段主方向。

原因是它和已经确认的“真实目录是用户可见事实”最一致，同时保留列表 / 网格两种用户浏览习惯。

## 4. Pass 掉的方案 B：资料来源式主视图

### 4.1 界面草图

```text
+--------------------------------------------------------------------+
| 产品资料                                          [搜索] [+ 添加来源]|
| 数据源 8 个 · 已索引 7 个 · 失败 1 个                               |
+--------------------------------------------------------------------+
| 名称                 来源类型       状态        最近更新      操作  |
|--------------------------------------------------------------------|
| 产品白皮书.pdf       文件           已索引      3 分钟前      ...   |
| 产品白皮书.md        处理产物       已索引      3 分钟前      ...   |
| 官网介绍             URL            可刷新      1 小时前      刷新  |
| 会议纪要             笔记快照       已索引      昨天          ...   |
+--------------------------------------------------------------------+
| 详情：官网介绍                                                     |
| 来源 URL：https://example.com                                      |
| 保存文件：/官网介绍.md                                             |
| 操作：[预览快照] [刷新] [查看文件] [删除来源]                       |
+--------------------------------------------------------------------+
```

### 4.2 这个方案如何解释资料

```text
用户看到的主对象：

  来源条目

真实目录的角色：

  每个来源条目背后的保存位置

来源条目负责解释：

  这份资料从哪里来
  当前索引状态是什么
  是否可刷新
```

### 4.3 典型操作

Agent 整理文件后的冲突点：

```text
初始状态：

  来源条目：官网介绍
  保存文件：/官网介绍.md

Agent 移动：

  /官网介绍.md -> /raw/web/官网介绍.md

UI 面临选择：

  A. 来源列表仍显示“官网介绍”，保存路径改成新路径
  B. 来源列表按目录重新分组
  C. 只在文件树里显示新路径

问题：

  如果主视图仍是来源列表，
  用户很难一眼看到 Agent 整理后的真实目录结构。
```

### 4.4 优点

- 接近当前实现，迁移成本较低。
- URL、笔记快照、云端文档这类“来源关系强”的材料解释很直接。
- 索引状态和失败原因容易按条目展示。

### 4.5 风险

- 与“UI 展示真实目录”的产品方向冲突。
- Agent 移动、重命名、整理目录后，来源列表会弱化真实结果。
- PDF 和 Markdown 同时可见时，用户可能分不清哪个是文件、哪个是来源。
- 可能迫使产品重新维护一套“UI 条目表”，这和前面讨论过的方向不一致。

### 4.6 结论

不作为主视图。

它可以退化成文件详情中的“来源信息”或筛选条件，但不应该决定知识库详情页的主结构。

## 5. 备选增强方案 C：双栏混合式主视图

### 5.1 界面草图

```text
+----------------------+---------------------------------------------+
| 目录                  | 当前目录：raw/                    [搜索][+] |
|----------------------|---------------------------------------------|
| v /                  | 名称                  类型       状态        |
|   v raw/             |---------------------------------------------|
|     产品白皮书.pdf   | 产品白皮书.pdf       PDF        原文件       |
|     产品白皮书.md    | 产品白皮书.md        MD         处理产物     |
|   > reports/         |                                             |
|   官网介绍.md        |                                             |
+----------------------+---------------------------------------------+
| 详情：产品白皮书.md                                                 |
| 来源：由 raw/产品白皮书.pdf 处理生成                                |
| 搜索：当前搜索使用这个 Markdown，不直接索引 PDF                      |
| 状态：已索引 · 用户未编辑 · 可重新处理                              |
| 操作：[预览] [打开] [重新处理] [刷新索引] [删除]                     |
+--------------------------------------------------------------------+
```

### 5.2 这个方案如何解释资料

```text
左侧：

  目录树，帮助用户理解结构

右侧：

  当前目录的文件列表，承载批量操作和排序

底部或右侧详情：

  解释来源、索引、处理关系、刷新能力
```

### 5.3 典型操作

用户在当前目录添加 URL：

```text
用户选中目录：raw/web/
        |
        v
点击 + 添加 URL
        |
        v
输入 https://example.com
        |
        v
抓取为 Markdown 快照
        |
        v
raw/web/ 下出现 example.md
        |
        v
详情显示：URL 快照 · 可刷新 · 来源 URL
```

Agent 整理目录：

```text
Agent 执行：

  /官网介绍.md -> /raw/web/官网介绍.md
  /产品白皮书.md -> /processed/产品白皮书.md

UI 更新：

  左侧目录树出现 raw/web/ 和 processed/
  右侧列表按当前选中目录展示
  点击资料后在详情弹层里保留来源解释
```

### 5.4 优点

- 比单列文件树信息密度更高。
- 适合文件多、目录深的知识库。
- 目录结构、文件列表、详情解释三者职责清楚。
- 可以自然承接批量选择、排序、筛选。

### 5.5 风险

- 实现和交互比方案 A 更重。
- 小知识库里可能显得复杂。
- 移动端或窄窗口适配成本更高。

### 5.6 结论

适合作为单区资料浏览器的增强形态。

第一阶段不默认做左侧常驻目录树。如果后续知识库目录明显变深、用户需要跨目录批量整理，双栏混合式可以作为增强形态引入。

## 6. Pass 掉的方案 D：Agent 工作台式主视图

### 6.1 界面草图

```text
+--------------------------------------------------------------------+
| 产品资料                                  [让 Agent 整理] [+ 添加] |
+--------------------------------------------------------------------+
| Agent 建议                                                           |
|--------------------------------------------------------------------|
| [ ] 发现 3 份重复文档：产品白皮书.md、产品白皮书_2.md ...           |
| [ ] 官网介绍.md 是 32 天前的 URL 快照，建议刷新                     |
| [ ] reports/ 下有 2 篇摘要引用了已更新的原始资料，可能过期          |
+--------------------------------------------------------------------+
| 文件入口                                                            |
|--------------------------------------------------------------------|
| [打开文件树] [搜索知识库] [查看全部资料]                            |
+--------------------------------------------------------------------+
```

### 6.2 这个方案如何解释资料

```text
用户看到的主对象：

  维护任务 / 建议 / 最近变化

真实目录的角色：

  变成一个入口，而不是主画面

Agent 的角色：

  从辅助能力变成主工作流
```

### 6.3 优点

- 能突出 Agent-managed 的愿景。
- 适合展示重复、过期、待刷新、整理建议。
- 对长期维护大型知识库有价值。

### 6.4 风险

- 第一眼看不到真实目录，削弱 folder-based 心智。
- 容易让用户觉得知识库是任务系统，而不是资料空间。
- 需要比较成熟的 Agent 诊断和建议能力，否则主视图会空。

### 6.5 结论

不作为第一阶段主视图。

可以作为后续右侧面板、顶部建议区或 Agent 入口，而不是替代文件树。第一阶段也不做“最近变更”这类长期变更日志。

## 7. 四个方案对比和最终选择

| 方案 | 主对象 | 优点 | 缺点 | 建议 |
|---|---|---|---|---|
| A. 单区资料浏览器 | 文件 / 目录 | 最符合真实目录；支持列表 / 网格；Agent 整理结果可见 | 状态标签和详情弹层要设计好 | 已选择 |
| B. 资料来源式 | 来源条目 | 继承现有 UI；来源解释直接 | 与真实目录和 Agent 整理冲突 | Pass |
| C. 双栏混合式 | 目录 + 文件列表 | 信息密度高；适合复杂目录 | 交互和实现更重 | 后续增强 |
| D. Agent 工作台式 | 维护任务 | 突出 Agent 管理 | 弱化真实目录；依赖 Agent 能力成熟 | Pass |

最终判断：

```text
第一阶段确定方向：

  方案 A：单区资料浏览器，支持列表 / 网格

后续增强：

  方案 C：双栏混合式可以作为后续增强形态

Pass 掉：

  用方案 B 继续作为主视图
  用方案 D 替代文件树
```

## 9. UI 决策会牵引的具体问题

### 9.1 主对象命名

选择文件管理器式后：

- 内部产品语言应从“数据源”转向“资料”（Knowledge Material）。
- 具体 UI 文案可以后续再决定为“资料”“文件”或更适合设计稿的名称。
- “来源”变成文件详情中的解释字段。
- “索引状态”变成文件状态。

如果继续叫“数据源”：

- 用户会预期看到的是来源记录，不是目录。
- PDF 生成 Markdown 后，两个文件同时出现会显得奇怪。
- Agent 移动文件后，来源记录和真实路径之间需要额外解释。

### 9.2 添加资料入口

文件管理器式 UI 下，添加动作天然发生在“当前目录”：

```text
选中 /raw/
  -> 点击 + 添加
  -> 新资料进入 /raw/
```

资料来源式 UI 下，添加动作更像“添加来源”：

```text
点击 + 添加来源
  -> 选择文件 / URL / 笔记
  -> 系统决定保存位置
```

这个差异会影响 URL 默认保存位置、文件夹上传位置和冲突 dialog。

### 9.3 资料详情弹层

如果以文件为主，点击资料后弹出的详情弹层至少要解释：

- 当前路径。
- 文件类型。
- 来源类型：本地上传、URL 快照、云端快照、笔记快照、处理产物、Agent 生成。
- 索引状态。
- 是否被搜索使用。
- 是否可刷新。
- 来源关系，例如 `report.md` 来自 `report.pdf`。
- 用户是否编辑过 Agent 生成内容或快照内容。

详情弹层应复用当前知识库 RAG 配置的页面级覆盖层心智：从页面上方覆盖出来，底层资料浏览器保持可见但不可直接操作，关闭后回到原来的列表或网格状态。它不是替代当前目录的独立页面，也不是三点菜单使用的小 Popover。

### 9.4 状态标签

建议状态标签分层，不要把所有信息挤在文件名后面：

```text
文件行上直接显示：

  类型 + 关键状态

详情弹层显示：

  完整来源、刷新、索引、处理、Agent 关系
```

示例：

```text
列表行：

  产品白皮书.md      MD · 处理产物 · 已索引

详情：

  来源文件：产品白皮书.pdf
  搜索使用：是
  用户编辑：否
  可重新处理：是
```

### 9.5 Agent 操作反馈

Agent 做完整理后，第一阶段只需要即时反馈，不在知识库 UI 中沉淀长期变更日志。

```text
即时反馈：

  Agent 已移动 4 个文件，生成 1 篇摘要
```

真实结果仍然由文件树表达：Agent 移动、生成、删除文件后，UI 通过真实目录展示当前状态。

不建议为了展示 Agent 结果而维护一个独立的“Agent 产物池”。Agent 写入目录就是普通文件。

### 9.6 搜索和目录定位

搜索结果应同时支持两种入口：

```text
打开内容：

  进入预览并定位到命中片段

定位文件：

  在目录树中选中该文件
```

这能兼顾“我要读内容”和“我要理解它在知识库哪里”。

## 11. UI / 技术细化暂缓项

以下问题暂不继续作为产品语义讨论，后续进入 UI 设计或技术设计：

1. `Data Source` tab 和相关入口在具体 UI 上最终显示什么文案？
2. 列表视图和网格视图分别展示哪些状态信息？
3. 资料详情弹层展示哪些字段和操作？
4. 状态标签哪些显示在列表行、网格卡片，哪些只显示在详情里？
5. URL / 云端刷新按钮放在文件行、详情弹层，还是右键菜单？
6. 搜索结果点击后默认打开预览，还是默认在目录中定位？
7. Agent 整理后的即时反馈只留在对话里，还是在当前页面短暂 toast / banner 展示？
8. 旧知识库迁移后，是直接展示重建后的目录，还是先展示迁移报告？
9. 笔记快照刷新身份失效时，UI 是只隐藏刷新入口，还是明确显示“无法刷新”？
10. `.cherry/` 等系统目录隐藏规则是否需要在 UI 上可解释？
