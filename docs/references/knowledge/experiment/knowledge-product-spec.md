## 1. 产品定位与要解决的问题

**一句话定位：知识库是一个「可搜索、可被 Agent 管理的资料文件夹」。**

知识库的产品形态从「先配检索模型、再上传资料的 RAG」，转向「**先放资料、默认就能搜，后续 embedding / rerank / 文件处理器越配越强**」的资料空间。用户文案始终叫「知识库」，不把「File Mode」这类内部概念暴露给用户。

要解决的核心问题：

| 用户诉求 | 产品表现 |
| --- | --- |
| 不想先理解向量 / RAG / embedding 才能建库 | 只输名称即可建库，未配 embedding 也能做全文搜索 |
| 资料是我自己的，不希望被锁进黑箱格式 | 导入即复制为知识库自有副本，UI 以真实文件夹为准 |
| 想让 AI 帮我用这些资料，但要可控 | Agent 作为帮手，低风险操作执行后报告，破坏性操作需确认 |
| 搜索要能告诉我答案来自哪 | 结果须带来源定位（哪个库 / 哪个文件 / 什么位置 / 什么类型） |

> 「为什么这不只是『带搜索的文件夹』、为何要做成独立的检索资产层」的论证，见 [调研报告 · File Mode + RAG 共识](./knowledge-research-report.md)，本文不展开。本文也不展开检索算法。

## 2. 产品原则

四条核心原则（并辅以「可控 / 可懂 / 可解释」的概括表述）：

### 2.1 创建门槛要低

只需输入名称即可建库并立即创建独立根目录；不要求先选 embedding 模型，不要求用户理解向量 / RAG。未配 embedding 时仍可立即进行全文搜索。

### 2.2 导入即复制，知识库拥有稳定副本

上传 / 导入 = 把数据源复制或沉淀为知识库**自有的稳定副本 / 快照**。外部来源的后续变化不会自动改写知识库内容。删除、刷新、索引、UI 都围绕这份自有副本设计。

### 2.3 真实目录是用户看到的事实

UI 内容来自真实文件夹；不维护虚拟目录表，元数据只用于解释、不替代真实目录；目录如何组织（raw / processed / reports 或平铺）由用户和 Agent 决定；索引 / chunk / 缓存等系统资产不出现在材料目录中。

### 2.4 Agent 是帮手，不是无边界自动化

整理目录、移动文件、生成摘要等低风险操作可以执行后报告；添加资料遇冲突要遵循冲突规则；刷新 URL / 云端快照会覆盖现有内容，需要确认；删除（尤其批量）需要确认；覆盖已有文件需要确认。

> 上述原则与调研里「可控 / 可懂 / 可解释」三原则同源，论证背景见 [调研报告](./knowledge-research-report.md)。

## 3. 用户的完整使用流程

### 3.1 创建知识库

用户只输入名称即可建库，系统为其创建独立根目录。即使尚未配置 embedding，也能立即进行基础全文搜索。

### 3.2 添加资料

用户添加资料时不要求先区分「原始 / 快照 / 处理产物」；UI 在必要时解释来源、能否刷新、是否由处理器生成。支持的来源形态包括本地文件 / 文件夹、URL、Cherry Studio 笔记、未来的云端文档、Agent 生成材料等（各自的产品行为见第 4 章）。

### 3.3 搜索

- 全文搜索是底座，即使未配 embedding 也可用；
- 配置 embedding 后增强语义检索能力；
- 配置 rerank 后改善结果排序；
- 搜索结果必须告知**来自哪个库、哪个文件、什么位置、什么类型**。

> 搜索底层「单一入口自适应 hybrid / 纯 BM25 + 降级」「分数语义」等口径见 [调研报告 · 检索栈](./knowledge-research-report.md)，技术落地见 [技术方案 · 检索栈落地](./knowledge-technical-design.md)，本文只写产品口径。

### 3.4 让 Agent 使用知识库

三种让 Agent 知道「用哪些库」的方式：

1. **在 Agent 配置中绑定知识库** — 每轮对话都携带这些知识库 id；
2. **在对话里 @ 知识库** — 当前对话把这些库作为临时候选；
3. **从知识库详情页进入 Agent** — 当前知识库成为本次对话候选，不写入持久配置。

**候选 id 是提示，不是权限边界。** 真正能否搜索 / 读取，受用户可见范围与未来权限规则约束。

## 4. 各类资料的产品行为

各类资料的产品视角行为口径如下。落盘路径、`captures/` 目录、`relativePath`、MinerU 产物存储等技术细节见 [技术方案 · 资料存储与处理](./knowledge-technical-design.md)。

### 4.1 本地文件 / 文件夹

- 复制原内容到库目录，默认不改名，文件夹保留原层级；
- 删除库内副本**不**删除电脑上的原文件；
- 用户从 Finder 等外部删除原文件，UI 移除并清理索引，**不二次确认**；
- 文件内容被改后检测并更新全文索引（已配语义索引则更新向量）。

### 4.2 URL 快照

- URL 不是实时网页引用，而是**快照**；
- 添加时抓取内容并保存为 Markdown，出现在当前目录并被索引；像普通 Markdown 一样展示与搜索；
- 刷新 = 重新抓取并覆盖当前快照（覆盖前需确认），并更新全文与语义索引；
- 最终命名规则待定。

> 产品目标态为「URL 抓为 Markdown 快照」；当前实现仍为每次 reindex 联网抓取、尚未落地本地快照，详见 [技术方案 · 数据模型 as-built 缺口](./knowledge-technical-design.md)。

### 4.3 云端文档快照（后续能力，本阶段不实现）

进入库后存为本地快照、非实时同步、手动刷新、刷新覆盖需确认；删除库内快照不删云端原文档；默认可按服务商（飞书 / 腾讯文档）放目录。云端不现在定死，因涉及企业权限、成员可见性、服务商目录 i18n、移动后刷新等待定问题。

### 4.4 笔记快照

- 按「导入即复制」处理：复制笔记索引到的本地源文件，成为库自有快照；
- 默认文件名用**源文件名**（非笔记标题 / 笔记 id）；
- 源文件或笔记变化不自动同步；手动刷新 = 重读源文件并覆盖（需确认）；
- 删除库内副本不删源文件、也不删笔记；移动 / 重命名后刷新尽量保留身份，无法确认来源时不靠路径猜测。

> 产品目标态为「笔记复制为本地 Markdown 快照」；当前实现笔记仍持久化 inline content、尚未落地快照，详见 [技术方案](./knowledge-technical-design.md)。

### 4.5 PDF 与文件处理产物

启用 MinerU / Paddle 等处理器后上传 PDF 可能生成 Markdown：

- 生成的 Markdown 是**独立可见文件**，不是隐藏系统资产（示例：`年报.pdf` 与 `年报.md` 同时可见）；
- 搜索索引 / 返回 **Markdown** 而非 PDF；
- 删 PDF **不**自动删 md；移 PDF **不**自动移 md；重新处理会再次生成 md；
- 目标 md 已存在按普通冲突处理；用户已移走旧 md 则重新处理不影响旧 md。

> 处理器选型（MinerU 等）与本地两阶段检索价值见 [调研报告](./knowledge-research-report.md)；MinerU 只存最终 Markdown、归属判定等实现见 [技术方案 · 文件处理与 MinerU](./knowledge-technical-design.md)。

### 4.6 Agent 生成材料

- 写入库目录即普通可见材料；不设隐藏「Agent 成果池」，不要求先发布 / 加入目录才显示；
- 用户可移动 / 重命名 / 删除 / 编辑；
- 可记录参考了哪些资料，但来源关系不限制编辑；
- 用户编辑后 UI 可标记「用户已编辑」。

## 5. UI 展示规则

### 5.1 文件管理器式主视图

- 第一阶段为类 ima 的单区资料浏览器，支持**列表 / 网格切换**；
- 导航靠返回 / 前进 / 面包屑 / 进入文件夹，默认不要求左侧常驻目录树；
- 点击资料打开**同页面语境的资料详情浮层 / 面板**（非列表内展开，也非仅操作项小菜单）；
- 来源 / 处理 / 索引 / 刷新 / Agent 生成等信息作为**文件状态与详情解释**，不重新成为页面主结构；
- 旧 Data Source 表需重新定位为文件详情 / 筛选 / 诊断入口。

**文件状态标签（只解释，不决定是否显示）：**

| 标签 | 含义 |
| --- | --- |
| URL 快照 | 来自 URL 抓取 |
| 可刷新 | 可重新抓取覆盖 |
| 云端快照 | 来自云端快照 |
| 处理产物 | 由 PDF 等处理生成 |
| Agent 生成 | 由 Agent 写入 |
| 用户已编辑 | 生成或快照后被用户改过 |
| 索引中 | 正在建索引 |
| 索引失败 | 索引建立失败 |
| 可能过期 | 来源变化后派生材料可能需重新生成 |

### 5.2 不做固定 raw / processed Tab

第一阶段**不做**用户可见的变更日志 / 操作历史 / 文件版本 / Agent 操作审计；Agent 操作后靠真实目录展示结果，需解释时由 Agent 在对话即时报告。不做固定 raw / processed Tab，改用**统一文件树 + 状态标签 + 筛选器**（如只看 URL 快照 / 处理产物 / Agent 生成）。

### 5.3 文件冲突体验

目标路径已存在时，弹出文件管理器式选择：

- **覆盖** = 用新文件替换旧文件；
- **保留副本** = 自动生成 `_2`、`_3` 后缀；
- **跳过** = 不添加这份新资料。

产品**不按内容哈希阻止重复**：两个内容一样但路径不同的文件可共存，后续由 Agent 整理重复。

> 注：「冲突三选一」为**产品目标态**；当前实现为 reject-on-conflict（实时添加冲突时直接报错，仅 v1 迁移去重用后缀），是否补齐 keep-both 待拍板，详见 [技术方案](./knowledge-technical-design.md) 对应 as-built 说明。

## 6. Agent 能力边界（产品语义）

本节只写**产品语义**。工具的技术字段、locator 令牌、`unsupported / conflict / forbidden` 状态、远程 `kb_*` 工具签名等见 [技术方案 · Agent 工具契约](./knowledge-technical-design.md)；5 个工具如何从多立场收敛而来，见 [调研报告 · 工具面调研](./knowledge-research-report.md)。

- **list** — 列出**当前用户可见的所有知识库**（不仅是候选）。候选只是提示，不是列表范围。
- **search** — 必须显式传入知识库 id，**不要求来自候选**，仍受用户可见范围约束。候选不是 allowlist，用户权限才是边界。
- **read** — 先 search 拿到 locator，再读取邻近上下文。核心输入是搜索结果位置，而非任意文件路径；长期不依赖本地文件系统（因未来可能云端 / 企业不可见 / 命中不一定来自本地文件）。
- **tree / listFiles** — 列出某库的文件树 / 目录，受可见范围约束。
- **manage（add / delete / refresh）** — add 普通添加不一定确认（但冲突需确认）；delete 需确认；refresh 覆盖需确认。

**远程 / 企业知识库的能力边界**：默认**只读优先**，写入 / 移动 / 删除 / 自动整理共享库作为后续受控能力（写 note / wiki 或提交变更请求，而非直接改共享原文）；权限必须在服务端强制执行。

> Remote File Mode「不整库同步 / 不本地挂载 / 服务端 ACL」等论证见 [调研报告 · 企业内网 Remote File Mode](./knowledge-research-report.md)。

## 7. 旧知识库迁移（产品行为）

迁移的产品可感知规则：

- **创建新文件夹型副本，不原地转换**；
- **不要猜**：如果旧数据不能证明某个文件夹层级、来源关系、chunk 映射或向量可复用关系，就**跳过不可证明的部分并记录诊断**，而不是生成一个看起来合理但无法追溯的结果；
- URL 迁移时重新抓取；复用旧 embedding 配置；
- **不自动改写** Agent / Assistant 的知识库绑定。

> 迁移的技术终态、id 重映射、向量复用判定、以及与当前实现的差距（as-built）见 [技术方案 · 迁移](./knowledge-technical-design.md)。

## 9. 第一阶段非目标与待定项

### 9.1 第一阶段非目标

不要求配 embedding 才能建库；不强制 raw / processed 目录；不用虚拟 UI 表决定展示；不用数据源列表替代文件管理器视图；不做变更日志 / 历史 / 版本 / 审计；不自动阻止内容重复；不做 URL / 云端实时同步；删 PDF 不自动删 md；不自动刷新覆盖 URL 快照；不做笔记实时引用 / 自动同步；不自动重写 Agent 已生成报告；不把候选 id 当 allowlist；不展开云端企业权限完整实现。

### 9.2 UI / 产品暂缓项

URL 命名规则；云端目录与 i18n；云端快照移动后刷新；企业版权限模型；read 与 tree / listFiles 的工具形态；AI SDK 工具确认策略；来源记录与移动 / 重命名追溯；笔记快照刷新体验；处理产物来源标记；旧向量复用与迁移诊断。

### 9.3 难回滚决策清单（是否写 ADR 待定）

以下 5 条为难回滚的产品决策：

1. 新知识库默认是文件夹型；
2. UI 以真实目录为准，不做虚拟目录表；
3. 处理生成的 Markdown 是可见独立文件；
4. 候选知识库不是 search allowlist；
5. 旧知识库迁移是新副本、不是原地转换。

## 10. 产品验收口径

以下为产品可验证的验收 checklist（工程门禁 / grep gate / 测试矩阵见 [技术方案 · 质量门禁](./knowledge-technical-design.md)）。

### 10.1 创建和上传

- [ ] 只输名称即可建库，并创建独立根目录；
- [ ] 上传保留文件夹层级、不改名；
- [ ] UI 以真实目录显示；
- [ ] 文件内容变更后更新索引；
- [ ] 外部删除原文件后 UI 移除并清理索引。

### 10.2 URL 快照

- [ ] 添加 URL 生成 Markdown，目录可见、可搜索；
- [ ] 刷新提示覆盖；
- [ ] 确认后覆盖快照并更新索引。

### 10.3 笔记快照

- [ ] 复制源文件、用源文件名；
- [ ] 以真实文件显示；
- [ ] 不自动同步；
- [ ] 刷新提示覆盖、覆盖后更新索引；
- [ ] 删除库内副本不删源文件。

### 10.4 PDF 处理

- [ ] 生成 Markdown；
- [ ] PDF 与 Markdown 同时显示；
- [ ] 搜索返回 Markdown；
- [ ] 删 PDF 不删 Markdown；
- [ ] 重新处理走冲突流程。

### 10.5 Agent 使用知识库

- [ ] 绑定 / @ / 详情页进入三种方式正确携带候选 id；
- [ ] list 列出可见的全部库；
- [ ] search 必须传 id、不限候选、受可见范围约束；
